# 📘 DDL - DROP & TRUNCATE (삭제 및 초기화)

## Summary 📝

> [!summary] 핵심 요약
>
>   * **DROP**: 테이블의 **구조(껍데기)와 데이터(알맹이)를 모두 삭제** (Auto Commit).
>   * **TRUNCATE**: 테이블의 **구조는 남기고 데이터만 모두 삭제** (Auto Commit, 속도 빠름).
>   * **CASCADE CONSTRAINTS**: 부모 테이블 삭제 시, 이를 참조하는 **자식 테이블의 제약조건(FK)까지 함께 삭제**.
>   * **PURGE**: 휴지통(Recycle Bin)을 거치지 않고 **완전 삭제**.

-----

## 1\. DROP TABLE (테이블 삭제) 🗑️

데이터베이스에서 테이블 객체 자체를 영구적으로 제거합니다.

### 1.1 기본 문법

```sql
DROP TABLE 테이블명 [CASCADE CONSTRAINTS] [PURGE];
```

  * **실행 결과**:
      * 테이블에 저장된 모든 **행(Row)** 삭제.
      * 테이블의 **구조(Column, Data Type)** 삭제.
      * 테이블에 걸린 **인덱스(Index)** 등 관련 객체 삭제.
      * **Auto Commit**: `ROLLBACK`으로 되돌릴 수 없음.

### 1.2 주요 옵션

| 옵션 | 설명 | 사용 시나리오 |
| :--- | :--- | :--- |
| **CASCADE CONSTRAINTS** | 제약조건 연쇄 삭제 | 다른 테이블이 이 테이블을 참조(FK)하고 있을 때, **참조하는 제약조건을 끊고** 테이블을 삭제함. |
| **PURGE** | 완전 삭제 | 오라클의 '휴지통(Recycle Bin)'에 남기지 않고 즉시 디스크에서 제거. (복구 불가) |

> [!warning] 주의: CASCADE CONSTRAINTS
> 이 옵션은 자식 테이블 자체를 삭제하는 것이 아니라, 자식 테이블에 걸려 있는 **외래키(FK) 제약조건만 삭제**합니다. 자식 테이블의 데이터는 그대로 남습니다.

-----

## 2\. TRUNCATE TABLE (테이블 초기화) ⚡

테이블을 삭제하지 않고, **저장 공간만 비우는** 명령어입니다.

### 2.1 기본 문법

```sql
TRUNCATE TABLE 테이블명;
```

  * **특징**:
      * 테이블 **구조(스키마)는 유지**됨.
      * `DELETE` 명령어보다 속도가 훨씬 빠름 (로그를 남기지 않음).
      * DDL이므로 **Auto Commit** 발생 (복구 불가).

-----

## 3\. 비교: DELETE vs TRUNCATE vs DROP ⚖️

면접 및 실무에서 가장 중요한 비교 포인트입니다.

| 구분 | DELETE (삭제) | TRUNCATE (초기화) | DROP (제거) |
| :---: | :---: | :---: | :---: |
| **명령어 종류** | **DML** (데이터 조작) | **DDL** (데이터 정의) | **DDL** (데이터 정의) |
| **삭제 대상** | 데이터 (행 단위) | 데이터 (전체) | **테이블 전체** (구조+데이터) |
| **저장 공간** | 반납 안 함 (비워만 둠) | **반납 함** (초기화) | **반납 함** (완전 제거) |
| **속도** | 느림 (로그 기록) | **빠름** | 빠름 |
| **복구(Rollback)** | **가능** | **불가능** | **불가능** |
| **WHERE절** | 사용 가능 | 사용 불가 | 사용 불가 |

-----

## 4\. 예시 (Example) 📌

### ✅ 시나리오: 부모-자식 테이블 삭제 테스트

**1. 일반 삭제 시도 (실패 상황)**

```sql
-- DEPT(부모)를 EMP(자식)가 참조 중일 때
DROP TABLE DEPT; 
-- ❌ 에러 발생: ORA-02449: 외래 키에 의해 참조되는 고유/기본 키가 테이블에 있습니다.
```

**2. 강제 삭제 (CASCADE 옵션)**

```sql
-- 참조하는 제약조건(FK)을 끊어버리고 부모 테이블 삭제
DROP TABLE DEPT CASCADE CONSTRAINTS;
-- ✅ 성공: EMP 테이블의 FK 제약조건이 사라지고 DEPT 테이블은 삭제됨.
```

**3. 데이터 초기화 및 완전 삭제**

```sql
-- EMP 테이블의 껍데기는 남기고 데이터만 싹 비우기 (빠름)
TRUNCATE TABLE EMP;

-- EMP 테이블을 휴지통 없이 완전히 삭제
DROP TABLE EMP PURGE;
```

-----

**수정 사항:**

  * 단순히 `DROP`만 다루지 않고, 혼동하기 쉬운 `TRUNCATE`를 함께 정리했습니다.
  * `DELETE`, `TRUNCATE`, `DROP`의 차이점을 **비교 표**로 만들어 한눈에 파악하도록 했습니다.

**제안 사항:**

  * **실무 팁**: 운영 중인 DB에서는 `DROP`이나 `TRUNCATE` 권한이 제한되는 경우가 많습니다. 데이터를 삭제할 때는 실수 방지를 위해 항상 `SELECT`로 대상을 먼저 확인하고 `DELETE`를 사용하는 것이 안전합니다.
