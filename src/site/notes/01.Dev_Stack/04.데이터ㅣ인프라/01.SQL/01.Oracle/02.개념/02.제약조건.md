# SQL 제약 조건(Constraints)의 종류와 활용법

> [!summary]
> **제약 조건(Constraints)**은 데이터베이스의 테이블에 올바른 데이터만 입력되도록 강제하는 **품질 관리 규칙**입니다. 데이터의 무결성(Integrity)을 유지하여 비즈니스 로직을 DB 수준에서 1차적으로 방어하는 역할을 합니다.

---

## 1. 주요 제약 조건의 종류 및 특징

데이터의 품질을 유지하기 위해 사용되는 6가지 핵심 제약 조건입니다.

| 제약 조건                | 핵심 역할    | 주요 특징 및 활용                                    |
| -------------------- | -------- | --------------------------------------------- |
| **PRIMARY KEY** (PK) | 행 식별자    | 중복 불가(`Unique`), 비어있음 불가(`NOT NULL`), 테이블당 1개 |
| **FOREIGN KEY** (FK) | 관계 형성    | 다른 테이블의 PK를 참조하여 데이터 간의 연결고리 생성               |
| **NOT NULL**         | 필수 입력    | 빈 값을 허용하지 않음 (이름, 아이디 등 필수 정보)                |
| **UNIQUE**           | 중복 방지    | 동일한 값이 들어올 수 없음 (이메일, 휴대폰 번호 등)               |
| **CHECK**            | 값의 범위 제한 | 조건에 맞는 데이터만 허용 (예: 나이 > 0, 성별 'M'/'F')        |
| **DEFAULT**          | 기본값 할당   | 값이 입력되지 않을 시 자동으로 채워지는 값 (예: 가입일)             |

---

## 2. 테이블 간의 연결: PK-FK 관계

데이터베이스에서 테이블은 서로 독립적으로 존재하지 않고 관계를 맺습니다. 이때 **부모 테이블의 PK를 자식 테이블이 FK로 가져와** 연결합니다.
### 인덱스(index)
`CREATE INDEX 인덱스명 ON 테이블명(컬럼명);`
- JOIN이나 WHERE절 또는 ORDER BY 절에서 자주 사용하는 컬럼명을 INDEX로 지정해두면 성능향상이 된다.

|**항목**|**장점 (PROS)**|**단점 (CONS)**|
|---|---|---|
|**조회 (SELECT)**|특정 조건을 만족하는 데이터를 **비약적으로 빠르게** 찾습니다.|인덱스가 너무 많으면 오히려 오라클의 최적화 도구(OPTIMIZER)가 혼란을 겪을 수 있습니다.|
|**정렬/그룹화**|이미 정렬되어 있어 `ORDER BY`나 `GROUP BY`가 매우 빨라집니다.|**데이터 추가/수정/삭제 (INSERT, UPDATE, DELETE)** 시 인덱스도 매번 다시 정렬해야 하므로 성능이 저하됩니다.|
|**조인 (JOIN)**|테이블 간의 관계를 맺을 때 연결 고리 역할을 수행하여 성능을 높입니다.|추가적인 **디스크 저장 공간**이 필요합니다. (보통 테이블 크기의 10~20% 내외)|



### 데이터 무결성 유지를 위한 삭제 정책

외래키(FK) 설정 시 부모 데이터가 삭제될 때의 행동을 지정할 수 있습니다.

* **ON DELETE CASCADE**: 부모 데이터 삭제 시 이를 참조하는 자식 데이터도 **함께 삭제**됩니다.
* **ON DELETE SET NULL**: 부모 데이터 삭제 시 자식 데이터의 해당 컬럼을 **NULL로 변경**합니다. (데이터 이력을 남겨야 할 때 사용)

## 3. 실무 활용 가이드

1. **식별자 설정**: 모든 테이블에는 고유한 **PK**를 부여하여 데이터 관리의 기준을 세웁니다.
2. **데이터 방어**: 잘못된 데이터 입력을 애플리케이션(Java 등)뿐만 아니라 DB 레벨에서 **CHECK, UNIQUE**로 이중 차단합니다.
3. **코드 효율화**: **DEFAULT**를 활용하여 시스템이 자동으로 입력할 수 있는 데이터(날짜, 초기값 0 등)에 대한 코딩 수고를 덜어줍니다.

## 4. 구현 예시


> [!EXAMPLE] 개념 비유
> **부모(MEMBER)**는 '학생'이고, **자식(APPLY)**은 '학생증'과 같습니다. 학생이 전학(삭제)가면 학생증도 폐기(CASCADE)하거나 반납 처리(SET NULL)해야 하는 것과 같은 원리입니다.

```sql
-- 1단계: 부모 테이블 (MEMBER)
CREATE TABLE MEMBER (
    USER_ID VARCHAR2(20) PRIMARY KEY,       -- 기본키: 회원 식별
    USER_NAME VARCHAR2(30) NOT NULL,        -- 이름은 반드시 필요
    EMAIL VARCHAR2(100) UNIQUE,             -- 이메일 중복 방지
    STATUS CHAR(1) DEFAULT 'Y' CHECK (STATUS IN ('Y', 'N')) -- 상태값 제한 및 기본값
);

-- 2단계: 자식 테이블 (VOLUNTEER_APPLY)
CREATE TABLE VOLUNTEER_APPLY (
    APPLY_NO NUMBER PRIMARY KEY,            -- 신청 번호
    USER_ID VARCHAR2(20),                   -- 회원 ID (FK로 사용될 컬럼)
    APPLY_DATE DATE DEFAULT SYSDATE,        -- 신청일 기본값: 현재 시간
    -- FK 연결 및 삭제 옵션 설정
    CONSTRAINT FK_MEMBER_TO_APPLY FOREIGN KEY (USER_ID) 
    REFERENCES MEMBER(USER_ID) ON DELETE CASCADE
);

```
