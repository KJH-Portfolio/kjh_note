# 📘 객체지향 설계 5원칙 (SOLID)
> [!summary] 핵심 요약
> 
> - **개념**: 로버트 C. 마틴이 정립한 객체지향 프로그래밍 및 설계의 5가지 기본 원칙
>     
> - **왜?**: 시간이 지나도 유지보수와 확장이 쉬운, 유연하고 결합도 낮은 소프트웨어 아키텍처를 구축하기 위함
>     
> - **언제?**: 클래스, 모듈, 함수 등의 구조를 새롭게 설계하거나 기존의 경직된 코드를 리팩토링할 때
>     
> - **핵심**: 시스템의 결합도는 낮추고(Loose Coupling) 응집도는 높이는(High Cohesion) ==객체지향 설계의 절대 기준점== 역할 수행
>     
## 1. 단일 책임 원칙 (SRP: Single Responsibility Principle)
- **개념**
    - 하나의 클래스는 단 하나의 책임(기능)만 가져야 함
    - ==클래스를 변경해야 하는 이유는 단 하나여야 함==
- **어떻게?**
    - 여러 액터(사용자)가 얽힌 갓 클래스(God Class)를 역할별로 분리하여 수정 시 파급 효과 최소화
```plaintext
[ SRP 위반 (God Class) ]
사용자 클래스 (User)
 ├── 회원 정보 관리 로직 (상태)
 ├── 데이터베이스 저장 로직 (DB 연동)
 └── 비밀번호 암호화 로직 (보안)
 ◀── 세 가지 이유로 클래스가 변경될 수 있음 (유지보수 악몽)

[ SRP 준수 (책임 분리) ]
사용자 정보 (User)       ---> 순수 데이터 관리
사용자 DB (UserRepository) ---> DB 저장 전담
보안 모듈 (AuthService)   ---> 암호화 전담
```
## 2. 개방-폐쇄 원칙 (OCP: Open-Closed Principle)
- **개념**
    - 소프트웨어 요소(클래스, 모듈, 함수 등)는 ==확장에는 열려(Open) 있어야 하고, 변경에는 닫혀(Closed) 있어야 함==
- **어떻게?**
    - 기존의 코드를 수정하지 않고도 애플리케이션의 기능을 확장할 수 있도록 다형성(인터페이스, 추상 클래스)을 적극 활용

|**구분**|**의미**|**구현 방식**|
|---|---|---|
|**확장 (Open)**|새로운 기능의 추가가 자유로움|새로운 자식 클래스(구현체) 생성|
|**변경 (Closed)**|새로운 기능 추가 시 기존 코드 수정 불필요|클라이언트 코드가 인터페이스(추상화)에 의존|

```java
// OCP를 준수한 결제 시스템 설계 예시
public interface PaymentStrategy {
    void pay(int amount);
}

// 1. 기존 코드 (수정될 필요 없음)
class CreditCardPayment implements PaymentStrategy {
    public void pay(int amount) { System.out.println("신용카드 결제"); }
}

// 2. 새로운 요구사항: 카카오페이 추가 (확장)
// 기존의 PaymentStrategy나 CreditCardPayment 코드 수정 없이 기능 추가 완료
class KakaoPayment implements PaymentStrategy {
    public void pay(int amount) { System.out.println("카카오페이 결제"); }
}
```
## 3. 리스코프 치환 원칙 (LSP: Liskov Substitution Principle)
- **개념**
    - 하위 객체는 상위 객체의 규약을 깨뜨리지 않고 ==완전히 대체 가능==해야 함
    - 상속의 올바른 사용 기준을 제시하는 원칙 (단순 코드 재사용 목적의 상속 금지)
- **어떻게?**
    - 부모 클래스가 의도한 계약(메서드의 파라미터, 반환값, 예외 처리, 논리적 동작)을 자식 클래스가 위반하지 않도록 설계
> [!warning] LSP 위반의 대표적 사례: 직사각형과 정사각형 문제
> 
> 부모인 직사각형의 규약(가로와 세로가 독립적으로 변함)을 자식인 정사각형(가로와 세로가 항상 같음)이 오버라이딩으로 깨뜨려, 다형성을 활용한 부모 참조 시 런타임 논리 오류 발생. 이 경우 상속이 아닌 독립된 타입으로 분리해야 함.
## 4. 인터페이스 분리 원칙 (ISP: Interface Segregation Principle)
- **개념**
    - 클라이언트는 자신이 사용하지 않는 메서드에 의존하도록 강제되어서는 안 됨
- **어떻게?**
    - 범용적인 거대 인터페이스 하나보다는, 특정 클라이언트를 위한 ==목적별 인터페이스 여러 개로 분리==
```plaintext
[ ISP 위반 (비대한 인터페이스) ]
다기능 복합기 인터페이스
 ├── 인쇄()  <-- 프린터 클라이언트는 이것만 필요함
 ├── 스캔()  <-- 스캐너 클라이언트는 이것만 필요함
 └── 팩스()  <-- 불필요한 메서드까지 의존(구현)해야 하는 문제 발생

[ ISP 준수 (인터페이스 분리) ]
프린터 인터페이스 ── 인쇄() ──▶ 프린터 클라이언트
스캐너 인터페이스 ── 스캔() ──▶ 스캐너 클라이언트
팩스 인터페이스   ── 팩스() ──▶ 팩스 클라이언트
```
## 5. 의존성 역전 원칙 (DIP: Dependency Inversion Principle)
- **개념**
    - 고수준 모듈(비즈니스 핵심 로직)은 저수준 모듈(세부 구현)에 의존해서는 안 되며, 둘 다 추상화에 의존해야 함
    - ==구현 클래스가 아닌 역할(인터페이스)에 의존==해야 함
- **어떻게?**
    - 변하기 쉬운 구체 클래스(Concrete Class) 대신, 변하지 않는 인터페이스(Interface)를 주입받아 사용

| **구분**     | **개념**               | **예시**          | **비고**         |
| ---------- | -------------------- | --------------- | -------------- |
| **고수준 모듈** | 시스템의 핵심 정책 및 비즈니스 로직 | 운전자, 자동차        | 변하지 않는 추상적인 흐름 |
| **저수준 모듈** | 핵심 로직을 돕는 세부 구현체     | 스노우 타이어, 일반 타이어 | 변하기 쉬운 구체적인 기술 |

```java
// [DIP 위반] 자동차가 구체적인 스노우 타이어에 직접 의존 (강결합)
class Car {
    SnowTire tire = new SnowTire(); 
}

// [DIP 준수] 자동차가 '타이어'라는 인터페이스(추상화)에 의존 (느슨한 결합)
class Car {
    Tire tire; // 외부에서 SnowTire나 NormalTire 등 어떤 타이어든 주입(DI) 가능
    
    Car(Tire tire) {
        this.tire = tire;
    }
}
```
