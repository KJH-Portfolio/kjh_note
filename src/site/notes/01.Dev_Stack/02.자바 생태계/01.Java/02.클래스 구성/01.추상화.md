# 객체 지향의 핵심: 추상화 (Abstraction)
> [!summary] 핵심 요약
> 
> - 주제: 현실 세계의 복잡성을 제거하고 핵심 속성과 동작만을 추출하여 모델링하는 기법
>     
> - 목적: 인터페이스와 구현의 분리를 통한 ==복잡성 제어== 및 시스템 유연성 확보
>     
> - 핵심: ==인터페이스(Interface)== 및 추상 클래스를 활용한 계약 기반 설계 준수
>     
## 추상화의 가치 및 기대 효과

| **항목**      | **설명**                          | **기대 효과**                     |
| ----------- | ------------------------------- | ----------------------------- |
| **복잡성 관리**  | 불필요한 세부 사항 제거 및 핵심 논리 집중        | 코드 명료성 및 가독성 확보               |
| **유연한 확장성** | 구현체에 직접 의존하지 않는 구조 설계           | ==느슨한 결합(Loose Coupling)== 구현 |
| **테스트 용이성** | 실제 객체 대신 모의 객체(Mock) 주입 가능      | 단위 테스트 효율성 및 독립성 증대           |
| **재사용성 향상** | 공통 속성 및 행위의 일반화(Generalization) | 코드 중복 최소화 및 ==유지보수 비용 절감==    |

---
## 적정 추상화 계층 결정을 위한 3대 원칙
- **==YAGNI (You Ain't Gonna Need It)==**
    - 현재 요구사항에 없는 미래 확장성을 대비한 과도한 선제적 추상화 지양
    - 실제 코드 중복이 발생하는 시점을 중심으로 모델링 수행
- **Rule of Three (3회의 법칙)**
    - 유사한 사례나 중복 로직이 ==3개 이상 도출되는 시점==을 일반화의 적기로 판단
    - 성급한 일반화로 인한 설계 왜곡 오류 방지
- **인지적 거리(Cognitive Distance) 관리**
    - 인터페이스와 실제 구현체 사이의 ==계층 깊이를 최대 2~3단계 이내==로 제한
    - 과도한 계층화로 인한 코드 추적 난이도 상승 및 가독성 저하 방지
---
## 설계 결정 매트릭스 (Decision Tree)
- **추상화 도입 여부 판단 흐름**
```plaintext
Abstraction Decision Tree
 ├─ 유지보수 시 변경 가능성이 높은가?
 │   ├─ Yes: 인터페이스 도입 (추상화 수행)
 │   └─ No: 구체 클래스(Concrete Class) 유지
 │
 ├─ 다양한 형태의 구현체가 필요한가? (다형성)
 │   ├─ Yes: 다형성 활용을 위한 상위 타입 정의
 │   └─ No: 단일 책임 클래스로 단순화
 │
 └─ 외부 시스템/라이브러리에 의존적인가?
     ├─ Yes: 어댑터 및 인터페이스로 의존성 격리
     └─ No: 내부 도메인 모델로 직접 구현
```

## 핵심 메커니즘 및 Java 구현 모델
- **추출 및 은닉**: 도메인 필수 속성 선별 및 내부 로직 격리를 통한 응집도 향상
- **==구성 우선 원칙 (Composition over Inheritance)==**: 상속을 통한 수직적 확장보다 객체 위임을 통한 수평적 결합 권장
### Java 기반 추상화 구현 예시
```java
//추상화된 설계: 차량 공통 인터페이스
 
public interface Vehicle {
    String getName();    // 차량의 이름 반환
    void drive();        // 주행 동작 계약
    int getMaxSpeed();   // 속성 정보 제공 계약
}

//구체 구현체: 자동차(Car)
public class Car implements Vehicle {
    private final String name;
    private final int maxSpeed;

    public Car(String name, int maxSpeed) {
        this.name = name;
        this.maxSpeed = maxSpeed;
    }

    @Override
    public String getName() {
        return name;
    }

    @Override
    public void drive() {
        System.out.println(name + "(이)가 최대 시속 " + maxSpeed + "km/h로 주행합니다.");
    }

    @Override
    public int getMaxSpeed() {
        return maxSpeed;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Car car = (Car) o;
        // 이름과 최고속도가 모두 같아야 같은 차량으로 판단
        return maxSpeed == car.maxSpeed && 
               java.util.Objects.equals(name, car.name);
    }

    @Override
    public int hashCode() {
        return java.util.Objects.hash(name, maxSpeed);
    }
}
```
#### 수직이 아닌 수평적 결합 예시(권장 방식)
```JAVA
// 부품들 (인터페이스)
interface MoveBehavior { void move(); }
interface WorkBehavior { void work(); }

// 실제 부품들
class Walk implements MoveBehavior { public void move() { System.out.println("걷기"); } }
class Fly implements MoveBehavior { public void move() { System.out.println("날기"); } }
class Cleaning implements WorkBehavior { public void work() { System.out.println("청소하기"); } }

// 로봇 클래스 (구성을 사용)
class CustomRobot {
    private MoveBehavior moveBehavior; // 부품을 가짐 (Has-a)
    private WorkBehavior workBehavior;

    public CustomRobot(MoveBehavior m, WorkBehavior w) {
        this.moveBehavior = m;
        this.workBehavior = w;
    }

    void perform() {
        moveBehavior.move();
        workBehavior.work();
    }
}

public class Main {
    public static void main(String[] args) {
        
        // 1. 걷고 청소하는 로봇 조립
        System.out.println("=== 1번 로봇 제작 ===");
        CustomRobot robot1 = new CustomRobot(new Walk(), new Cleaning());
        robot1.perform(); 
        // 출력: 걷기 / 청소하기

        System.out.println("\n=== 2번 로봇 제작 ===");
        // 2. 날고 청소하는 로봇 조립 (클래스를 새로 만들 필요 없이 부품만 교체!)
        CustomRobot robot2 = new CustomRobot(new Fly(), new Cleaning());
        robot2.perform();
        // 출력: 날기 / 청소하기

        // 3. 만약 새로운 동작(예: 요리)이 추가된다면?
        // interface WorkBehavior를 구현한 'Cooking' 클래스 하나만 만들어서 
        // 바로 CustomRobot에 넣어주면 끝납니다.
    }
}
```
