# 객체 지향의 핵심: 추상화 (Abstraction)
> [!summary] 핵심 요약
> 
> - **추상화**: 복잡한 현실 세계에서 핵심 속성과 동작만을 추출하여 모델링하는 기법
>     
> - **목적**: ==복잡성 제어== 및 인터페이스와 구현 분리를 통한 시스템 유연성 확보
>     
> - **핵심**: 인터페이스(Interface) 및 추상 클래스(Abstract Class)를 통한 계약 기반 설계
>     
## 추상화의 가치 및 적정 수준 가이드라인
### 1. 추상화의 목적 및 기대 효과

| **항목**  | **설명**                                     |
| ------- | ------------------------------------------ |
| 복잡성 관리  | 불필요한 세부 사항 제거 및 논리적 명료성 확보                 |
| 유연한 확장성 | 구현체에 의존하지 않는 ==느슨한 결합(Loose Coupling)== 구현 |
| 테스트 용이성 | 모의 객체(Mock) 활용 및 단위 테스트 효율성 증대             |
| 재사용성 향상 | 공통 속성과 행위의 그룹화를 통한 코드 중복 최소화               |

### 2. 적정 계층 결정을 위한 3대 원칙
- **==YAGNI (You Ain't Gonna Need It)==**: 현재 요구사항에 없는 미래 확장성을 고려한 선제적 추상화 지양 및 실제 중복 발생 시점 중심의 모델링
- **Rule of Three (3회의 법칙)**: 유사 사례가 3개 이상 도출되는 시점을 일반화의 적기(Sweet Spot)로 판단하여 성급한 일반화 오류 방지
- **인지적 거리(Cognitive Distance) 관리**: 인터페이스와 구현체 사이의 ==계층 깊이를 최대 2~3단계 이내==로 제한하여 코드 추적 가능성 및 가독성 확보

## 설계 결정 매트릭스 (Decision Matrix)
```plaintext
Abstraction Decision Tree
 ├─ 변경 가능성이 높은가?
 │   ├─ Yes: 인터페이스 도입 (추상화)
 │   └─ No: 구체 클래스 유지
 ├─ 다양한 구현체가 필요한가?
 │   ├─ Yes: 다형성 활용 (추상화)
 │   └─ No: 단일 책임 클래스 (구체화)
 └─ 외부 라이브러리/시스템 의존적인가?
     ├─ Yes: 어댑터/인터페이스로 격리 (추상화)
     └─ No: 내부 도메인 모델로 구현
```

## 핵심 메커니즘 및 구현 모델
- **추출 및 은닉**: 도메인 필수 속성 선별 및 내부 로직 격리를 통한 응집도 향상
- **구성 우선 원칙 (Composition over Inheritance)**: 상속을 통한 수직적 확장보다 객체 위임을 통한 수평적 결합 권장
### Java 기반 구현 예시
```java
// 추상화된 설계: 차량 공통 인터페이스
public interface Vehicle {
    void drive();        // 동작 계약
    int getMaxSpeed();   // 속성 접근 계약
}

// 구체 구현체: 자동차
public class Car implements Vehicle {
    private int maxSpeed;

    public Car(int maxSpeed) { 
        this.maxSpeed = maxSpeed; 
    }

    @Override
    public void drive() { 
        System.out.println("자동차 주행"); 
    }

    @Override
    public int getMaxSpeed() { 
        return maxSpeed; 
    }
}
```
