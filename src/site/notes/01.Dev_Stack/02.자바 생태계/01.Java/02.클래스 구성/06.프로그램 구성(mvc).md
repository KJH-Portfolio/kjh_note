# Java 계층형 아키텍처 및 도메인 모델 심화 설계
> [!summary] 핵심 요약
> 
> - 내용: 계층간 역할 분리(SoC) 및 VO(Value Object)의 로직 포함 여부에 따른 설계 모델 정의
>     
> - 목적: 서비스 계층의 비대화 방지 및 객체지향적 응집도 향상을 통한 유지보수성 확보
>     
> - 핵심: 데이터와 로직이 결합된 ==풍부한 도메인 모델(Rich Domain Model)== 지향 및 계층 간 결합도 최소화
>     
## 1. 계층형 아키텍처 구성 요소 (Refined)
- **계층별 책임 및 데이터 흐름**
    - 각 계층은 고유의 목적을 가지며 단방향으로 통신하여 시스템 결합도 완화

|**구성 요소**|**핵심 역할**|**데이터 흐름 및 로직 위치**|
|---|---|---|
|**VO / Domain**|데이터 본연의 가치 저장|==자체 데이터 기반 연산 및 검증 로직== 포함|
|**DAO**|저수준 데이터 접근 (DB)|SQL 실행 및 `ResultSet`의 객체 매핑 전담|
|**Service**|비즈니스 프로세스 조율|여러 도메인 객체 협력을 관리하는 오케스트레이터|
|**Controller**|클라이언트 접점 제어|사용자 입력 유효성 검사 및 서비스 호출 분기|
|**View / Run**|인터페이스 및 실행 환경|정보 시각화 및 프로그램 생명주기 관리 시작점|

## 2. VO(도메인) 설계 패턴 비교: 빈약함 vs 풍부함
- **도메인 모델 유형별 특성**
    - 도메인 지식의 캡슐화 수준에 따른 아키텍처 품질 차이

|**구분**|**빈약한 도메인 모델 (Anemic)**|**풍부한 도메인 모델 (Rich)**|
|---|---|---|
|**VO 내부**|Getter / Setter만 존재|데이터 및 ==비즈니스 로직== 포함|
|**로직 위치**|모든 로직이 Service에 집중|객체 스스로 해결 가능한 로직 내포|
|**Service 상태**|로직 과부하로 인한 비대화 발생|계층 간 협력 조율 역할로 경량화|
|**응집도**|낮음 (데이터와 로직 분리)|==높음 (데이터와 로직 결합)==|

- **VO 내 삽입 권장 로직 (Self-Containment)**
    - 자체 유효성 검증: 필드 값의 범위, 형식, 필수 여부 확인 (예: `price < 0` 예외 발생)
    - 도메인 계산: 객체 내부 상태를 이용한 가공 (예: 수량과 단가를 통한 총액 계산)
    - 상태 전이 판단: 특정 상태로의 변경 가능 여부 확인 (예: 결제완료 상태에서만 배송시작 가능)
## 3. 로직 배치 설계 및 주의사항 (Guardrails)
- **VO 내부 배치 금지 로직 (Anti-Patterns)**
> [!warning] 설계상 주의사항(시스템 자원 절약 법칙)
> 
> 1. **외부 의존성 포함 금지**: VO 내부에서 ==DAO 호출 또는 DB 접근== 엄격히 금지
>     
> 2. **인프라 로직 배제**: API 호출, 파일 입출력 등 시스템 인프라 관련 코드 삽입 금지
>     
> 3. **다수 객체 연동**: 여러 VO가 얽힌 복합 프로세스는 ==Service 계층==에서 처리
>     
- **도메인 모델 확장 구조도**
    - Service는 흐름을 제어하고, 핵심 판단은 도메인 객체가 수행하는 이상적 구조
```plaintext
Domain-Driven Layered Structure
 ├─ [Service Layer]: 전체 트랜잭션 및 도메인 간 흐름 제어
 │   └─ "주문 처리 시작" -> (A)재고확인 -> (B)결제진행 -> (C)주문생성
 │
 └─ [Domain Layer (VO)]: 개별 객체의 데이터 주권 확보
     ├─ ProductVO: 할인율 적용 계산, 재고 부족 여부 판단
     ├─ MemberVO: 등급별 혜택 검증, 비밀번호 일치 확인
     └─ OrderVO: 최종 결제 금액 산출, 주문 상태 유효성 검사
```
## 4. 실전 적용 가이드 및 데이터 이동
- **객체지향적 설계 실무 적용**
    - 상태 검증: Controller에서 전달된 원시 데이터를 VO 생성 단계에서 ==자가 검증(Self-Validation)== 수행
    - 비즈니스 연산: Service가 VO의 메서드를 호출하는 객체지향적 메시징 구조 작성
    - 재사용성: 특정 VO에 종속된 로직의 내부 응집으로 다양한 서비스에서의 ==코드 중복 제거==
    - 테스트 용이성: DB 연결 없이 VO 내부 메서드만으로 독립적인 ==단위 테스트(Unit Test)== 수행
