# Java 람다식(Lambda Expression) 실무 활용 패턴
> [!summary] 핵심 요약
> 
> - **개념**: 단 하나의 추상 메서드를 갖는 인터페이스(함수형 인터페이스)를 익명 내부 클래스 대신 간결하게 표현하는 문법
>     
> - **왜?**: 보일러플레이트 코드(불필요한 익명 클래스 선언)를 제거하고, 핵심 비즈니스 로직(What)에 집중하여 가독성 극대화
>     
> - **언제?**: 컬렉션 데이터를 조작(Stream API)하거나, 메서드에 특정 동작(로직) 자체를 파라미터로 전달해야 할 때
>     
> - **핵심**: 동작 파라미터화 구현, `this` 스코프 명확화 및 ==Effective Final 제약==을 통한 메모리 안정성 확보
>     
## 1. 익명 내부 클래스 대체 및 패러다임 전환
- **익명 내부 클래스의 한계와 람다식의 등장**
    - 이름이 없는 일회용 객체인 ==익명 내부 클래스의 장황한 코드 구조 개선==
    - 단일 추상 메서드(SAM, Single Abstract Method)를 갖는 '함수형 인터페이스' 구현 시 람다식으로 완벽히 대체
    - 데이터를 다루는 관점이 객체(상태) 중심에서 동작(함수) 중심으로 이동
    - **어떻게?**: `new 인터페이스명()` 및 메서드 오버라이딩 선언부를 모두 생략하고 `(매개변수) -> {실행문}` 구조로 압축
```plaintext
[ 익명 내부 클래스 vs 람다식 코드 압축 구조 ]

     [기존: 익명 내부 클래스 (Anonymous Inner Class)]       [개선: 람다식 (Lambda Expression)]
                                                      │
 Runnable task = new Runnable() {                     │
     @Override                                        │    Runnable task = () -> System.out.println("실행!");
     public void run() {                              │
         System.out.println("실행!");                 │    (불필요한 인터페이스명, 메서드명 완벽히 소거됨)
     }                                                │
 };                                                   │
```
## 2. 실무 주요 활용 사례 (Top 5)
- **주요 활용 상황 및 목적**
    - **데이터 가공**: 대량의 데이터 컬렉션을 필터링, 변환, 집계 (Stream API 기반)
    - **동작 파라미터화**: 특정 조건이나 로직 자체를 메서드의 인자로 전달 (Strategy Pattern)
    - **지연 실행(Lazy Execution)**: 즉시 실행하지 않고, 특정 이벤트나 조건 발생 시점의 콜백(Callback)으로 지정
    - **어떻게?**: `Predicate`, `Consumer`, `Comparator` 등 Java 내장 함수형 인터페이스와 결합하여 비즈니스 로직 구현

|**활용 분야**|**주요 메서드/인터페이스**|**특징 및 목적**|
|---|---|---|
|**1. 컬렉션 조작**|`forEach()`, `removeIf()`|명시적 반복문(for, while) 없이 간결한 순회 및 조건부 요소 삭제 처리|
|**2. Stream API**|`filter()`, `map()`, `collect()`|데이터 파이프라인 구축 시 필수 요소 (람다식과 가장 강력한 시너지 발휘)|
|**3. 정렬 (Sorting)**|`Comparator`|복잡한 객체 비교 로직을 한 줄의 수학적 표현식으로 압축|
|**4. 멀티스레딩**|`Runnable`|비동기 실행을 위한 작업 단위(Task)를 초경량으로 정의 및 실행|
|**5. 이벤트 핸들링**|`ActionListener` 등|GUI 환경 등에서 사용자 액션(클릭 등)에 따른 반응형 로직 직관적 구현|

```java
// [1] 컬렉션 조작 (Collection)
List<Integer> scores = new ArrayList<>(Arrays.asList(90, 75, 85, 60, 100));
scores.forEach(score -> System.out.println(score)); // 전체 순회 출력
scores.removeIf(score -> score < 80);               // 80점 미만 조건부 삭제

// [2] Stream API 활용 (필터링 및 변환 파이프라인)
List<String> names = Arrays.asList("김철수", "이영희", "김민수");
List<String> result = names.stream()
    .filter(name -> name.startsWith("김"))  // '김'씨 성을 가진 요소 필터링
    .map(name -> name + "님")               // 문자열 변환
    .collect(Collectors.toList());          // 최종 리스트로 수집

// [3] 정렬 (Comparator)
// 기존 익명 클래스(new Comparator(){...}) 구조를 완벽히 대체한 직관적 객체 정렬
people.sort((p1, p2) -> p1.getAge() - p2.getAge()); 

// [4] 스레드(Thread) 생성 및 실행 (Runnable 구현)
new Thread(() -> System.out.println("비동기 작업 실행!")).start();

// [5] GUI 이벤트 핸들링 (버튼 클릭 콜백)
JButton button = new JButton("Click");
button.addActionListener(e -> System.out.println("버튼 클릭됨!"));
```
## 3. 람다식과 익명 내부 클래스의 결정적 차이 (Scope Binding)
- **`this` 키워드의 해석 기준 분리**
    - 람다식은 익명 클래스를 단순히 짧게 쓴 것이 아니라, ==스코프(Scope)를 처리하는 방식이 근본적으로 다름==
    - 익명 내부 클래스: 자신만의 고유한 객체 스코프를 생성하여 `this`는 익명 클래스 자체를 가리킴
    - 람다식 (Lexical Scope): 새로운 객체 스코프를 만들지 않고 람다식을 감싸고 있는 외부 블록의 스코프를 그대로 공유
    - **어떻게?**: 외부 클래스의 상태(멤버 변수)에 접근해야 할 때 람다식을 사용하면 코드가 훨씬 직관적으로 동작함
```plaintext
[ this 키워드 참조 영역 (Scope Binding) ]

 외부 클래스 (OuterClass) ──────────┐ [this] ◀── 동일한 객체로 인식
 │                                  │  ▲
 │  ┌── 익명 내부 클래스 ──────┐    │  │ (람다의 this는 외부를 향함)
 │  │ this ──▶ 자기 자신 지칭  │    │  │
 │  └──────────────────────────┘    │  │
 │                                  │  │
 │  ┌── 람다식 (Lambda) ───────┐    │  │
 │  │ this ────────────────────┼────┘
 │  └──────────────────────────┘
 └──────────────────────────────────┘
```
```java
public class ScopeExample {
    private String value = "외부 클래스(Outer)";

    public void testScope() {
        // 1. 익명 내부 클래스의 this
        Runnable innerClass = new Runnable() {
            private String value = "익명 클래스(Inner)";
            @Override
            public void run() {
                // this는 Runnable 익명 객체 자신을 가리킴
                System.out.println(this.value); // 결과: "익명 클래스(Inner)"
            }
        };

        // 2. 람다식의 this (Lexical Scope)
        Runnable lambda = () -> {
            // 람다식은 자신만의 독립된 this를 갖지 않음
            // this는 ScopeExample 객체(외부)를 가리킴
            System.out.println(this.value); // 결과: "외부 클래스(Outer)"
        };

        innerClass.run();
        lambda.run();
    }
}
```

|**비교 항목**|**익명 내부 클래스 (Anonymous Class)**|**람다식 (Lambda Expression)**|
|---|---|---|
|**`this`의 의미**|==익명 클래스 객체 자신==|==외부 클래스 객체== (Lexical Scope)|
|**독립적 상태 유지**|내부에 인스턴스 변수 선언 가능|내부에 인스턴스 변수 선언 불가|
|**컴파일 산출물**|`Outer$1.class` 형태의 파일 별도 생성|별도의 클래스 파일 생성 없음 (런타임 동적 생성)|

## 4. 외부 변수 참조 제약 (Effective Final 메커니즘)
- **자유 변수 캡처 (Variable Capture) 현상**
    - 람다식이나 익명 내부 클래스 내부에서 외부 메서드의 지역 변수를 참조하는 동작
    - **어떻게?**: 외부 지역 변수의 값을 복사하여 람다식 내부(Heap 영역)에 보관하여 사용
- **Effective Final의 정의와 필요성**
    - 명시적으로 `final` 키워드를 붙이지 않아도, ==값이 한 번 할당된 후 변경되지 않는 상태==를 의미
    - 생명주기 불일치 해결: 지역 변수(Stack)는 메서드 종료 시 즉각 소멸하나, 람다식(Heap)은 객체로서 더 오래 생존할 수 있음
    - 동시성 문제 예방: 여러 스레드에서 람다식이 실행될 때 원본 값과 복사본 간의 ==데이터 오염(Desync) 방지==
    - **어떻게?**: 컴파일러가 해당 변수의 수정 시도를 감지하면 즉시 컴파일 에러를 발생시켜 안전성 강제
```plaintext
[ 지역 변수 캡처(Capture)와 생명주기 문제 해결 ]

   Stack (메서드 실행 흐름)                 Heap (객체 메모리)
 ┌────────────────────────┐             ┌────────────────────────┐
 │ captureExample() 프레임│             │                        │
 │                        │             │  [람다식 인스턴스]     │
 │  count = 10 ───────────┼─(값 복사)─▶ │    - 캡처된 count = 10 │
 │                        │             │    - 로직 실행 대기    │
 └────────────────────────┘             │                        │
    ▲ (메서드 종료 시 즉시 소멸)        └────────────────────────┘
                                           ▲ (GC 수거 전까지 생존)

 ※ 딜레마: Stack의 원본(count)이 사라져도 Heap의 복사본은 살아남아야 함.
 ※ 해결책: 두 값의 불일치 및 예측 불가능한 버그를 막기 위해, 
           원본과 복사본 모두 '수정 불가(Effective Final)' 상태로 강제함.
```
```java
public void captureExample() {
    int count = 10; // (Effective Final 상태: 초기화 이후 수정되지 않음)
    int sum = 0;    // (이후 로직에서 값이 변경될 변수)

    // 람다식에서 외부 지역 변수(count) 참조
    Runnable r = () -> {
        System.out.println("카운트: " + count); // [성공] 읽기 전용으로 복사(Capture)되어 정상 사용됨
        
        // sum += 10; // [컴파일 에러] 람다식 내부에서 외부 지역 변수 변경 불가
        // count++;   // [컴파일 에러] 마찬가지로 내부에서 수정 불가
    };

    // count = 20; // [컴파일 에러] 원본 값을 수정하는 순간 count는 Effective Final 지위를 상실하여 람다식에서 참조 불가해짐
    sum = 20;      // 람다식 내부에서 참조하지 않는 변수는 자유롭게 수정 가능

    r.run();
}
```
