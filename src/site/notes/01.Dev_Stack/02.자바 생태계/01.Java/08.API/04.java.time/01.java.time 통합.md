# Java Date and Time API (`java.time`) 아키텍처 개요
> [!summary] 핵심 요약
> 
> - **개념**: 도메인 주도 설계(DDD) 원칙을 반영하여 시간의 역할을 명확히 분리한 Java 8의 날짜/시간 표준 아키텍처
>     
> - **왜?**: 기존 `java.util.Date`가 가진 가변성(Mutable) 문제와, '절대 시점'과 '지역 시간'을 하나의 클래스로 뭉뚱그려 처리하던 설계적 결함을 해결하기 위함
>     
> - **언제?**: 글로벌 서비스의 타임존 설계, 데이터베이스 타임스탬프 동기화, 도메인 내 시간 관련 비즈니스 규칙(Rule)을 모델링할 때
>     
> - **핵심**: '기계의 시간(절대 시점)'과 '인간의 시간(지역 달력)'을 완벽히 분리하고, 모든 상태 변경을 **불변 객체(Immutable)**로 제어함
>     
## 1. 코어 설계 철학 (Core Design Philosophy)
- **불변성 보장 (Immutability & Thread-Safety)**
    - 패키지 내의 모든 핵심 클래스는 상태를 변경할 수 없는 불변 객체로 설계됨
    - 멀티스레드 환경에서 자원을 공유하더라도 ==동시성(Concurrency) 이슈가 원천적으로 발생하지 않음==
    - 상태를 변경하는 연산은 내부 값을 수정하는 대신, 연산이 적용된 **새로운 인스턴스를 반환**함
- **유창한 인터페이스 (Fluent API)**
    - 빌더 패턴(Builder Pattern)과 유사하게 메서드 체이닝을 활용하여 가독성 높은 시간 연산 파이프라인 구축 가능
    - `TemporalAdjusters`를 활용해 "이번 달의 마지막 금요일"과 같은 복잡한 도메인 규칙을 선언적으로 작성 가능
## 2. 시간 모델링의 분리 (Separation of Time Context)
- **기계의 시간과 인간의 시간**
    - `java.time` 패키지는 시간을 바라보는 관점을 시스템(Machine)과 사용자(Human)로 분리함
    - **Machine Time**: 타임존의 개념이 없는 우주적 절대 시점 (`Instant`)
    - **Human Time**: 특정 지역의 달력과 시계에 종속되는 상대적 시각 (`LocalDateTime`, `ZonedDateTime`)
    - ==글로벌 아키텍처 설계 시, 내부 시스템(DB, 통신)은 기계의 시간을, 화면 렌더링은 인간의 시간을 사용하는 것이 표준==
```plaintext
[ java.time Architecture Overview ]

      [ Machine Time ]                     [ Human Time ]
      (Absolute Point)                 (Contextual & Floating)
    ┌──────────────────┐                 ┌──────────────────┐
    │     Instant      │ ────+ZoneId───▶ │  ZonedDateTime   │
    │  (UTC Standard)  │                 │ (Absolute+Local) │
    └──────────────────┘                 └──────────────────┘
                                                  ▲
                                                  │ +ZoneId
                                                  │
                                         ┌──────────────────┐
                                         │  LocalDateTime   │
                                         │ (Floating Point) │
                                         └──────────────────┘

 ※ 핵심 요약:
 1. Instant: 전 세계 어디서나 동일한 절대 타임스탬프 (기계용)
 2. LocalDateTime: 달력상의 표기일 뿐, 위치 정보가 없어 절대 시점을 알 수 없음 (로컬용)
 3. ZonedDateTime: 절대 시점과 지역 달력 규칙이 결합된 완전한 시간 객체 (서비스용)
```
## 3. 시스템 경계별 시간 파이프라인 (Data Flow & Pipeline)
- **계층 간 데이터 변환 전략**
    - **Database/Log**: 시간대 파편화를 막기 위해 `Instant`를 사용하거나, UTC 기준의 밀리초 타임스탬프(`long`)로 저장
    - **Domain Layer**: 비즈니스 로직 계산 시 `LocalDateTime` 또는 `ZonedDateTime`을 활용하여 객체지향적 연산 수행
    - **Presentation (API/UI)**: `DateTimeFormatter`를 거쳐 ISO-8601 표준 규격의 문자열로 변환 후 클라이언트에 직렬화 전달
    - **어떻게?**: 계층 간 경계를 넘나들 때마다 포맷터와 ZoneId를 활용해 적절한 책임의 객체로 캐스팅함
```java
// API 응답용 글로벌 표준 포맷 (ISO-8601 기반)
private static final DateTimeFormatter GLOBAL_FORMATTER = DateTimeFormatter.ISO_ZONED_DATE_TIME;

// DB의 절대 시점(Instant) 데이터를 특정 클라이언트의 지역 시간(String)으로 변환하는 파이프라인
public String generateClientResponse(Instant dbTimestamp, String clientZoneId) {
	// 1. 기계의 시간(Instant)에 클라이언트의 맥락(ZoneId)을 주입하여 인간의 시간으로 변환
	ZonedDateTime localizedTime = dbTimestamp.atZone(ZoneId.of(clientZoneId));
	
	// 2. 비즈니스 규칙 적용 (예: 2일 뒤 만료) -> 불변 객체 체이닝
	ZonedDateTime expirationTime = localizedTime.plusDays(2);
	
	// 3. Presentation Layer: 문자열로 직렬화하여 반환
	return expirationTime.format(GLOBAL_FORMATTER);
}
```
