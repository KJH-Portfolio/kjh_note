# Java 날짜 및 시간 포맷팅 (`DateTimeFormatter`)
> [!summary] 핵심 요약
> 
> - **개념**: 날짜 및 시간 객체를 문자열로 변환(Formatting)하거나, 문자열을 객체로 역변환(Parsing)하는 전용 클래스
>     
> - **왜?**: 기존 `SimpleDateFormat`의 스레드 불안정성(Thread-Unsafe)을 극복하기 위해 다중 스레드 환경에서도 데이터가 꼬이지 않는 불변(Immutable) 객체로 설계됨
>     
> - **언제?**: 클라이언트 API 응답으로 날짜를 텍스트로 내보내거나, 외부 시스템에서 전달받은 문자열 날짜 데이터를 애플리케이션 내부용 객체로 읽어들일 때 필수 사용
>     
> - **핵심**: 객체 생성 비용 절감 및 성능 최적화를 위해 **`static final` 상수로 선언하여 애플리케이션 전역에서 재사용**하는 것이 실무 표준
>     
## 포맷터 생성 및 양방향 변환 (Formatting & Parsing)
- **Thread-Safe 구조 및 불변성 보장**
    - 포맷터 인스턴스는 내부 상태가 변하지 않으므로 동시에 여러 스레드가 접근해도 안전함
    - 기본 제공되는 국제 표준 상수(`ISO_LOCAL_DATE`, `ISO_INSTANT` 등)를 우선적으로 활용
    - ==파싱(`parse`) 시 문자열과 포맷터의 패턴이 1자리라도 불일치하면 즉시 `DateTimeParseException` 예외 발생==
    - **어떻게?**: `ofPattern()`으로 변환 규격을 생성하고, 이를 날짜 객체의 `format()`이나 정적 메서드 `parse()`에 주입하여 사용
```plaintext
[ Formatting & Parsing Flow ]

  ┌───────────────┐                  ┌───────────────┐
  │ LocalDateTime │  --- format ---> │    String     │
  │   (Object)    │                  │ ("2026-04-15")│
  └───────────────┘  <--- parse ---- └───────────────┘
                           │
                   ┌───────┴───────┐
                   │   Formatter   │  <-- 변환 기준 (yyyy-MM-dd)
                   └───────────────┘
```
```java
// 1. 포맷터 캐싱
private static final DateTimeFormatter KOR_FORMATTER = 
    DateTimeFormatter.ofPattern("yyyy년 MM월 dd일 HH시 mm분");

public String convertToString(LocalDateTime date) {
    // 2. Formatting: 객체 -> 문자열 변환
    return date.format(KOR_FORMATTER);
}

public LocalDateTime convertToObject(String dateStr) {
    // 3. Parsing: 문자열 -> 객체 역변환
    return LocalDateTime.parse(dateStr, KOR_FORMATTER);
}
```

|**항목**|**동작 원리 및 역할**|**반환 타입 등 기타 속성**|
|---|---|---|
|`ofPattern()`|지정된 문자열 규격을 가진 불변 포맷터 인스턴스 생성|`DateTimeFormatter`|
|`format()`|시간 객체 자신을 지정된 포맷의 **문자열로 출력**|`String`|
|`parse()`|텍스트 데이터를 해석하여 **시간 객체로 복원**|`LocalDateTime`, `LocalDate` 등|

## 패턴 작성 문법 및 주의점 (Pattern Syntax)
- **대소문자 구분의 엄격성**
    - 패턴 작성 시 영문 대소문자에 따라 매핑되는 시간 단위가 완전히 달라짐
    - ==특히 '월(M)'과 '분(m)', '24시간제(H)'와 '12시간제(h)'의 혼동으로 인한 논리적 버그 발생 빈도가 매우 높음==
    - **어떻게?**: 개발팀 내 공통 `Utils` 또는 상수로 표준 패턴을 중앙 집중화하여 파편화와 휴먼 에러를 방지
> [!warning] 주요 패턴 혼동 주의점
> 
> - **`y` (Year)**: 연도 (예: yyyy -> 2026)
>     
> - **`M` (Month)**: 월 (예: MM -> 04) - **반드시 대문자 사용**
>     
> - **`m` (Minute)**: 분 (예: mm -> 30) - **반드시 소문자 사용**
>     
> - **`H` (Hour)**: 24시간제 시 (0~23) - **대문자**
>     
> - **`h` (Hour)**: 12시간제 시 (1~12) - **소문자**
>     
> - **`d` (Day)**: 일 (예: dd -> 15)
>     
## 다국어 지원 및 지역화 (Locale)
- **문자열 기반 시간 데이터의 지역화 처리**
    - 요일(`E`), 월 이름(`MMM`, `MMMM`), 오전/오후(`a`) 등 텍스트로 표현되는 시간 요소는 패턴뿐만 아니라 언어 설정(`Locale`)의 영향을 직접적으로 받음
    - 시스템 기본(System Default) 언어 설정에 의존할 경우, 서버의 OS 환경(한국/미국)에 따라 파싱 및 포맷팅 결과가 달라지는 파편화 문제 발생
    - ==운영체제 환경에 구애받지 않도록 코드 레벨에서 `Locale`을 명시적으로 고정하는 것이 글로벌 서비스의 필수 안전 조치==
    - **어떻게?**: `ofPattern`의 두 번째 인자로 전달하거나, 체이닝 메서드 `.withLocale()`을 사용하여 적용
```plaintext
[ Locale Application Flow ]

  ┌─────────────┐       ┌───────────┐       ┌────────────────────┐
  │   Pattern   │   +   │  Locale   │   =   │ Formatter Instance │
  │("MMM dd, y")│       │ (English) │       │   (Oct 15, 2026)   │
  └─────────────┘       └───────────┘       └────────────────────┘
                              │
                 ※ 한글 환경(Locale.KOREA) 적용 시 -> "10월 15, 2026"
```
```java
// 1. 영문 날짜 텍스트 파싱 (해당 언어의 Locale 지정 필수)
// -> Locale.US 누락 시 기본 시스템이 한국어라면 "Oct"를 인식하지 못해 DateTimeParseException 발생
String usDateStr = "Oct 15, 2026";
DateTimeFormatter usFormatter = DateTimeFormatter.ofPattern("MMM dd, yyyy", Locale.US);
LocalDate parsedDate = LocalDate.parse(usDateStr, usFormatter);

// 2. 한국어 형식으로 출력 포맷 생성 (명시적 Locale 고정)
DateTimeFormatter korFormatter = DateTimeFormatter.ofPattern("yyyy년 MMM dd일 EEEE", Locale.KOREA);

// 3. 변환 결과: "2026년 10월 15일 목요일"
String korFormatted = parsedDate.format(korFormatter);
```
