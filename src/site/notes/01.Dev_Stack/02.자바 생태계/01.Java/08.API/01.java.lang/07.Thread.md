# 📘 Java 멀티스레딩 핵심: Thread와 Synchronized 심층 분석
> [!summary] 핵심 요약
> 
> - **개념**: 프로세스 내 독립적인 실행 흐름을 생성하는 `Thread`와 공유 자원의 동시 접근을 제어하는 `Synchronized` 메커니즘
>     
> - **왜?**: 효율적인 병렬 처리를 통해 애플리케이션의 응답성과 성능을 극대화하고, 다중 스레드 환경에서 발생할 수 있는 데이터 불일치(Data Inconsistency) 방지
>     
> - **언제?**: 대용량 데이터의 분산 처리, 비동기 네트워크 통신, 그리고 여러 스레드가 동일한 자원(변수, 객체)을 동시에 읽고 수정해야 하는 임계 영역(Critical Section) 보호 시
>     
> - **핵심**: 스레드 구현 시 확장성이 뛰어난 ==Runnable 인터페이스== 사용을 권장하며, 동기화 성능 저하를 막기 위해 락(Lock)의 범위를 최소화하는 ==블록(Block) 레벨 동기화== 적용 필수
> - 해당 내용은 고전적 방식이며 현재는 스레드 풀 방식의 ==Concurrent== 유틸을 사용함

>     
## Thread: 병렬 실행 메커니즘과 메모리 구조
- **멀티스레드 환경의 메모리 할당 특징**
    - **개별 영역(Stack)**: 각 스레드는 고유한 메서드 호출 스택과 로컬 변수를 가져 독립적인 실행 보장
    - **공유 영역(Heap)**: 생성된 객체 인스턴스는 모든 스레드가 공유하므로, 메모리를 효율적으로 사용하지만 ==동시 접근 시 데이터 무결성 훼손(Race Condition)== 위험 존재
```plaintext
[ 멀티스레드 JVM 메모리 구조 ]

      [공유 자원 영역]                      [스레드 개별 영역]
 ┌──────────────────────┐             ┌─────────────────────────┐
 │                      │ ◀─────────  │ Thread 1 (Stack Memory) │
 │     Heap Memory      │             │ - 로컬 변수, 실행 흐름  │
 │(객체, 인스턴스 변수) │             ├─────────────────────────┤
 │                      │ ◀─────────  │ Thread 2 (Stack Memory) │
 │  ※ 동기화(Lock) 대상 │             │ - 로컬 변수, 실행 흐름  │
 └──────────────────────┘             └─────────────────────────┘
 
- 순서 보장 x
```
- **스레드 구현 방식 2가지 비교**

|**구분**|**Thread 클래스 상속**|**Runnable 인터페이스 구현**|
|---|---|---|
|**확장성**|다른 클래스 상속 불가 (자바 단일 상속 제약)|==다른 클래스 상속 가능 (다중 구현 지원)==|
|**설계 측면**|스레드 제어 로직과 비즈니스 로직 결합됨|작업(Task)과 스레드(실행자)의 명확한 객체지향적 분리|
|**실무 적용**|단기적이고 단순한 테스트 외 지양|**실무 표준 권장 방식** (스레드 풀 연계 용이)|

```java
// 1. Thread 클래스 상속 (유연성 부족으로 지양)
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread 상속 실행");
    }
}

// 2. Runnable 인터페이스 구현 (확장성 우수로 권장)
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable 구현 실행");
    }
}

public class Main {
    public static void main(String[] args) {
        // 객체 생성 및 start()를 통한 독립적 스레드 호출
        new MyThread().start();
        new Thread(new MyRunnable()).start(); 
    }
}
```
- **스레드 생명주기 (Lifecycle) 상태**
    - `NEW`: 스레드 객체가 생성되었지만 아직 `start()`가 호출되지 않은 초기 상태
    - `RUNNABLE`: JVM 스케줄러에 의해 실행 대기 중이거나 실제 CPU를 점유하여 실행 중인 상태
    - `WAITING` / `BLOCKED`: 다른 스레드의 작업 완료를 기다리거나 락(Lock)을 얻기 위해 일시 정지된 상태
    - `TERMINATED`: `run()` 메서드 로직이 모두 완료되어 완전히 종료된 상태
```java
public class ThreadLifecycleDemo {
    public static void main(String[] args) throws InterruptedException {
        Object lock = new Object();

        // 1. NEW: 스레드 객체 생성 (아직 start 전)
        Thread t1 = new Thread(() -> {
		    // 3.(RUNABLE 지속) t1이 실행되어 해당 실행법대로 실행 시작
		    
            // sleep() 상태에서 interrupt()로 호출하는것 대비해 예외처리
            try {
                // 4. TIMED_WAITING: 잠시 대기 상태 유도
                Thread.sleep(500); 
                
                // 5. BLOCKED: 다른 스레드가 잡고 있다면, 락(Lock)을 기다림
                synchronized (lock) {
                    System.out.println("락을 획득했습니다!");
                }
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        System.out.println("상태 1: " + t1.getState()); // NEW

        // 2. RUNNABLE: 실행 대기 또는 실행 중
        t1.start();
        System.out.println("상태 2: " + t1.getState()); // RUNNABLE

        // t1이 sleep에 들어갈 시간을 조금 줌
        Thread.sleep(100);
        System.out.println("상태 3: " + t1.getState()); // TIMED_WAITING

        // 락을 메인 스레드가 먼저 가로채서 t1을 BLOCKED로 만듦
        synchronized (lock) {
            Thread.sleep(1000); 
            System.out.println("상태 4 (메인이 락 잡는 중): " + t1.getState()); // BLOCKED
        }

        // 6. TERMINATED: 작업 완료 후 종료
        t1.join(); // t1이 완전히 끝날 때까지 기다림
        System.out.println("상태 5: " + t1.getState()); // TERMINATED
    }
}
```
## Synchronized: 데이터 무결성과 동기화 제어
- **동기화가 필요한 치명적 버그 상황**
    - **경쟁 조건**: 2개 이상의 스레드가 하나의 공유 자원에 동시 접근하여 값을 변경할 때, 실행 타이밍에 따라 결과값이 왜곡되는 현상
    - **원자성 훼손**: `count++`와 같은 연산은 내부적으로 '읽기->증가->쓰기'의 3단계로 나뉘어 있어, 중간에 다른 스레드가 개입할 수 있음
    - **가시성 문제**: 한 스레드가 변경한 값이 메모리에 즉시 반영되지 않아, 다른 스레드가 옛날 값을 참조할 수 있음
- **모니터(Monitor)의 역할과 원리**
	- **모니터 사용:** 객체가 `synchronized(lock){}`의 인자로 전달되는 순간, 해당 객체의 모니터가 활성화되어 **'사용 중(Locked)'** 상태로 전환
	- **차단과 대기:** 다른 스레드가 같은 객체의 모니터를 사용하려 할 때, 이미 '사용 중'이라면 열쇠가 반납될 때까지 **BLOCKED(대기)** 상태로 멈춤
	- **배타적 접근:** 모니터는 오직 =='열쇠를 쥔 단 한 명의 스레드'==에게만 코드 실행 권한을 부여하여 질서를 유지함
- **효율적인 동기화 적용**
    - 무분별한 `synchronized` 남발은 다수의 스레드를 대기 상태로 만들어 애플리케이션의 심각한 성능 병목(Bottleneck) 유발
    - 메서드 전체에 거는 것 보다 꼭 필요한 코드 한 두 줄만 적용하여 락의 점유 시간을 최소화

| **적용 범위**  | **메서드 시그니처 및 문법**                   | **Lock 기준 및 동작 특징**                       | **성능** |
| ---------- | ----------------------------------- | ----------------------------------------- | ------ |
| **메서드 레벨** | `public synchronized void method()` | 메서드가 포함된 **객체 전체(`this`)에 락 발생**          | 낮음     |
| **블록 레벨**  | `synchronized(this) { ... }`        | ==임계 영역(Critical Section) 최소화==로 특정 코드만 락 | 비교적 우수 |

```java
// 메서드 레벨 동기화
public synchronized void useMoney(int use) {
    money -= use;
}

// 블록 레벨 동기화
public void useMoney(int use){
	synchronized(this){
		money -= use;
	}
}

// 락 객체 사용
public void useMoney(int use){
	synchronized(lock){
		money -= use;
	}
}

// ReentrantLock 방식 (향상된 방식-해당 노트 참조)
private final Lock lock = new ReentrantLock();
public void useMoney(int use) {
    lock.lock(); 
    try {
        money -= use;
    } finally {
        lock.unlock(); // 반드시 해제
    }
}
```
## Java SE 소속 및 API 특성
- **표준 라이브러리 및 문법 지원**
    - `java.lang.Thread`: `java.lang` 패키지에 포함되어 있어 별도의 `import` 선언 없이 즉시 사용 가능한 표준 클래스
    - `synchronized`: 메서드 선언부나 제어문 블록에 사용하는 자바 언어의 네이티브 예약어(Keyword)
