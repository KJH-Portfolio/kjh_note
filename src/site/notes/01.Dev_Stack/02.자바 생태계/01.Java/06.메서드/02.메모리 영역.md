# JVM 런타임 데이터 영역 (Runtime Data Area)
> [!summary] 핵심 요약
> 
> - **개념**: JVM이 자바 프로그램을 실행하기 위해 OS로부터 할당받는 메모리 공간
>     
> - **왜?**: 제한된 시스템 자원을 효율적으로 사용하고 메모리 누수 및 프로그램 강제 종료를 방지하기 위함
>     
> - **언제?**: 자바 애플리케이션의 실행 및 객체 생성, 메서드 호출 등 런타임 환경 전반
>     
> - **핵심**: 영역별 ==데이터 저장 대상==과 ==생명주기(Lifecycle)==의 정확한 매핑 및 구분
>     
## JVM 메모리 구조 분리
- **스레드(Thread) 공유 여부 기준**
    - 모든 스레드가 자원을 공유하는 영역과 개별 스레드마다 독립적으로 생성되는 영역으로 분류
    - 목적과 특성에 맞춘 최적화된 데이터 적재 및 수명 관리
```plaintext
[ JVM Runtime Data Area 구조도 ]

 ┌────────────────────────────────────────────────────────┐
 │ Thread Shared Area (모든 스레드 공유)                     │
 │  ├─ Method Area (Static): 클래스 메타데이터, 정적 변수      │
 │  └─ Heap Area           : 동적 할당 객체, 배열             │
 └────────────────────────────────────────────────────────┘
                               ▲ (참조)
 ┌─────────────────────────────┴──────────────────────────┐
 │ Thread Private Area (스레드별 독립 생성)                   │
 │  ├─ Stack               : 메서드 프레임, 지역/매개 변수      │
 │  ├─ PC Register         : 현재 실행 중인 명령어 주소        │
 │  └─ Native Method Stack : C/C++ 등 네이티브 코드 실행      │
 └────────────────────────────────────────────────────────┘
```
## 주요 3대 메모리 영역 비교
- **메모리 할당 체계**
    - 변수 선언 위치 및 객체 생성 방식에 따른 저장소 배정

| **영역**                 | **저장 대상**                    | **생명주기 및 관리 방식**                     | **주요 특징**                              |
| ---------------------- | ---------------------------- | ------------------------------------ | -------------------------------------- |
| **Stack**<br>`임시공간`    | 지역 변수, 매개변수, 메서드 프레임         | 메서드 호출 시 생성 / 종료 시 소멸 (LIFO)         | 높은 접근 속도, 스레드별 독립성 및 안전성 보장            |
| **Heap**<br>`객체`       | `new` 키워드로 생성된 인스턴스, 배열      | 동적 생성 / 불필요 시 ==GC(가비지 컬렉터)==에 의해 소멸 | 참조형 데이터의 실제 데이터가 위치하는 공유 공간            |
| **Method**<br>`Static` | `static` 변수/메서드, 클래스 정보, 상수풀 | 클래스 로드 시 적재 / JVM 종료 시 소멸            | 전역 공유 자원이므로 ==동시성(Thread-Safe) 제어 필수== |

## 메모리 관련 주요 에러 (Runtime Error)
- **비정상 종료 위험**
    - 부적절한 메모리 할당이나 참조 해제 실패 시 발생
    - 시스템 리소스 고갈로 인한 애플리케이션 즉시 강제 종료
> [!warning] 메모리 초과 에러 주요 유형
> 
> - **StackOverflowError**: 종료 조건이 없는 재귀 호출이나 무한 루프로 인해 Stack 메모리의 한계 용량을 초과할 때 발생합니다.
>     
> - **OutOfMemoryError (OOM)**: 더 이상 사용하지 않는 객체의 참조를 해제하지 않아(메모리 누수), Heap 영역의 공간이 완전히 고갈되었을 때 발생합니다.
>     
## 메모리 할당 위치 판별 및 에러 실습
- **코드 실행 단계별 메모리 적재 매핑**
    - 변수의 종류(인스턴스, 정적, 지역)에 따른 할당 영역 대조
```java
public class MemoryExample {
    // 1. Method(Static) 영역: 프로그램 시작 시 적재되어 전역 공유
    static int staticCount = 0;  

    // 2. Heap 영역: new 키워드로 객체 생성 시 할당
    int instanceCount = 0;       

    // 3. Stack 영역: 메서드 호출 시 프레임 생성 (매개변수 포함)
    public void method(int param) {  
        // 4. Stack 영역: 메서드 내부 지역 변수
        int localVar = param + 1;    
        System.out.println(localVar);
    }

    public static void main(String[] args) {
        // Heap 영역: MemoryExample 실제 인스턴스 생성
        // Stack 영역: Heap 영역의 주소를 가리키는 참조 변수 'ex' 할당
        MemoryExample ex = new MemoryExample();  
        
        // Stack 영역: method 실행을 위한 새로운 프레임 적재
        ex.method(5);                
    }
}
```
- **치명적 에러 발생 케이스**
    - 잘못된 코드 작성으로 인한 메모리 초과 유발 패턴
```java
public class ErrorExample {
    
    // 1. StackOverflowError 유발
    public static void recursive() {
        // 종료 조건 없는 무한 재귀 호출로 Stack 메모리 지속 누적
        recursive(); 
    }

    // 2. OutOfMemoryError 유발
    public static void createLeak() {
        List<int[]> list = new ArrayList<>();
        while (true) {
            // 무한 루프 내 객체 생성 및 리스트 저장
            // 참조가 계속 유지되므로 GC가 메모리를 수거하지 못함
            list.add(new int[1000000]); 
        }
    }
}
```
