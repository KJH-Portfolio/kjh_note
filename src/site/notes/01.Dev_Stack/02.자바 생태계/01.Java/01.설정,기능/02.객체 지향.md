# Java 객체 지향 프로그래밍 (OOP) 핵심 가이드
> [!summary] 핵심 요약
> 
> - 현실 세계의 개념을 객체 단위로 추상화하여 소프트웨어에 투영하는 프로그래밍 패러다임.
>     
> - 클래스, 인스턴스, 캡슐화, 상속, 다형성, 추상화의 6대 핵심 가치 제공.
>     
> - SOLID 설계 원칙 준수를 통한 유지보수성 및 코드 재사용성 극대화 목표.
> - $ 필드(데이터)와 함수(메서드)가 하나의 객체로 결합하여 사용하는 형태
>     
## 1. 객체 지향 설계 목적

| **항목**     | **상세 내용**                              |
| ---------- | -------------------------------------- |
| **현실 모델링** | 도메인 객체 및 행위의 코드화 및 실세계 비즈니스 로직 투영      |
| **모듈화**    | 객체 단위 책임 분리를 통한 독립적 구동 및 변경 영향 최소화     |
| **재사용성**   | 상속, 합성, 인터페이스 기반의 기존 코드 활용도 제고         |
| **유지보수성**  | 일관된 설계 구조 확보를 통한 코드 이해도 증진 및 수정 용이성 확보 |

## 2. 객체 지향 핵심 개념

| **개념**                  | **설명**                                   |
| ----------------------- | ---------------------------------------- |
| **클래스 (Class)**         | 객체 생성을 위한 상태(필드)와 행위(메서드)가 정의된 설계도       |
| **인스턴스 (Instance)**     | 클래스를 통해 메모리(Heap)에 할당된 실체화된 객체           |
| **캡슐화 (Encapsulation)** | 데이터 보호를 위한 필드 은닉(Private) 및 제한적 인터페이스 제공 |
| **상속 (Inheritance)**    | 부모 클래스의 특성을 자식이 물려받아 계층 구조 형성 및 코드 중복 제거 |
| **다형성 (Polymorphism)**  | 하나의 타입으로 여러 객체 구현체를 참조하여 유연한 구조 구현       |
| **추상화 (Abstraction)**   | 복잡한 내부 로직을 제외하고 핵심적인 기능과 속성만을 추출 및 정의    |

## 3. SOLID 설계 원칙

|**원칙**|**상세 정의**|**핵심 가치**|
|---|---|---|
|**SRP (단일 책임)**|하나의 클래스는 단 하나의 변경 이유와 책임만을 보유|높은 응집도|
|**OCP (개방-폐쇄)**|확장에는 열려 있어야 하고, 변경에는 닫혀 있어야 함|유연한 확장|
|**LSP (리스코프 치환)**|자식 클래스는 부모 클래스의 기능을 완전히 대체 가능해야 함|계층적 일관성|
|**ISP (인터페이스 분리)**|사용하지 않는 인터페이스에 의존하지 않도록 세분화된 분리 수행|불필요한 의존성 제거|
|**DIP (의존 역전)**|고수준 모듈은 저수준 모듈의 구현이 아닌 인터페이스에 의존|결합도 완화|

## 4. Java OOP의 기술적 특징
- **정적 타입 시스템**: 컴파일 시점의 타입 체크를 통한 런타임 안정성 보장.
- **클래스 기반 구조**: 모든 코드가 클래스 내부에 존재해야 하는 엄격한 객체 중심 설계.
- **풍부한 표준 API**: 컬렉션 프레임워크, 입출력(I/O) 등 검증된 객체 지향 라이브러리 제공.
- **디자인 패턴 최적화**: 인터페이스와 추상 클래스를 활용한 GoF 디자인 패턴 구현 최적화.
## 5. 설계 가이드라인 및 주의점
> [!tip] 고품질 설계를 위한 전략
> 
> - **정보 은닉의 철저화**: 모든 필드는 `private`으로 선언하고, 필요한 경우에만 최소한의 Getter/Setter 제공.
>     
> - **합성(Composition) 우선**: 상속(IS-A)의 경직성을 피하기 위해 가능한 한 합성을 통한 기능 확장 권장.
>     
> - **인터페이스 활용**: 구체적인 구현 클래스보다 인터페이스 타입 참조를 통한 결합도 감소 유도.
>     
> - **무분별한 Setter 금지**: 객체의 상태 변경은 의도가 명확한 메서드(Domain Logic)를 통해 수행.
>

## 6. 구현 예시
### 클래스 설계 및 캡슐화
```java
package com.example.model;

/**
 * 캡슐화가 적용된 사용자 도메인 모델
 */
public class User {
    private String name; 
    private int age;     

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // 데이터 보호를 위해 필드 직접 노출 대신 로직(Method) 제공
    public boolean isAdult() {
        return age >= 18;
    }

    public String getName() {
        return name;
    }
}
```
### 상속과 다형성의 결합
```java
package com.example.shape;

// 공통 인터페이스 역할을 하는 추상 클래스
public abstract class Shape {
    public abstract double calculateArea();
}

// 구체적인 구현체: 원
public class Circle extends Shape {
    private final double radius;

    public Circle(double radius) {
        this.radius = radius;
    }

    @Override
    public double calculateArea() {
        return Math.PI * radius * radius;
    }
}

// 다형성 활용 예시
Shape circle = new Circle(5.0);
System.out.println("Area: " + circle.calculateArea());
```
