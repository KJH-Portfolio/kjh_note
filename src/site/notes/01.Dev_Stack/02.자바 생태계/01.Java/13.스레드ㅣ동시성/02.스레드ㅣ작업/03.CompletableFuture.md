# CompletableFuture와 비동기 흐름 제어 (Java 비동기 기초 3단계)
> [!summary] 핵심 요약
> 
> - **개념**: 스레드의 대기(Blocking) 없이 비동기 작업 파이프라인을 연결, 병합, 복구하는 고급 동시성 제어 객체
>     
> - **왜?**: 기존 `Future`의 `get()` 호출 시 발생하는 병목 현상과 콜백 지옥(Callback Hell) 문제를 해결하기 위함
>     
> - **언제?**: 외부 API 다중 호출, DB 비동기 조회 후 데이터 가공 등 복잡한 비동기 I/O 제어 상황
>     
> - **핵심**: 스레드 차단 없는 ==메서드 체이닝(Chaining)== 적용 및 ==우아한 예외 복구(exceptionally)== 파이프라인 구축
>     
## 기존 Future의 한계와 논블로킹(Non-blocking) 패러다임
- **기존 `Future` 방식의 치명적 한계**
    - 비동기 작업 지시 후 결과를 얻기 위해 `future.get()` 호출 시, 메인 스레드가 ==결과가 나올 때까지 멈추는(Blocking)== 병목 현상 발생
    - 다중 작업 결과의 조합 및 완료 후 후속 콜백(Callback) 실행의 구조적 설계가 매우 까다로움
- **어떻게? (해결 방안)**
    - `CompletableFuture`를 도입하여 결과 대기 없이 후속 작업을 미리 예약(체이닝)하는 논블로킹 설계 적용
        - 해당 자료형은 메인 스레드에게 빈 상자를 넘겨주어 기다리지 않게 함
    - `CompletableFuture` 내부에서 생성되는 기본 스레드 풀은 애플리케이션 종료 시 함께 꺼지는 ==데몬(Daemon) 스레드==로 동작함
```plaintext
[ 블로킹(Blocking) vs 논블로킹(Non-blocking) 체이닝 비교 ]

▶ 기존 Future.get() 방식 (병목 발생)
  Main Thread : [작업 지시] ──▶ (결과 올 때까지 대기/차단) ──▶ [결과 수령 및 가공]
  Sub Thread  :             └─▶ [ API 호출 및 연산 중... ] ─┘

▶ CompletableFuture 체이닝 (논블로킹 흐름 제어)
  Main Thread : [작업 지시 + 이후 할 일(콜백) 예약] ──▶ (멈추지 않고 본인 업무 수행)
  Sub Thread  :             └─▶ [ API 연산 완료 ] ───▶ [예약된 가공 작업 자동 실행]
```
## 핵심 비동기 체이닝 파이프라인 (Chaining)
- **메서드 체이닝 동작 원리**
    - 선행 비동기 연산 완료 직후, 결과값을 전달받아 메인 스레드 차단 없이 ==후속 로직을 연속 실행==함
- **어떻게? (구현 방식)**
    - 파이프라인의 반환 타입과 목적에 맞춰 `Apply`, `Accept`, `Run` 계열의 메서드를 순차적으로 연결함

|**항목**|**동작 원리 및 역할**|**반환 타입**|
|---|---|---|
|**`thenApply(Function)`**|이전 결과를 받아 데이터를 **새로운 값으로 가공 및 변환** (Map 역할)|`CompletableFuture<U>`|
|**`thenAccept(Consumer)`**|이전 결과를 받아 **소비(출력, DB 저장 등)** 후 파이프라인 종료|`CompletableFuture<Void>`|
|**`thenRun(Runnable)`**|이전 결과와 무관하게 작업 완료 신호만 받아 **단순 후속 로직 실행**|`CompletableFuture<Void>`|

```java
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class AsyncChainingExample {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newFixedThreadPool(3);

        // 1. 작업 시작: Supplier 인터페이스를 활용하여 비동기 데이터 공급
        CompletableFuture<Void> pipeline = CompletableFuture.supplyAsync(() -> {
            return "원본 데이터";
        }, executor)

        // 2. 데이터 가공 (thenApply): 값을 받아서 변형 (입력 O, 반환 O)
        .thenApply(result -> {
            return "[" + result + "] 1차 가공 완료";
        })

        // 3. 데이터 소비 (thenAccept): 최종 처리 및 출력 (입력 O, 반환 X)
        .thenAccept(finalResult -> {
            System.out.println("최종 출력: " + finalResult);
        })

        // 4. 후속 로직 (thenRun): 모든 작업 완료 후 실행 (입력 X, 반환 X)
        .thenRun(() -> {
            System.out.println("모든 체이닝 작업 완료 후 정리 로직 실행");
        });

        pipeline.join(); // 데몬 스레드 종료 방지용 메인 스레드 대기
        executor.shutdown();
    }
}
```
## 우아한 예외 처리 (Graceful Exception Handling)
- **비동기 파이프라인 안전성 확보**
    - 체인 내부 어느 곳에서 에러가 발생하든 메인 스레드의 다운 없이 일괄적이고 안전하게 복구(Recover) 수행
- **어떻게? (구현 방식)**
    - `exceptionally`를 통한 폴백(Fallback) 데이터 지정 또는 `handle`을 통한 성공/실패 통합 분기 제어 적용

|**항목**|**동작 원리 및 역할**|**반환 타입 등 기타 속성**|
|---|---|---|
|**`exceptionally`**|예외 발생 시 캐치하여 ==기본값(Default Value)으로 대체== 후 정상 흐름 복구|에러 발생 시에만 동작함|
|**`handle`**|정상 결과(Result)와 예외(Exception)를 모두 인자로 받아 최종 반환 분기|성공/실패 무관하게 항상 실행됨|

```java
public class ExceptionHandleExample {
    public static void main(String[] args) {
        CompletableFuture.supplyAsync(() -> {
            // 50% 확률로 예외 발생 시뮬레이션
            if (Math.random() < 0.5) throw new RuntimeException("DB 접속 시간 초과");
            return "정상 뉴스 데이터";
        })
        
        // [방식 1] exceptionally: 에러 발생 시 플랜 B 데이터 공급
        .exceptionally(ex -> {
            System.out.println("에러 감지: " + ex.getMessage());
            return "로컬 백업 뉴스 데이터"; 
        })
        
        // [방식 2] handle: 에러 유무와 상관없이 통합 검증 및 분기 처리
        .handle((result, ex) -> {
            if (ex != null) {
                return "에러 감지"; // 예기치 않은 추가 예외 방어
            }
            return result + " (검증 완료)"; // 정상 데이터 통과
        })
        .thenAccept(System.out::println)
        .join();
    }
}
```
## 다중 비동기 작업 병합 (Composition & Combination)
- **비동기 흐름 통합 제어**
    - 서로 연관되거나 독립적인 여러 백그라운드 작업을 효율적으로 묶어서 하나의 결과물로 도출함

|**항목**|**동작 원리 및 역할**|**반환 타입 등 기타 속성**|
|---|---|---|
|**`thenCompose()`**|두 작업 간 **순차적 의존성** 존재 시 중첩된 퓨처를 ==평면화(Flatten)==하여 연결|A 완료 후 B 실행 (의존적)|
|**`thenCombine()`**|두 작업이 **완벽히 독립적**일 때 병렬 실행 후 두 결과 취합 및 병합|A, B 동시 실행 (독립적)|
|**`allOf()`**|N개의 비동기 작업이 ==모두 완료될 때까지 대기== (장바구니 일괄 결제 등)|전체 작업 성공 보장 후 다음 로직 진행|
|**`anyOf()`**|N개의 비동기 작업 중 **가장 빨리 끝난 1개의 결과만 채택** (서버 이중화 등)|가장 빠른 응답값 우선 반환|
```java
// A의 결과물을 B의 입력값으로 쓸 때
cfA.thenCompose(a -> cfB(a));

// A와 B를 동시에 돌리고 결과만 합칠 때
cfA.thenCombine(cfB, (a, b) -> a + b);

// N개가 전부 완료되어야 다음으로 넘어갈 때
CompletableFuture.allOf(cfA, cfB, cfC).join();

// N개 중 가장 먼저 끝나는 놈의 결과를 쓸 때
CompletableFuture.anyOf(cfA, cfB, cfC).thenAccept(res -> ...);
```
## 실행 제어 및 생명주기 관리 (Execution & Lifecycle)
- **동기화 대기 및 자원 누수 방지**
    - 비동기 작업의 무한 대기를 차단하고 메인 스레드와의 라이프사이클을 안전하게 일치시킴

|**항목**|**동작 원리 및 역할**|**반환 타입 등 기타 속성**|
|---|---|---|
|**`join()`**|모든 비동기 작업이 끝날 때까지 ==메인 스레드를 대기==시킴|`get()`과 달리 Unchecked Exception을 던짐|
|**`orTimeout()`**|지정된 시간 내 미완료 시 `TimeoutException` 발생 (Java 9+)|==무한 대기(Deadlock) 방지== 및 시스템 격리|
|**`shutdown()`**|할당된 외부 스레드 풀(ExecutorService) 팀 해체|누락 시 치명적인 ==메모리 누수== 유발|
```java
// 모든 작업이 끝날 때까지 여기서 멈춰서 기다림
cfA.join();

// 3초 안에 안 끝나면 에러 발생시킴 (exceptionally로 이동)
cfA.orTimeout(3, TimeUnit.SECONDS);

// 이제 새 손님 안 받음. 일하던 사람들만 마저 하고 퇴근해!
executor.shutdown();
```
