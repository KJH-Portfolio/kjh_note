# Spring Boot @Async와 비동기 파이프라인 실전 (Java 비동기 실무)
> [!summary] 핵심 요약
> 
> - **개념**: 스프링(Spring) 환경에서 스레드 풀, `Supplier`, `CompletableFuture`를 결합하여 구현하는 선언적 비동기 처리 아키텍처
>     
> - **왜?**: 복잡한 스레드 생성, 관리, 반납 로직(보일러플레이트)을 프레임워크에 위임하여 ==비즈니스 로직에만 집중==하기 위함
>     
> - **언제?**: 외부 API 통신, 대용량 데이터 처리, 메일 발송 등 메인 스레드(Tomcat)의 응답 지연을 차단해야 하는 실무 웹 애플리케이션
>     
> - **핵심**: 설정(`Config`), 공급(`Service`), 조립(`Controller`)의 3단계 계층 구조를 준수하고, ==내부 호출(Self-Invocation) 시 비동기가 무효화되는 프록시 한계== 숙지
>     
## 1. 설정 (Config): 스레드 풀 인프라 세팅
- **동작 원리 및 역할**
    - `@EnableAsync` 어노테이션을 통해 스프링 애플리케이션의 비동기 프로세싱(AOP 기반) 활성화
    - `ThreadPoolTaskExecutor`를 빈(Bean)으로 등록하여 개발자가 스레드 생성 개수, 대기열(Queue) 크기를 명시적으로 통제함
    - 애플리케이션 종료 시 워커 스레드의 안전한 자원 반납(Graceful Shutdown)까지 ==스프링 라이프사이클에 자동 연계==됨
- **어떻게? (구현 방식)**
    - 별도의 Configuration 클래스를 생성하여 목적에 맞는 스레드 풀을 커스텀 빈으로 등록하여 사용
```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.scheduling.annotation.EnableAsync;
import org.springframework.scheduling.concurrent.ThreadPoolTaskExecutor;
import java.util.concurrent.Executor;

@EnableAsync // 1. 스프링 비동기 기능 활성화 (설정 누락 시 동기로 작동함)
@Configuration
public class AsyncConfig {

    @Bean(name = "customTaskExecutor")
    public Executor threadPoolTaskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(3);   // 기본 유지 워커 스레드 수
        executor.setMaxPoolSize(10);   // 최대 확장 가능한 스레드 수
        executor.setQueueCapacity(50); // 스레드 부족 시 대기할 큐(Queue) 공간
        executor.setThreadNamePrefix("Async-Worker-"); // 디버깅용 스레드 이름 접두사
        executor.initialize();
        return executor;
    }
}
```
## 2. 서비스 (Service): 작업 공급자(Supplier) 위임
- **동작 원리 및 역할**
    - 무거운 비즈니스 로직을 담당하는 메서드에 `@Async`를 부착하여 거대한 `Supplier` 역할 부여
    - 메서드 호출 즉시 스프링 프록시(Proxy)가 개입하여 해당 로직을 ==워커 스레드에게 위임==하고 메인 스레드는 해방됨
- **어떻게? (구현 방식)**
    - 반환 타입은 반드시 `CompletableFuture<T>`로 감싸서 반환해야 상위 호출자(Controller)에서 파이프라인 체이닝 가능
```java
import org.springframework.scheduling.annotation.Async;
import org.springframework.stereotype.Service;
import java.util.concurrent.CompletableFuture;

@Service
public class UserService {

    // 1. Config에서 등록한 스레드 풀의 Bean 이름을 명시하여 작업 지시
    @Async("customTaskExecutor") 
    public CompletableFuture<String> fetchUserDataAsync(String userId) {
        System.out.println(Thread.currentThread().getName() + " - 외부 API 통신 중...");
        
        try { Thread.sleep(2000); } catch (InterruptedException e) {} // 무거운 I/O 작업 시뮬레이션
        
        String result = "User(" + userId + ", VIP)";
        
        // 2. 작업 결과를 CompletableFuture 객체로 포장하여 즉시 반환
        return CompletableFuture.completedFuture(result); 
    }
}
```
## 3. 컨트롤러 (Controller): 파이프라인 조립 및 응답
- **동작 원리 및 역할**
    - 서비스에서 던져준 `CompletableFuture` 객체를 받아 추가적인 데이터 가공 및 예외 처리를 선언적으로 파이프라이닝(Pipelining)함
    - 메인(Tomcat) 스레드는 작업만 예약해두고 즉시 다른 사용자의 HTTP 요청을 처리하러 이동하므로 ==서버의 동시 처리량(Throughput) 극대화==
- **어떻게? (구현 방식)**
    - `thenApply`, `exceptionally` 등의 논블로킹 체이닝 메서드를 조합하여 안전한 로직 흐름 구축
```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.RestController;
import java.util.concurrent.CompletableFuture;

@RestController
public class UserController {

    private final UserService userService;

	//보통 Lombok의 RequiredArgsConstructor어노테이션으로 이 생성자를 대체함
    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/api/users/{userId}/welcome")
    public CompletableFuture<String> sendWelcomeMessage(@PathVariable String userId) {
        
        // 1. 서비스 호출 (메인 스레드는 멈추지 않고 파이프라인만 예약함)
        return userService.fetchUserDataAsync(userId)
            
            // 2. 정상 데이터 가공 (데이터 변형 맵핑)
            .thenApply(userData -> {
                System.out.println(Thread.currentThread().getName() + " - 메시지 템플릿 생성 중...");
                return userData + "님, 환영합니다!";
            })
            
            // 3. 장애 발생 시 복구 (우아한 예외 처리)
            .exceptionally(ex -> {
                System.err.println("API 장애 포착: " + ex.getMessage());
                return "게스트님, 환영합니다. (백업 캐시 응답)";
            });
    }
}
```
## 4. 치명적 함정: 내부 호출 (Self-Invocation) 오류
- **동작 원리 및 한계점**
    - 스프링의 `@Async`는 AOP(관점 지향 프로그래밍) 기반의 프록시(Proxy) 패턴으로 동작함
    - 외부 클래스에서 빈(Bean)을 주입받아 호출할 때는 프록시가 정상 개입하지만, ==동일 클래스 내부에서 `this.메서드()` 형태로 호출 시 프록시를 완전히 우회==함
    - 결과적으로 새로운 스레드가 생성되지 않고 메인 스레드가 작업을 떠맡아 **동기(Synchronous)로 작동하는 장애 발생**
```plaintext
[ 스프링 @Async 프록시 내부 호출 실패 아키텍처 ]

[ 🟢 정상 호출 (외부에서 진입) ]
Controller ──▶ [ UserService Proxy ] ──(새 스레드 생성)──▶ Target Method (비동기 성공!)

[ 🔴 비정상 호출 (내부 Self-Invocation) ]
Controller ──▶ [ UserService Proxy ] ──▶ Target Method A (일반 메서드)
                                                │
                                                ▼ (프록시를 안 거치고 직접 호출!)
                                         Target Method B (@Async 붙어있음)
                                         (결과: 동기 처리되어 메인 스레드 정지됨)
```
```java
@Service
public class ChatService {

     // [🔴 비동기 작동 안 함] 내부 호출 (Self-Invocation)
     // 이 메서드를 외부에서 호출하면, 안에서 부르는 sendAsync는 동기(일반 메서드)로 작동함.
    public void processWithInternalCall() {
        System.out.println(Thread.currentThread().getName() + " - 일반 메서드 실행");
        
        // 같은 클래스 내의 메서드를 직접 호출 (this. 가 생략된 형태)
        // 프록시(대리인)를 거치지 않기 때문에 @Async가 무시됨
        sendAsyncNotification(); 
    }

     // [🟢 비동기 정상 작동] 외부 호출 전용
     // 이 메서드를 Controller 등 '다른 클래스'에서 호출해야만 비동기로 동작함.
    @Async("customTaskExecutor")
    public void sendAsyncNotification() {
        try { Thread.sleep(2000); } catch (InterruptedException e) {}
        System.out.println(Thread.currentThread().getName() + " - 비동기 알림 완료");
    }
}

// ----------------------------------------------------------------------

@RestController
@RequiredArgsConstructor
public class ChatController {
    private final ChatService chatService;

    @GetMapping("/test")
    public void test() {
        // 1. 실패 케이스: 서비스의 일반 메서드를 호출하면 내부에서 @Async를 직접 불러서 망함
        chatService.processWithInternalCall(); 

        // 2. 성공 케이스: 외부(Controller)에서 서비스의 @Async 메서드를 직접 호출함
        // 스프링 대리인이 개입하여 새 스레드를 할당해줌
        chatService.sendAsyncNotification(); 
    }
}
```

|**비교 항목**|**프록시 개입 정상 호출**|**Self-Invocation 오류**|
|---|---|---|
|**호출 주체**|외부 빈 (Controller 등)|자기 자신 (`this`) 내부 호출|
|**작업자**|워커 스레드로 정상 위임 (비동기)|메인 스레드가 독박 (동기 처리)|
|**해결책**|구조 유지|==`@Async` 메서드를 별도의 Service 클래스로 완전히 분리==|
