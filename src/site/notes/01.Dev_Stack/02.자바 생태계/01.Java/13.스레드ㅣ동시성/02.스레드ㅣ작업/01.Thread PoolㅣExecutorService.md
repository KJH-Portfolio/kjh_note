# 스레드 풀과 ExecutorService (Java 비동기 기초 1단계)
> [!summary] 핵심 요약
> 
> - **개념**: 미리 생성된 스레드 그룹(Pool)을 유지하고, 들어오는 작업(Task)을 큐(Queue)를 통해 할당하여 재사용하는 자원 관리 아키텍처
>     
> - **왜?**: 요청마다 새로운 스레드를 생성하면 트래픽 폭주 시 ==메모리 고갈(OOM) 및 서버 마비==가 발생하므로, 동시 실행 스레드 수의 물리적 제한이 필수적이기 때문
>     
> - **언제?**: 다중 클라이언트 요청을 처리하는 웹 서버, 대용량 데이터 비동기 병렬 처리 등 시스템 자원을 보호해야 하는 애플리케이션 전반
>     
> - **핵심**: 반환값 추적과 예외 처리가 유연한 ==`submit()` 메서드를 실무 표준으로 사용==하며, 애플리케이션 종료 시 반드시 `shutdown()`을 호출하여 자원 반납 필수
>     
## 고전적 동적 생성의 한계와 스레드 풀 아키텍처
- **고전적 스레드 생성 방식의 치명적 문제점**
    - **자원 오버헤드**: 스레드 생성 및 소멸은 OS 커널과 상호작용하는 무거운 작업으로, 매 요청마다 동적 생성 시 CPU 및 메모리 할당 비용 급증
    - **안정성 결여**: 생성 개수 제한이 없어 트래픽 스파이크 발생 시 메모리 부족(OOM)으로 인한 시스템 전체 마비 위험
    - **컨텍스트 스위칭 병목**: CPU 코어 수 대비 스레드가 과도하게 늘어나면, 스레드 간 전환(Context Switching)에 자원을 낭비하여 실질적 처리량 저하
- **스레드 풀(Thread Pool)의 동작 원리**
    - **스레드 사전 생성 및 대여**: 필요한 만큼의 워커 스레드(Worker Thread)를 미리 만들어 풀(Pool)에 보관하고, 할당된 작업 완료 시 다시 반납받아 재사용
    - **부하 조절 (Throttling)**: 처리 한계를 넘는 요청은 즉시 실행하지 않고 ==작업 대기열(Task Queue)에 적재==하여 서버의 가용 자원 임계치 보호
```plaintext
[ 스레드 풀 (ExecutorService) 동작 아키텍처 ]

  [클라이언트 요청]  [작업 대기열]      [미리 생성된 스레드 풀]
     Task 1 ──┐
     Task 2 ──┼─▶   [ Task Queue ] ───▶ [ Thread 1 ] (Active: Task 1 처리 중)
     Task 3 ──┘       (FIFO 대기)       [ Thread 2 ] (Idle  : 대기 중)
     Task 4 ──┘                         [ Thread 3 ] (Active: Task 2 처리 중)
                                        [ Thread N ]
                                             │
                                             ▼
                                       [결과 반환 및 스레드 풀로 복귀]
```
## 작업 제출 방식과 예외 처리 (execute vs submit)
- **작업 단위 인터페이스 (Runnable vs Callable)**
    - `Runnable`: 반환값이 없고(`void`), 체크 예외(Checked Exception)를 외부로 던질 수 없는 단순 실행 목적
    - `Callable`: 제네릭 타입의 ==결과값을 반환(return)==하며, 외부로 예외 전파가 가능함
- **작업 제출 메서드의 결정적 차이**
    - **`execute(Runnable)`**: 반환값이 없으며, 작업 중 예외 발생 시 ==해당 스레드를 강제 종료하고 폐기==함. 이후 스레드 풀 개수를 맞추기 위해 신규 스레드를 재생성해야 하므로 자원 낭비 유발
    - **`submit(Callable)`**: `Future` 객체를 반환하며, 예외 발생 시 스레드를 죽이지 않고 예외 정보를 `Future` 내부에 안전하게 보관함. 스레드는 파기되지 않고 즉시 재사용됨
- **어떻게? (실무 활용)**
    - 스레드 생존을 보장하고, 에러를 `Future.get()` 시점에 일관되게 핸들링하기 위해 실무에서는 무조건 `submit()` 사용을 권장함
```java
import java.util.concurrent.*;

public class ThreadPoolSubmitExample {
    public static void main(String[] args) throws InterruptedException {
        // 스레드 2개를 유지하는 고정 크기 풀 생성
        ExecutorService executor = Executors.newFixedThreadPool(2);

        // [방식 1] execute() - 예외 즉시 발생 및 스레드 사망
        // 반환값이 없는 Runnable 익명 구현 객체
        executor.execute(() -> {
            System.out.println("execute 스레드: " + Thread.currentThread().getName());
            throw new RuntimeException("execute() 치명적 에러 발생!"); 
        });

        Thread.sleep(500); // 출력 혼선 방지

        // [방식 2] submit() - 예외 숨김 및 스레드 생존 (재사용)
        // 반환값이 있는 Callable 익명 구현 객체
        Future<String> future = executor.submit(() -> {
            System.out.println("submit 스레드: " + Thread.currentThread().getName());
            if (에러 조건) throw new RuntimeException("submit() 치명적 에러 발생!");
            return "정상 처리 결과";
        });

        try {
            // .get()을 호출하는 순간(블로킹 발생), 내부에 보관되었던 예외가 밖으로 던져짐
            System.out.println("결과: " + future.get()); 
        } catch (ExecutionException e) {
            System.out.println("get() 호출 시 예외 포착: " + e.getCause().getMessage());
        } catch (Exception e) {
            e.printStackTrace();
        }

        executor.shutdown();
    }
}
```

| **생성 팩토리 메서드 (Executors)**      | **핵심 특징 및 동작 원리**               | **실무 권장 상황**                     |
| ------------------------------- | ------------------------------- | -------------------------------- |
| **`newFixedThreadPool(n)`**     | 설정한 n개만큼 고정된 스레드 유지 (부족 시 큐 대기) | **가장 일반적인 실무 표준** (예측 가능한 자원 점유) |
| **`newCachedThreadPool()`**     | 필요 시 제한 없이 스레드 생성, 60초 미사용 시 파기 | 짧고 가벼운 작업이 비주기적으로 대량 발생할 때       |
| **`newSingleThreadExecutor()`** | 단 1개의 스레드만으로 순차적 작업(FIFO) 처리    | 작업의 실행 순서가 무조건 보장되어야 하는 경우       |
| **`newScheduledThreadPool(n)`** | 특정 시간 지연 후 실행하거나, 주기적으로 반복 실행   | 스케줄러 기능 필요 시 (주기적 로그 백업 등)       |

## 안전한 자원 반납 (Graceful Shutdown)
- **명시적 종료의 필요성**
    - 메인 스레드가 종료되어도 스레드 풀 내부의 워커 스레드가 대기 상태로 살아있으면 JVM 프로세스가 종료되지 않음 (메모리 누수 원인)
- **어떻게? (3단계 우아한 종료 절차)**
    - **1단계 `shutdown()`**: 새로운 작업 접수를 즉각 거절하되, 큐에 남아있는 기존 대기 작업은 끝까지 실행 보장
    - **2단계 `awaitTermination()`**: 지정된 타임아웃 시간 동안 기존 작업들이 모두 완료되기를 블로킹 상태로 대기
    - **3단계 `shutdownNow()`**: 타임아웃 초과 시, 대기 중인 작업을 취소하고 ==실행 중인 스레드에 강제 인터럽트(Interrupt) 신호 전송==하여 무력화
```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.TimeUnit;

public class GracefulShutdownExample {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newFixedThreadPool(3);

        executor.submit(() -> System.out.println("비동기 작업 처리 중..."));

        // 1. 우아한 종료 시작 (신규 요청 차단, 기존 작업은 마무리)
        executor.shutdown();

        try {
            // 2. 최대 10초간 기존 작업 완료 대기 (모두 완료되면 true 반환)
            if (!executor.awaitTermination(10, TimeUnit.SECONDS)) {
                
                // 3. 10초 초과 시에도 안 끝난 작업이 있다면 강제 취소 명령
                System.err.println("작업 타임아웃. 강제 종료 시도");
                executor.shutdownNow(); 
            }
        } catch (InterruptedException e) {
            // 서브 스레드를 대기 중 메인 스레드가 인터럽트를 당한 비상 상황 시 즉시 강제 종료
            executor.shutdownNow();
            Thread.currentThread().interrupt(); // 인터럽트 상태 복원
        }
    }
}
```
