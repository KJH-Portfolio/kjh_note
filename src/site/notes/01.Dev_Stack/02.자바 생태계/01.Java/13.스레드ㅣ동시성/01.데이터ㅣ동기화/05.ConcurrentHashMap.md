# ConcurrentHashMap과 원자적 변수 (Java 동시성 제어 4단계: 현대적 완성)
> [!summary] 핵심 요약
> 
> - **개념**: 락(Lock)의 범위를 최소 단위로 쪼개거나 아예 사용하지 않고 멀티스레드 환경의 데이터 무결성을 보장하는 자료구조 및 변수 타입
>     
> - **왜?**: 맵 전체를 잠그는 레거시 방식(Hashtable 등)이 유발하는 ==극심한 성능 병목(Bottleneck) 현상을 해결==하여 처리량을 극대화하기 위함
>     
> - **언제?**: 다수의 스레드가 동일한 Map이나 List에 동시 다발적으로 접근하거나, 카운트 증가(`count++`)와 같은 단일 연산이 빈번하게 발생할 때
>     
> - **핵심**: 버킷 단위로 락을 분할하는 **ConcurrentHashMap**의 구조와 하드웨어 레벨에서 원자성을 보장하는 ==CAS(Compare-And-Swap) 알고리즘== 숙지
>     
## ConcurrentHashMap과 Lock Striping 기법
- **기존 Global Lock 방식의 한계**
    - `Hashtable`이나 `Collections.synchronizedMap`은 한 스레드가 데이터를 읽기만 해도 전체 맵의 접근이 차단되는 치명적 병목 지점(Bottleneck) 발생
- **버킷 단위 동기화 (Lock Striping / Node-level Lock)**
    - 내부 배열을 여러 개의 버킷(Node) 단위로 분할하여 개별적으로 락(Lock)을 적용함
    - 서로 다른 버킷에 접근하는 스레드들은 대기열 간섭 없이 완벽한 병렬 처리 수행
- **빈 버킷에 대한 무결성 보장 (Lock-Free 삽입)**
    - 새로운 노드를 삽입하려는 버킷이 비어있을 경우, 무거운 락 대신 CAS 알고리즘을 사용하여 노드를 추가해 성능 극대화
- **조회 성능 최적화 (volatile 활용)**
    - `get()` 메서드 호출 시 락을 전혀 사용하지 않음 (Non-blocking)
    - 내부 노드의 필드를 `volatile`로 선언하여 변경된 데이터의 메모리 가시성(Visibility) 즉시 확보
- **주의 및 제약 사항**
    - 멀티스레드 환경에서 '데이터 부재'와 '의도적인 Null'을 구분할 수 없는 모호성(Ambiguity) 방지를 위해 ==Key와 Value에 null 사용 엄격히 금지==

```plaintext
[ ConcurrentHashMap 버킷 단위 분할 락(Lock) 구조 ]

  들어오는 스레드          전체 맵 (ConcurrentHashMap)
┌──────────────┐     ┌───────────────────────────────────┐
│ Thread 1 (W) │ ──▶ │ [버킷 0] ◀ (Lock 적용 - 수정 중)  │

└──────────────┘     │ [버킷 4]                          │
┌──────────────┐     │ [버킷 5] ◀ (Lock 적용 안됨)       │
│ Thread 3 (W) │ ──▶ │ [버킷 6] ◀ (Lock 적용 - 수정 중)  │

   수정하므로 스레드 간 블로킹(대기) 없이 동시에 병렬 실행됨.
```

| --------------- | ---------------- | ------------------------------- | ----------------------------- |

| **동시성 병목**      | 매우 심각함           | 심각함                             | **거의 없음 (병렬 처리 극대화)**         |
| **읽기(Read) 작업** | 락 획득 필수          | 락 획득 필수                         | **락 없이 즉시 읽기 (Non-blocking)** |
| **Null 키/값 허용** | 불가능              | 가능 (원본이 HashMap일 경우)            | **불가능 (예외 발생)**               |
## 원자적 변수(Atomic Variables)와 CAS 알고리즘
- **일반 연산의 원자성(Atomicity) 훼손 문제**
    - `count++` 코드는 1줄처럼 보이지만, CPU 레벨에서는 [읽기 -> 증가 -> 저장]의 3단계로 분리 실행됨
    - 연산 도중 타 스레드가 개입하면 데이터 유실(Race Condition) 발생
- **CAS (Compare-And-Swap) 메커니즘 적용**
    - 하드웨어의 지원을 받아 '기존 메모리 값'과 '내가 기대하는 값'이 일치할 때만 새 값으로 교체(Swap)하는 ==락-프리(Lock-Free) 동기화 기법==
- **Spin Lock (루프 재시도)**
    - 타 스레드가 먼저 값을 수정해 CAS 검증에 실패하면, 최신 값을 다시 읽어와 성공할 때까지 무한 루프(`while`)로 재시도함
```java
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounterExample {
    // 무거운 락(synchronized) 없이 동시성을 완벽히 보장하는 원자적 정수 변수 객체
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        // 내부적으로 무한 루프(Spin Lock)와 CAS 알고리즘을 사용하여 원자성 보장
        count.incrementAndGet(); // 일반 변수의 ++count 연산과 동일한 논리적 동작
    }

    public int getCount() {
        return count.get();
    }
}
```
```plaintext
[ CAS (Compare-And-Swap) 알고리즘 논리 흐름 ]

 [스레드 A의 메모리 갱신 시도]
        │
        ▼
 1. 현재 메모리의 값(V) 읽기 ──▶ (예: V = 10)
        │
        ▼
 2. 기대하는 값(E) 설정 ───▶ (예: E = 10) // 작업을 시작할 때 봤던 값 
    변경할 새 값(N) 준비 ──▶ (예: N = 11)
        │
        ▼
 3. [CAS 하드웨어 명령어 실행]
    메모리 값(V) == 기대 값(E) 인가?
        │
      ┌─┴──────────────────────────────┐
  (일치함 - 충돌 없음)                (불일치 - 다른 스레드가 먼저 수정함)
      │                                │
      ▼                                ▼
 4. 값을 N(11)으로 갱신         4. 갱신 실패! 
      │                         최신 값으로 E를 업데이트 후
      ▼                         [1번 단계로 돌아가 재시도(Spin Lock)]
 [작업 성공 및 종료]
```

|**Atomic 클래스**|**다루는 데이터 타입**|**주요 실무 활용 예시**|
|---|---|---|
|**`AtomicInteger`**|`int` 정수형|실시간 방문자 수 카운팅, 고유 트랜잭션 ID 순차 발급|
|**`AtomicLong`**|`long` 정수형|대용량 고유 식별자(PK) 생성, 누적 파일 크기 합산|
|**`AtomicBoolean`**|`boolean` 논리형|시스템 초기화 여부 플래그, 스레드 풀 단일 진입점 제어|
|**`AtomicReference`**|임의의 객체(`Object`)|동시성 큐(Queue)의 노드 연결, 복합 상태 객체의 원자적 교체|
