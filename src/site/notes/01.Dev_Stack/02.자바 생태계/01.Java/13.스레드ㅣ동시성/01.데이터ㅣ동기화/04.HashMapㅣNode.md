# Java HashMap 내부 구조: Entry에서 Node로의 진화
> [!summary] 핵심 요약
> 
> - **개념**: Java 8부터 `HashMap`의 데이터 저장 기본 단위가 단순 배열 형태의 `Entry`에서 연결 리스트 및 트리를 지원하는 `Node` 구조로 개편된 아키텍처
>     
> - **왜?**: 동일한 해시값을 가지는 데이터가 많아질 경우 발생하는 해시 충돌 시, 기존 연결 리스트 탐색(O(N))으로 인한 치명적인 성능 저하 방지
>     
> - **언제?**: 대규모 데이터를 다루는 `Map` 자료구조에서 데이터 조회가 빈번하게 일어나 성능 최적화가 요구될 때
>     
> - **핵심**: 버킷 내 데이터 개수에 따라 ==Linked List와 Red-Black Tree를 동적으로 변환==하여 검색 시간 복잡도를 O(log N)으로 방어
>     
## Node 객체의 내부 코드 구조
- **기본 단위의 변경**
    - Java 7 이하: `HashMap.Entry<K,V>[]` 객체 배열 구조 구현
    - Java 8 이상: `Map.Entry<K,V>`를 구현한 단일 연결 리스트 형태의 `HashMap.Node<K,V>`로 변경
    - 트리 변환 시: `Node`를 상속받은 `TreeNode<K,V>`를 사용하여 이진 탐색 트리 구조 형성
    - **어떻게?**: 기존 Entry 구조에서 Node 객체로 마이그레이션하여 해시 충돌 시 연결 리스트나 트리 형태로 데이터를 확장 보관
```java
// 1. 기본 연결 리스트 구조를 위한 Node 클래스 (Java 8+)
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;     // 불변: 한 번 계산된 해시값 캐싱
    final K key;        // 불변: 키 객체
    V value;            // 가변: 값 객체 (중복 키 유입 시 변경됨)
    Node<K,V> next;     // 가변: 해시 충돌 시 다음 노드를 가리키는 포인터

    Node(int hash, K key, V value, Node<K,V> next) {
        this.hash = hash;
        this.key = key;
        this.value = value;
        this.next = next;
    }
}

// 2. Red-Black Tree 구조를 위한 TreeNode 클래스
static final class TreeNode<K,V> extends LinkedHashMap.Entry<K,V> {
    TreeNode<K,V> parent;  // 부모 노드
    TreeNode<K,V> left;    // 왼쪽 자식 트리
    TreeNode<K,V> right;   // 오른쪽 자식 트리
    TreeNode<K,V> prev;    // 삭제 최적화를 위한 이전 노드 링크
    boolean red;           // Red-Black Tree의 색상 속성 (균형 유지용)
}
```
```plaintext
[ HashMap 버킷(Bucket) 메모리 구조 ]

  Array (해시 테이블)
  ┌─────┐
  │ [0] │──▶ null (데이터 없음)
  ├─────┤
  │ [1] │──▶ [Node: Hash/K/V] ──▶ [Node: Hash/K/V] ──▶ null 
  ├─────┤       (Linked List 구조: 데이터 7개 이하)
  │ [2] │──▶ null
  ├─────┤
  │ [3] │──▶ [TreeNode] (Red-Black Tree 구조: 데이터 8개 이상)
  └─────┘       ├── [TreeNode: Left]
                └── [TreeNode: Right]
```
## 중복 데이터 유입 시 처리 및 정렬 메커니즘
- **데이터 식별 기준**
    - HashMap에서 데이터의 고유성은 `hashCode()`의 리턴값과 `equals()` 메서드의 결과로 판별
    - 새로운 데이터 삽입(`put`) 시, 기존 노드들과 비교하여 ==Key 중복==과 ==Hash 충돌==을 명확히 구분하여 처리
    - **어떻게?**: 삽입 요청 시 해시값을 계산하고 버킷 인덱스를 할당한 후, 내부 충돌 여부에 따라 값을 덮어쓰거나 노드를 연장
```plaintext
[ 데이터 삽입(Put) 흐름도 및 중복 처리 로직 ]

         [put(key, value) 호출]
                 │
                 ▼
       [1. key.hashCode() 계산] ──▶ 버킷 인덱스 할당
                 │
                 ▼
   ┌──▶ [2. 해당 버킷에 노드가 존재하는가?] ──(No)──▶ [새로운 Node 생성 후 저장 (종료)]
   │             │
   │           (Yes)
   │             ▼
   │    [3. Key 객체 완전 일치 검증]
   │    (hash가 같고, equals()가 true인가?)
   │             │
   │      (Yes)  │  (No: Hash 충돌)
   │       ▼     ▼       ▼
   │ [Value 덮어씀]   [4. 다음 노드로 이동 (next)]
   │    (종료)           │
   │                     ▼
   │                  (일치 검증 반복...)
   │				  (끝까지 일치하는 Key가 없으면 리스트 마지막에 새 Node 추가)
   │
   └─▶ ※ [임계치 도달 여부 확인]: 리스트 길이가 8 이상이면 Tree로 구조 변경
```
- **상황별 처리 방식**
    - 키(Key) 완전 중복: 기존 노드의 `value`만 새로운 값으로 덮어씀 (데이터 개수 증가 없음)
    - Hash Collision (해시값 동일, 키 다름): 기존 노드의 `next` 포인터에 새 노드 연결 (리스트 꼬리 추가)
    - 정렬 방식 (Tree 형태): Red-Black Tree 변환 상태에서 충돌 발생 시, 키 객체의 `Comparable` 인터페이스 기준으로 크기를 비교해 좌/우 정렬. 기준이 동일하면 고유 메모리 주소(`System.identityHashCode()`)를 2차 기준으로 삼아 강제 대소 분리
    - **어떻게?**: 데이터 편향으로 인한 선형 탐색 방지를 위해 트리의 이진 탐색 로직을 활용하여 저장 위치 결정
```plaintext
[ 단일 버킷(인덱스 1번) 내부의 상태 변화 과정 ]

[ 1단계: 초기 삽입 (랜덤 값 42, 7, 89 ... 순으로 7개 삽입 시) ]
버킷[1] ──▶ [Key:42] ──▶ [Key:7] ──▶ [Key:89] ──▶ [Key:15] ... (선형 탐색 O(N))
※ 데이터를 찾으려면 앞에서부터 하나씩 다 열어봐야 함.

========================================================================

[ 2단계 & 3단계: 임계치(8) 돌파 후 100개까지 삽입 완료 시 ]
※ 순서대로 일렬로 쌓이는 것이 아니라, 크기(compareTo)를 기준으로 좌/우로 찢어짐

                            [ 루트 노드: 대략 중간값(예: 45) ]
                             ↙                           ↘
               (45보다 작은 그룹)                          (45보다 큰 그룹)
                  [ 23 ]                                     [ 76 ]
                 ↙      ↘                                   ↙      ↘
            [ 12 ]      [ 35 ]                          [ 60 ]      [ 89 ]
           ↙      ↘    ↙      ↘                       ↙      ↘    ↙      ↘
        [ 7 ]   [18]  [28]   [42]                  [52]   [68]  [82]   [95]
        ↙  ↘     ...   ...    ...                   ...    ...   ...    ...
     [1]   [10]                                                      [98] [100]

※ 핵심 요약: 
1. 100개의 데이터가 몰려도, 트리의 최대 높이(Depth)는 6~7층 수준으로 억제됨.
2. 탐색 시 선형 이동이 아닌 루트 기준 우측 편향 이동으로 O(log N) 속도 보장.
```
## Red-Black Tree 균형 유지 메커니즘
- **Double Red 발생 및 해결 원칙**
    - 새 노드는 삽입 시 항상 **Red** 속성을 가짐
    - 이로 인해 부모 노드와 자식 노드가 연속으로 Red가 되는 규칙 위반(Double Red) 현상 발생
    - **어떻게?**: 부모의 형제 노드인 =='삼촌 노드'의 색상 상태==에 따라 Rotation(회전) 또는 Recoloring(색상 재조정)을 수행하여 균형 복구
- **Rotation (회전)**
    - 조건: 삼촌 노드가 **Black이거나 없을 때** 수행
    - 동작: 트리를 물리적으로 회전시켜 높이를 비틀고 색상 재조정 병행
    - **어떻게?**: 3 -> 2 -> 1 순서로 삽입되어 좌측 편향된 구조를 2를 중심으로 우회전(Right Rotation)시켜 루트로 끌어올리고 색상을 안정화
```plaintext
[ Rotation 흐름도: 3 -> 2 -> 1 순차 삽입 시 (삼촌 노드 없음) ]

  [1. Double Red 발생]     [2. Right Rotation 수행]    [3. Recoloring 및 완성]
      [3] (Black)                [2] (Red)                [2] (Black/Root)
      /                          /   \                    /   \
    [2] (Red)        ──▶       [1](R) [3](B)     ──▶    [1](R) [3](R)
    /
  [1] (Red) *문제발생!
```
- **Recoloring (색상 재조정)**
    - 조건: 삼촌 노드가 이미 **Red 상태일 때** 수행
    - 동작: 물리적 회전 없이 부모와 삼촌을 Black으로 승격시키고, 할아버지 노드를 Red로 변경
    - **어떻게?**: 트리의 구조는 그대로 둔 채 색상만 반전시켜 Double Red 위반 사항을 트리의 상단으로 밀어 올려 해결
```plaintext
[ Recoloring 흐름도: 삼촌 노드가 Red인 상태에서 새 노드 삽입 시 ]

  [1. Double Red 발생]                  [2. Recoloring 완료]
        [3] (Black)                          [3] (Red) *루트일 경우 다시 Black 변환
        /    \                               /    \
     [2](R)  [4](R) *삼촌(Red)  ──▶       [2](B)  [4](B) *부모/삼촌 Black 승격
     /                                   /
  [1](R) *문제발생!                    [1](R)
```

| **구분**         | **발생 조건**                | **해결 방식**                                      |
| -------------- | ------------------------ | ---------------------------------------------- |
| **Rotation**   | 부모의 형제(삼촌)가 **Black/없음** | 트리를 물리적으로 회전시켜 트리의 높이를 낮추고 노드 색상을 재배치함         |
| **Recoloring** | 부모의 형제(삼촌)가 **Red**      | 트리의 구조 유지, 부모/삼촌을 Black으로 변경하고 할아버지를 Red로 승격시킴 |

## 자료구조 동적 변환 임계치 (Threshold)
- **변환 원리**
    - 해시 충돌 빈도에 따라 탐색 속도(List)와 메모리 효율성(Tree) 사이의 트레이드오프 동적 조절
    - Tree 구조는 검색은 빠르나 노드 생성 및 균형 유지에 ==메모리와 연산 리소스가 List보다 약 2배 소모==됨
    - **어떻게?**: `HashMap` 내부의 고정된 임계값(상수)을 기준으로 구조 변환 트리거 동작

|**속성명**|**임계값**|**동작 원리 및 역할**|
|---|---|---|
|`TREEIFY_THRESHOLD`|8|단일 버킷 노드가 8개 이상 도달 시 Linked List를 **Red-Black Tree로 변환** (O(N) -> O(log N))|
|`UNTREEIFY_THRESHOLD`|6|노드 삭제로 6개 이하로 감소 시 Tree를 **Linked List로 복구** (메모리 최적화)|
|`MIN_TREE_CAPACITY`|64|트리 변환을 위한 최소 해시 테이블 크기 전제 조건 (64 미만 시 배열 리사이징 우선 수행)|
```plaintext
[ 트리 변환(Treeify) vs 리사이징(Resize) 결정 흐름도 ]

             [ 시작: 특정 버킷에 신규 데이터 삽입 ]
                               │
                               ▼
               [ 해당 버킷의 노드 개수가 8개인가? ] ──(아니오)──▶ [ 리스트 구조 유지 / 종료 ]
                               │
                             (예)
                               │
                               ▼
             [ 테이블 전체 크기(Capacity)가 64 미만인가? ]
                               │
              ┌───────(예)─────┴─────(아니오)──────┐
              │                                    │
              ▼                                    ▼
       [ resize() 실행 ]                 [ treeifyBin() 실행 ]
    (테이블 크기 2배 확장)               (Red-Black Tree로 변환)
              │                                    │
              ▼                                    ▼
    [ 노드 재분산(Rehash) ]             [ 검색 효율 최적화 완료 ]
              │                                    │
              └────────────────┬───────────────────┘
                               │
                               ▼
                            [ 종료 ]
```
> [!tip] 8과 6 사이의 버퍼(Buffer)
> 
> 임계값을 8(생성)과 6(해제)으로 차이를 두는 이유는, 데이터 개수가 7개와 8개를 반복적으로 오르내릴 때 자료구조가 지속적으로 변환되며 발생하는 과부하(Thrashing) 현상을 방지하기 위한 안전 장치입니다.
