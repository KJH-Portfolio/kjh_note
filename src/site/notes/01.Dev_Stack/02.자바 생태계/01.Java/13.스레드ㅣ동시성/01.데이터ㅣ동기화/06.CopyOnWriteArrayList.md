# CopyOnWriteArrayList (Java 동시성 제어 5단계: 조회 최적화 리스트)
> [!summary] 핵심 요약
> 
> - **개념**: 쓰기(Write) 연산 시 기존 배열을 통째로 복사하여 업데이트하고, 읽기(Read) 연산은 락(Lock) 없이 원본을 조회하는 스레드 안전(Thread-safe) 리스트
>     
> - **왜?**: `Vector`나 `Collections.synchronizedList`가 유발하는 전역 락(Global Lock) 방식의 조회 병목 현상을 완벽히 제거하기 위함
>     
> - **언제?**: 데이터의 삽입/삭제/수정은 매우 드물지만, 순회 및 조회 작업이 압도적으로 많이 발생하는 환경 (예: 이벤트 리스너, 시스템 설정 캐시)
>     
> - **핵심**: 배열 전체 복사로 인한 메모리 오버헤드가 극심하므로 반드시 ==조회 위주의 제한적 환경에서만 사용==해야 함
>     
## CopyOnWriteArrayList의 핵심 동작 원리
- **배열 복사(Copy-On-Write) 메커니즘**
    - 데이터 추가, 수정, 삭제 등 쓰기 작업 발생 시, 내부 배열의 새로운 복사본을 통째로 생성함
    - 새로운 복사본에서 변경 작업을 완벽히 수행한 뒤, 최종적으로 원본 참조를 교체(Swap)하여 갱신함
- **동시성 충돌 완전 배제 (Lock-Free Read)**
    - 한 스레드가 데이터를 수정하기 위해 복사 및 작업을 진행하는 도중에도, 다른 스레드들은 ==변경 이전의 '원본 배열'을 락 없이 안전하게 조회== 가능함
- **순회(Iteration) 안전성 보장**
    - `iterator()` 호출 시점의 배열 상태(스냅샷)를 기준으로 순회 동작을 수행함
    - 순회 도중 타 스레드에 의해 리스트 구조가 변경(요소 추가/삭제)되더라도 `ConcurrentModificationException` 예외가 절대 발생하지 않음
- **어떻게? (실무 적용 가이드라인)**
    - 매 쓰기 작업마다 내부 배열을 통째로 복사하는 무거운 메모리 할당 및 복사 오버헤드가 발생함
    - 따라서 쓰기가 잦은 로직에 적용 시 GC(Garbage Collector) 부하와 심각한 성능 저하를 초래하므로 용도를 엄격히 제한해야 함
```plaintext
[ Copy-On-Write 메커니즘 동작 구조 ]

  현재 상태 (원본 배열 참조)
 ┌─────────────────────────┐     [조회 스레드들]
 │ [ A ] [ B ] [ C ]       │ ◀── Thread 1 (락 없이 즉시 읽기)
 └─────────────────────────┘ ◀── Thread 2 (락 없이 즉시 읽기)
             │
             ▼ (쓰기 작업 발생: 스레드 3이 "D" 추가)
[쓰기 스레드 (Thread 3)]
 1. 원본 복사 + 크기 확보 (Lock 적용)
 ┌───────────────────────────────┐
 │ [ A ] [ B ] [ C ] [   ]       │
 └───────────────────────────────┘
 2. 새 데이터(D) 삽입
 ┌───────────────────────────────┐
 │ [ A ] [ B ] [ C ] [ D ]       │
 └───────────────────────────────┘
 3. 참조 교체 (Swap)
 원본 배열 참조 ──▶ 새 배열로 변경 (이후 새로운 조회는 여기를 읽음)
```
```java
import java.util.Iterator;
import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;

public class CopyOnWriteExample {
    public static void main(String[] args) {
        // 스레드 안전성을 보장하는 CopyOnWriteArrayList 생성
        List<String> list = new CopyOnWriteArrayList<>();
        list.add("Item 1");
        list.add("Item 2");

        // 순회(Iteration) 시작: 이 시점의 배열 상태("Item 1", "Item 2")의 스냅샷을 확보함
        Iterator<String> iterator = list.iterator();

        // 순회 도중 다른 쓰기 작업(수정)이 발생
        // 내부적으로는 배열이 새롭게 복사되어 "Item 3"이 추가됨 (원본은 유지)
        list.add("Item 3"); 

        // ConcurrentModificationException 예외가 발생하지 않으며, 안전하게 순회 완료
        while (iterator.hasNext()) {
            System.out.println(iterator.next());
            // 출력 결과: Item 1, Item 2
            // (스냅샷 생성 이후에 추가된 Item 3은 현재 순회에 반영되지 않음)
        }
    }
}
```

|**비교 항목**|**ArrayList**|**Vector / SynchronizedList**|**CopyOnWriteArrayList**|
|---|---|---|---|
|**동시성 보장**|지원 안함|모든 메서드에 무거운 락(Lock) 적용|**쓰기: 배열 복사, 읽기: 락 없음**|
|**읽기 성능**|빠름|느림 (경합 및 락 병목 발생)|**매우 빠름 (락 프리 구조)**|
|**쓰기 성능**|빠름|느림 (락 대기)|**매우 느림 (메모리 복사 비용)**|
|**순회 안정성**|예외 발생 가능성 매우 높음|외부 동기화(synchronized 블록) 필수|**완벽 보장 (스냅샷 기반)**|
