# ReentrantLock과 고도화된 동기화 (Java 동시성 제어 3단계)
> [!summary] 핵심 요약
> 
> - **개념**: 명시적으로 락(Lock)을 획득하고 해제할 수 있는 `java.util.concurrent.locks` 패키지의 고급 동기화 메커니즘
>     
> - **왜?**: 암묵적인 `synchronized` 키워드가 유발하는 무한 대기(Deadlock) 위험을 방지하고, 읽기/쓰기 작업을 분리하여 성능 병목 현상을 해결하기 위함
>     
> - **언제?**: 락 획득 대기 시간(Timeout)이 필요하거나, 대기열의 순서 보장(Fairness)이 필수적이거나, 읽기 비중이 압도적으로 높은 임계 영역(Critical Section)을 보호할 때
>     
> - **핵심**: 명시적 제어이므로, 예외 발생 시 무한 락 상태에 빠지지 않도록 ==락 해제(`unlock()`) 로직은 반드시 `finally` 블록 내부에 작성==해야 함
>     
## 기존 synchronized의 한계 (통잠금의 부작용)
- **무한 대기 (Deadlock 취약성)**
    - 락을 획득할 때까지 스레드가 Blocked 상태에서 무한정 대기함
    - 중간에 대기를 취소(Interrupt)하거나 ==타임아웃(Timeout)을 설정하는 것이 원천적으로 불가능==함
- **공정성(Fairness) 결여**
    - 대기열에 있는 스레드 중 먼저 온 스레드가 먼저 락을 얻는다는 보장이 없음
    - 특정 스레드가 락을 영원히 획득하지 못하는 기아(Starvation) 현상 발생 가능성 존재
- **비효율적인 독점**
    - 단순 조회(읽기) 작업과 수정(쓰기) 작업을 구분하지 않음
    - 데이터 변경이 없는 읽기 작업들끼리도 서로를 차단하여 동시 처리 성능 심각하게 저하됨
## ReentrantLock의 핵심 기능과 제어 방식
- **세밀한 락 제어를 위한 3대 기능**
    - **타임아웃 (Timeout)**: 지정된 시간만큼만 락 획득을 대기하고, ==실패 시 프로그램이 멈추지 않고 우회 로직 수행==
    - **공정성 (Fairness)**: 인스턴스 생성 시 옵션을 주어 큐(Queue)에 대기한 시간이 가장 긴 스레드에게 우선적으로 락 제공
    - **인터럽트 (Interrupt)**: 락 대기 중인 스레드에 작업 취소(`InterruptedException`) 명령 하달 지원
- **어떻게? (구현 방식)**
    - `lock()` 메서드로 락을 명시적으로 잠그고, `unlock()`으로 명시적 해제
```java
public class AdvancedLockExample {
    // 공정성(Fairness)을 true로 설정하여 순서 보장 (먼저 대기한 스레드 우대)
    private final Lock lock = new ReentrantLock(true); 

    public void performTaskWithTimeout() {
        try {
            // 1. 최대 2초 동안만 락 획득 대기 (Deadlock 원천 차단)
            if (lock.tryLock(2, TimeUnit.SECONDS)) {
                try {
                    System.out.println("임계 영역(Critical Section) 안전하게 실행");
                } finally {
                    // 2. 예외 발생 여부와 무관하게 반드시 락 해제 (필수 규칙)
                    lock.unlock(); 
                }
            } else {
                // 3. 락 획득 실패 시 무한 대기하지 않고 우회 로직 실행
                System.out.println("락 획득 실패: 서버 혼잡, 대체 로직 수행");
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt(); // 스레드 인터럽트 상태 복구
        }
    }
}
```

| **구분**         | **synchronized (기존 방식)** | **ReentrantLock (고도화 방식)**        |
| -------------- | ------------------------ | --------------------------------- |
| **락 획득/해제**    | 암묵적 (블록 진입/종료 시 자동)      | 명시적 (`lock()`, `unlock()` 명시적 호출) |
| **타임아웃 지원**    | 불가 (무한 대기)               | 가능 (`tryLock(time)` 활용)           |
| **순서 보장(공정성)** | 불공정 (Unfair 고정)          | 생성자 옵션(`true`)으로 공정/불공정 선택 가능     |
| **락의 분리**      | 불가 (객체 단위 단일 통잠금)        | `ReadWriteLock`을 통해 읽기/쓰기 락 분리 가능 |

## ReadWriteLock: 조회 성능 극대화 아키텍처
- **읽기와 쓰기의 분리를 통한 동시성 최적화**
    - **Read Lock (공유 락)**: 데이터 수정이 없는 단순 조회 시 획득. ==다른 스레드의 Read Lock과 동시 접근을 허용==하여 트래픽 병목 완전 해소
    - **Write Lock (배타 락)**: 데이터 수정 시 획득. 해당 락이 활성화되는 순간, 다른 스레드의 모든 읽기/쓰기 접근 완벽 차단
```java
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class CacheManager {
    // 1. ReadWriteLock 인스턴스 생성 및 읽기/쓰기 락 분리
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();
    
    private String cachedData = "기본 데이터";

    // 2. 읽기 작업: 여러 스레드가 동시에 Read Lock 획득 가능 (병렬 처리)
    public String readData() {
        readLock.lock();
        try {
            System.out.println(Thread.currentThread().getName() + " - 데이터 조회 중: " + cachedData);
            return cachedData;
        } finally {
            readLock.unlock(); // 예외 방지를 위해 finally에서 해제
        }
    }

    // 3. 쓰기 작업: 단 하나의 스레드만 Write Lock 획득 가능 (다른 읽기/쓰기 모두 대기)
    public void writeData(String newData) {
        writeLock.lock();
        try {
            System.out.println(Thread.currentThread().getName() + " - 데이터 업데이트 중: " + newData);
            this.cachedData = newData;
        } finally {
            writeLock.unlock();
        }
    }
}
```
```plaintext
[ ReadWriteLock 동시 접근 제어 매트릭스 ]

  현재 활성화된 락    추가 Read 요청      추가 Write 요청
┌──────────────────┬───────────────────┬───────────────────┐
│  Read Lock (읽기)│   허용 (병렬 처리)│   대기 (차단)     │ ◀ 다수의 조회 요청 동시 처리
│  Write Lock(쓰기)│   대기 (차단)     │   대기 (차단)     │ ◀ 완벽한 데이터 무결성 보장
└──────────────────┴───────────────────┴───────────────────┘

※ 핵심 요약:
1. 읽기끼리는 서로 막지 않는다. (조회 트래픽 병목 해소)
2. 쓰기가 개입하는 순간(수정), 모든 접근은 멈추고 줄을 서야 한다.
```
