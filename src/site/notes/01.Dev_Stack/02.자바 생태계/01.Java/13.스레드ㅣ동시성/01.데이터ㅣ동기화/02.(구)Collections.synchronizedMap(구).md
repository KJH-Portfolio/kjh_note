# Collections.synchronizedMap의 구조와 한계 (Java 동시성 제어 2단계)
> [!summary] 핵심 요약
> 
> - **개념**: 일반 `HashMap` 등의 컬렉션 객체를 감싸서(Wrap) 스레드 안전성을 강제로 부여하는 헬퍼(Helper) 메서드
>     
> - **왜?**: `Hashtable`이라는 구시대적 클래스에 얽매이지 않고, 현대적인 `Map` 인터페이스의 유연성을 활용하면서 동기화를 적용하기 위함
>     
> - **언제?**: 레거시 코드를 현대적 컬렉션으로 마이그레이션하거나, 복잡한 동시성 제어가 필요 없는 단순한 환경에서 스레드 안전한 맵이 필요할 때
>     
> - **핵심**: 객체 포장 방식만 세련되어졌을 뿐, 여전히 ==맵 전체를 통째로 잠그는(Lock) 성능 병목 현상은 그대로 존재==함
>     
## 래퍼(Wrapper) 패턴 기반의 구조적 발전
- **현대적 인터페이스 활용**
    - `Hashtable` 단일 클래스만 사용해야 했던 과거와 달리, `HashMap`, `TreeMap`, `LinkedHashMap` 등 다양한 구현체를 동기화 맵으로 변환 가능함
    - Java 컬렉션 프레임워크의 다형성(Polymorphism)을 온전히 활용할 수 있는 과도기적 발전 형태
- **어떻게? (내부 동작 원리)**
    - 메서드 자체가 아닌, 내부적으로 원본 맵(Backing Map)과 동기화를 위한 `mutex`(상호배제용 자물쇠) 객체를 함께 품는 데코레이터(Decorator) 패턴을 사용함
    - 사용자의 요청이 들어오면 `mutex`에 락을 건 뒤, 원본 맵의 메서드를 대신 호출하여 결과를 반환함
```plaintext
[ Collections.synchronizedMap 내부 통잠금 구조 ]

  멀티 스레드 접근 요청
  ├─ 스레드 A (get) ──▶ [ Mutex (자물쇠) 획득 ] ──┐
  ├─ 스레드 B (put) ──▶ [ 대기 - BLOCKED ]        │
  └─ 스레드 C (del) ──▶ [ 대기 - BLOCKED ]        │
			                                      ▼
  ┌───────────────── SynchronizedMap ─────────────────┐
  │                                                   │
  │   synchronized (mutex) {                          │
  │       // 한 번에 단 한 명만 원본 Map에 접근 가능  │
  │       return backingMap.get(key);                 │
  │   }                                               │
  │                                                   │
  │  ┌───────────────── Backing Map ────────────────┐ │
  │  │           (일반 HashMap 인스턴스)            │ │
  │  └──────────────────────────────────────────────┘ │
  └───────────────────────────────────────────────────┘
```
## 통잠금의 유산과 순회(Iteration) 시의 치명적 결함
- **변하지 않은 성능 한계**
    - 락을 거는 주체가 메서드(`this`)에서 내부 `mutex` 객체로 바뀌었을 뿐, ==하나의 스레드가 맵을 점유하면 타 스레드의 접근이 전면 차단되는 직렬화 현상은 동일==함
    - 읽기(`get`) 스레드와 쓰기(`put`) 스레드가 서로를 가로막아 멀티 코어 환경의 이점을 살리지 못함
- **어떻게? (수동 동기화의 강제)**
    - `put()`, `get()` 등 단일 연산은 맵 내부에서 락을 걸어주어 안전함
    - 단, `Iterator`나 향상된 `for`문을 통한 ==맵 전체 순회 시에는 스레드 안전성이 자체 보장되지 않으므로 수동 락 처리가 필수==임
    - 순회 도중 다른 스레드가 데이터를 삽입/삭제하면 즉시 `ConcurrentModificationException` 예외가 발생하며 애플리케이션이 강제 종료될 위험 존재
```java
import java.util.*;

public class SynchronizedMapExample {
    public static void main(String[] args) {
        // 1. 일반 HashMap을 생성함과 동시에 동기화 맵으로 래핑(Wrapping)
        Map<String, String> syncMap = Collections.synchronizedMap(new HashMap<>());

        // 2. 단일 데이터 조작은 자동으로 동기화 보장 (안전함)
        syncMap.put("UserA", "Seoul");
        syncMap.put("UserB", "Busan");

        // 3. [주의] 전체 순회(Iteration) 시에는 반드시 수동 Lock 명시 필요
        // 동기화 블록으로 감싸지 않으면 순회 도중 타 스레드 개입 시 예외 발생
        synchronized (syncMap) {
            Iterator<String> iterator = syncMap.keySet().iterator();
            while (iterator.hasNext()) {
                System.out.println("Key: " + iterator.next());
            }
        }
    }
}
```
## 1단계(Hashtable)와 2단계(SynchronizedMap) 스펙 비교

| **비교 항목**   | **Hashtable (1단계)**         | **Collections.synchronizedMap (2단계)** |
| ----------- | --------------------------- | ------------------------------------- |
| **기반 구조**   | 낡은 `Dictionary` 상속 (확장성 부족) | 최신 `Map` 인터페이스 구현 (유연한 다형성)           |
| **동기화 방식**  | 객체 자신(`this`)에 메서드 레벨 락 적용  | 내부 `mutex` 객체에 블록 레벨 락 적용             |
| **순회 안전성**  | 자체 제공되는 `Enumeration`은 안전함  | `Iterator` 사용 시 ==반드시 수동 동기화 필요==     |
| **Null 허용** | Key, Value 모두 절대 불가         | 래핑된 원본 맵(예: `HashMap`)의 허용 정책을 따름     |
