# 객체지향 재사용 전략: 상속(Inheritance) vs 합성(Composition)
> [!summary] 핵심 요약
> 
> - **개념**: 객체지향 설계에서 기존 클래스의 코드를 재사용하고 결합 방식을 결정하는 두 가지 핵심 기법 (is-a vs has-a)
>     
> - **왜?**: 중복 코드를 제거하고, 유지보수성과 유연성이 뛰어난 아키텍처를 구축하기 위함
>     
> - **언제?**: 새로운 객체를 설계할 때 기존 기능을 확장하거나 다양한 기능을 조합해야 하는 상황
>     
> - **핵심**: 현대 시스템 설계에서는 강한 결합에 따른 부작용을 막기 위해 ==상속보다 합성을 우선(Prefer composition over inheritance)==하는 것을 원칙으로 함
>     
## 상속 (Inheritance)
- **개념**
    - 하위 클래스가 상위 클래스의 특성(필드, 메서드)을 그대로 이어받아 재사용하거나 확장하는 기법
    - 부모와 자식 간의 관계가 ==컴파일 타임(정적)==에 결정되는 물리적 결합
    - **어떻게?**: Java 등 객체지향 언어에서 `extends` 키워드를 사용하여 부모 클래스 명시

|**특성**|**설명**|**비고**|
|---|---|---|
|관계 유형|is-a (A는 B의 일종이다)|예: 고래(자식)는 포유류(부모)다|
|재사용 방식|화이트박스 재사용|내부 구현이 하위 클래스에 그대로 노출됨|
|결합도|매우 높음|부모 클래스 변경 시 자식 클래스 전체에 파급 효과 발생|

```plaintext
[ 상속의 위계 구조 ]
└─ Mammal (부모: 호흡, 수유 기능)
    ├─ Dog (자식: 짖기 기능 추가)
    └─ Whale (자식: 수영 기능 추가)
```
- **치명적 한계점 (사용을 경계해야 하는 이유)**
    - **캡슐화 파괴**: 부모 클래스의 내부 구현이 자식에게 노출되어 ==강한 결합== 발생
    - **유연성 저하**: 컴파일 단계에 관계가 굳어져, 런타임에 부모 클래스의 행동을 변경하는 것이 불가능
    - **클래스 폭발**: 다중 상속이 제한된 환경에서 새로운 조합이 필요할 때마다 불필요한 파생 클래스가 기하급수적으로 증가
## 합성 (Composition)
- **개념**
    - 기존 클래스를 새로운 클래스의 필드(인스턴스 변수)로 포함시켜 그 기능을 위임받아 사용하는 기법
    - 객체 간의 관계가 ==런타임(동적)==에 결정되는 유연한 논리적 위임
    - **어떻게?**: 필요한 기능의 인터페이스나 객체를 생성자 또는 세터를 통해 주입받아 내부에서 메서드 호출

|**특성**|**설명**|**비고**|
|---|---|---|
|관계 유형|has-a (A는 B를 포함한다)|예: 자동차(본체)는 엔진(부품)을 갖는다|
|재사용 방식|블랙박스 재사용|객체의 내부 구현은 숨기고 퍼블릭 인터페이스만 활용|
|결합도|낮음|포함된 객체의 내부가 변경되어도 외부 클래스에 미치는 영향 최소화|

```java
public class Car {
    // 1. 기능을 가진 객체를 필드로 포함 (합성)
    private Engine engine; 

	//생성자를 통해 주입
    public Car(Engine engine) {
        this.engine = engine;
    }

    public void drive() {
        // 2. 직접 구현하지 않고 주입받은 엔진 객체에 기능을 위임 (Delegation)
        engine.start();
        System.out.println("자동차가 움직입니다.");
    }
}
```
## 실전 설계 가이드: 무엇을 써야 하는가?
- **상속을 허용하는 전제 조건 (매우 제한적 적용)**
    - 두 객체의 관계가 ==진정한 의미의 is-a== 관계일 때 (단순 코드 재사용이 목적이라면 상속 금지)
    - 부모 클래스가 하위 확장을 명시적으로 고려하여 설계 및 문서화되었을 때
    - 하위 클래스가 부모의 행동 규약을 완벽히 준수(리스코프 치환 원칙, LSP)할 때
- **합성을 도입해야 하는 경우 (기본 원칙)**
    - 객체의 동작 방식을 ==실행 시점에 동적으로 변경==해야 할 때
    - 여러 독립적인 클래스의 기능을 자유롭게 조합하여 새로운 복합 기능을 생성할 때
    - 클래스 간 의존성을 낮춰 단위 테스트(Mocking) 작성과 유지보수를 용이하게 할 때

|**비교 항목**|**상속 (Inheritance)**|**합성 (Composition)**|
|---|---|---|
|관계 결정 시점|컴파일 타임 (정적)|런타임 (동적)|
|캡슐화|낮음 (부모 내부 노출)|높음 (인터페이스만 노출)|
|유연성|낮음 (구조 변경 어려움)|높음 (객체 교체 용이)|
|실무 권장도|신중히 제한적으로 사용|**최우선적으로 고려**|
