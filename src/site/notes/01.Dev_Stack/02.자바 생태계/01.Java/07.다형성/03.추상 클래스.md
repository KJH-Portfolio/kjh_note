# 객체지향 추상 클래스 (Abstract Class)
> [!summary] 핵심 요약
> 
> - **개념**: 공통 기능(일반 메서드) 및 상태(필드) 제공과 세부 구현(추상 메서드)의 강제를 동시에 수행하는 미완성 설계도
>     
> - **왜?**: 여러 클래스 간의 데이터와 로직 중복을 막고, 자식 클래스가 반드시 가져야 할 필수 동작의 누락을 방지하기 위함
>     
> - **언제?**: 연관성이 높은 객체들이 데이터(상태)를 공유하거나, 부모가 주도하는 전체 실행 흐름 안에서 일부 기능만 다르게 구현해야 할 때
>     
> - **핵심**: `abstract` 키워드를 사용하며, 단독으로 ==인스턴스화할 수 없으므로 반드시 상속을 통해 완성==해야 함
>     
## 추상 클래스 기본 구조 및 특징
- **설계 원리 및 목적**
    - 여러 구체 클래스(Concrete Class)의 공통된 상태(인스턴스 변수)와 행위(메서드)를 상위로 추출하여 통합 (일관성 유지)
    - 자식 클래스에게 특정 메서드의 구현 책임을 위임하여 다형성(Polymorphism) 보장

|**구성 요소**|**세부 특징**|**비고**|
|---|---|---|
|**선언 형식**|`abstract class 클래스명 { }`|`abstract` 제어자 명시 필수|
|**추상 메서드**|`abstract` 제어자 명시 및 ==구현부(Body) 없음== (`{ }` 부재)|자식에서 오버라이딩 필수|
|**일반 멤버**|일반 필드, 생성자, 일반 메서드를 자유롭게 포함 가능|상속을 통한 기본 상태/기능 제공|
|**인스턴스화**|`new` 키워드를 통한 ==직접 객체 생성 절대 불가==|부모 타입 참조 변수로는 활용 가능 (업캐스팅)|

```java
abstract class Animal {
    // 추상 클래스는 인터페이스와 달리 인스턴스 변수(상태)를 가질 수 있음
    String name;
    
    // 자식의 super() 호출에 의해 초기화되는 생성자
    Animal(String name) { 
        this.name = name; 
    }

    public void eat() {
        System.out.println(name + "이(가) 밥을 먹습니다.");
    }
    
    // 자식 클래스에서 반드시 구현해야 할 핵심 기능
    public abstract void sound();
}

class Dog extends Animal {
    Dog(String name) { 
        super(name); // 부모 상태 초기화 필수
    }
    
    @Override
    public void sound() { 
        System.out.println("멍멍!"); 
    }
}
```
> [!warning] 설계 및 상속 제약 사항
> 
> - **구현의 연쇄**: 자식 클래스가 부모의 추상 메서드를 하나라도 오버라이딩하지 않으면, 해당 자식 클래스 역시 `abstract`로 선언되어야 함.
>     
> - **단일 상속 원칙**: 일반 클래스와 동일하게 다중 상속 불가능 (오직 하나의 부모만 상속 가능).
>     
## 모던 자바 설계: 추상 클래스 vs 인터페이스 비교
- **인터페이스 `default` 메서드 도입 이후의 설계 기준**
    - Java 8 이후 인터페이스도 구현부(`default`)를 가질 수 있게 되면서, 단순히 '기능의 기본 구현을 제공한다'는 목적만으로는 추상 클래스의 사용이 정당화되지 않음
    - 현대 객체지향에서 **추상 클래스를 반드시 선택해야 하는 핵심 상황**은 다음 두 가지로 좁혀짐
- **1. 상태(필드/변수)를 직접 공유하고 관리해야 할 때**
    - 자식들이 공통으로 가져야 할 ==데이터(인스턴스 변수)와 그 관리 로직이 중복==될 때 사용
    - 인터페이스는 상수(`static final`)만 가질 수 있으므로, 객체마다 달라지는 개별 상태를 저장할 수 없음
    - **적용 예시**: 모든 유저가 `id`, `email`, `password` 상태를 가질 때, `BaseUser` 추상 클래스에 필드를 선언하고 `getPassword()` 등의 공통 상태 제어 메서드를 사전 구현하여 자식(Admin, Customer)에게 상속
- **2. 템플릿 메서드 패턴을 통한 실행 흐름 제어가 필요할 때**
    - 전체적인 ==실행 시나리오(흐름)는 부모가 고정==해두고, 중간의 특정 단계만 자식에게 구현을 강제하고 싶을 때 사용
    - 필드(상태)와 실제 구현 코드가 섞여 있어야 하며, `final` 키워드나 `protected` 접근 제어자를 통해 외부의 접근과 오버라이딩을 엄격하게 통제해야 하므로 추상 클래스가 절대적으로 유리함

|**비교 항목**|**추상 클래스 (Abstract Class)**|**모던 인터페이스 (Java 8+ Interface)**|
|---|---|---|
|**핵심 목적**|공통 **상태(필드)** 및 실행 흐름의 공유와 구조화|순수 기능(역할) 계약 강제 및 다중 구현 보장|
|**상태 (필드)**|==일반 인스턴스 변수 선언 가능== (상태 보관)|상수(`public static final`)만 선언 가능|
|**메서드 제어**|`protected`, `final` 등 다양한 접근/제어자 활용 가능|`public` 위주 (`private` 헬퍼 메서드 지원)|
|**상속/구현**|단일 상속만 허용 (`extends`)|다중 구현 허용 (`implements`)|

## 추상 클래스 필요 예시
#### 1. 자식들이 각각의 상태를 가져야 할 때
```java
abstract class BaseUser {
    protected int hp; // ✅ 각 객체마다 별도로 가질 수 있는 '상태' 필드

    public BaseUser(int hp) {
        this.hp = hp;
    }

    public void takeDamage(int damage) {
        this.hp -= damage; // ✅ 공통 상태를 직접 제어하는 로직을 미리 구현 가능!
        System.out.println("현재 체력: " + hp);
    }
}
```
#### 2. 템플릿 메서드 패턴으로 공통 로직 사이에 특정 로직만 추상화할때
```java
abstract class RamenRecipe {
    // 템플릿 메서드: 전체적인 흐름을 정의 (final로 오버라이딩 방지)
    public final void cook() {
        boilWater();
        addMainIngredients(); // 자식이 구현할 부분
        serve();
    }

    private void boilWater() {
        System.out.println("물을 100도까지 끓입니다.");
    }

    private void serve() {
        System.out.println("그릇에 예쁘게 담습니다.");
    }

    // 자식마다 달라져야 하는 핵심 단계만 추상 메서드로!
    protected abstract void addMainIngredients();
}

class ShinRamen extends RamenRecipe {
    @Override
    protected void addMainIngredients() {
        System.out.println("신라면 스프와 면을 넣습니다.");
    }
}
```
