# 다형성 방법 선택 전략
> [!summary] 핵심 요약
> 
> - **개념**: 실행 시점(Runtime)에 알고리즘(전략)을 캡슐화하여 상호 교체 가능하게 만드는 행동 디자인 패턴
>     
> - **왜?**: 기존 코드(Context)의 수정 없이 새로운 기능(Strategy)을 유연하게 추가하여 ==OCP(개방-폐쇄 원칙)==를 준수하기 위함
>     
> - **언제?**: 결제 방식, 할인 정책, 정렬 알고리즘 등 동일한 목적을 수행하지만 세부 구현 로직이 다양한 상황에 직면했을 때
>     
> - **핵심**: 변하는 행위를 인터페이스로 묶고, 이를 ==합성(Composition)==을 통해 동적으로 주입받아 위임(Delegation)
>     
## 객체지향 4대 개념의 실무 통합 구조
- **구성 요소별 핵심 역할**
    - `Context` (문맥): 알고리즘을 실행하는 주체 객체, 내부적으로 전략 인터페이스를 필드로 보유
    - `Strategy` (전략): 다양한 알고리즘이 공통으로 지켜야 할 외부 동작 규약
    - `ConcreteStrategy` (구체적 전략): 인터페이스를 실제로 구현한 개별 알고리즘 로직

| **객체지향 개념**                 | **전략 패턴 내에서의 역할 및 시너지**                                      |
| --------------------------- | ------------------------------------------------------------ |
| **인터페이스 (Interface)**       | 변하는 행위의 표준 규약(메서드)을 강제하여 다형성(Polymorphism)의 기반 제공 (`can-do`) |
| **합성 (Composition)**        | Context가 Strategy를 필드로 소유하여 런타임에 동적 교체 및 결합도 감소 (`has-a`)    |
| **추상 클래스 (Abstract Class)** | 구체적 전략들 간에 중복되는 상태나 로직이 있을 경우 이를 묶어서 재사용 (선택적 뼈대 구축)         |
| **상속 (Inheritance)**        | 구체적 전략 객체가 추상 클래스의 공통 기능을 물려받아 다형성(업캐스팅) 실현 (`is-a`)         |

```plaintext
[ 전략 패턴 구조 및 관계도 ]
(Context: 주문 시스템)

                            <<합성 (has-a)>>
┌──────────────────────┐                        <<인터페이스 (can-do)>>
│ OrderContext         │ ──────────────────────▶┌──────────────────────┐
│ - strategy: Payment  │                        │ PaymentStrategy      │
│ + executePayment()   │                        │ + pay(amount)        │
└──────────────────────┘                        └──────────────────────┘
           ▲                                               ▲
           │                                               │ <<구현 (implements)>>
           │ (클라이언트가 런타임 주입)                    │
           │                              (선택적: 공통 로직 Abstract Class)
           │                                    ┌──────────────────────┐
           │                                    │ AbstractPayment      │
           │                                    │ # authenticate()     │
           │                                    └──────────────────────┘
           │                                               ▲
           │                                               │ <<상속 (extends)>>
           │                                               │
           │                              ┌────────────────┴────────────────┐
           │                              │                                 │
           └───────────────────▶┌─────────┴──────────┐            ┌─────────┴──────────┐
                                │ CreditCardStrategy │            │ KakaoPayStrategy   │
                                │ + pay(amount)      │            │ + pay(amount)      │
                                └────────────────────┘            └────────────────────┘
```
## 전략 패턴 구현 예시 (Java)
> [!example] 4가지 개념이 모두 적용된 결제 시스템 설계
> 
> 인터페이스로 규약을 세우고, 추상 클래스로 공통 로직을 처리하며, 합성을 통해 유연성을 확보하는 실무적 접근 방식
```java
// 1. [인터페이스] 변하는 행동(결제)의 표준 규약 정의
public interface PaymentStrategy {
    void pay(int amount);
}

// 2. [추상 클래스] 공통 로직(결제 모듈 로깅, 보안 인증 등)의 재사용 (선택적)
public abstract class AbstractPayment implements PaymentStrategy {
    protected void authenticate() {
        System.out.println("공통 로직: 결제 모듈 보안 인증 완료");
    }
}

// 3. [상속] 추상 클래스를 상속받아 구체적인 전략 구현 (다형성 확보)
public class CreditCardStrategy extends AbstractPayment {
    @Override
    public void pay(int amount) {
        authenticate(); // 부모의 공통 로직 재사용
        System.out.println(amount + "원 신용카드 결제 처리 진행");
    }
}

public class KakaoPayStrategy extends AbstractPayment {
    @Override
    public void pay(int amount) {
        authenticate();
        System.out.println(amount + "원 카카오페이 결제 처리 진행");
    }
}

// 4. [합성] Context는 구체 클래스가 아닌 인터페이스를 필드로 소유 (느슨한 결합)
public class OrderContext {
    private PaymentStrategy strategy; // 동적 교체를 위한 합성(has-a)

    // 런타임에 외부에서 전략을 주입(변경) 가능 (의존성 주입)
    public void setPaymentStrategy(PaymentStrategy strategy) {
        this.strategy = strategy;
    }

    public void executePayment(int amount) {
        // 구체적인 결제 방식은 알 필요 없이 인터페이스의 메서드만 호출 (동적 바인딩)
        strategy.pay(amount);
    }
}
```
## 도입 장단점 및 고려 사항
- **주요 이점**
    - ==조건문(if-else) 폭발 방지==: 새로운 로직 추가 시 기존 클래스 내부의 분기문 수정 없이 새로운 클래스만 추가
    - **런타임 유연성**: 실행 도중 객체의 동작 방식을 자유롭게 변경 가능 (예: 카드 결제 실패 시 포인트 결제로 즉시 스위칭)
    - **테스트 독립성**: 각 전략을 독립적인 클래스로 분리하여 개별적인 단위 테스트(Unit Test) 환경 구축 용이
- **주의 및 제약 사항**
    - **관리 복잡도 증가**: 전략마다 새로운 클래스가 생성되므로 전체 프로젝트의 클래스 파일 개수 증가
    - **클라이언트의 인지 부담**: 이 패턴을 사용하는 호출자(Client)가 ==각 전략 클래스의 차이점을 사전에 명확히 인지==하고 적절한 객체를 생성하여 주입해야 함
