# 자바 모던 인터페이스 (Java 8+ 심화)
> [!summary] 핵심 요약
> 
> - **개념**: Java 8과 9에 걸쳐 도입된 인터페이스의 구현 기능(`default`, `private` 등)과 함수형 프로그래밍 지원 문법
>     
> - **왜?**: 기존 구현체에 영향을 주지 않고 인터페이스를 확장(하위 호환성)하며, 람다식을 통해 불필요한 보일러플레이트 코드를 최소화
>     
> - **언제?**: 안정적으로 서비스 중인 레거시 인터페이스에 새로운 공통 기능을 추가하거나, 스트림(Stream) API 등에서 동작 파라미터화를 구현할 때
>     
> - **핵심**: `default` 메서드의 목적성 파악 및 ==단 하나의 추상 메서드==만 허용하는 함수형 인터페이스(`@FunctionalInterface`) 규칙 숙달
>     
## 모던 인터페이스 확장 메서드 (Java 8+)
- **인터페이스 진화 배경**
    - **기존 순수 추상 메서드 한계**: 인터페이스에 새 기능 추가 시 모든 구현체(클래스)를 수정해야 하는 구조적 문제 발생
    - **하위 호환성 보장**: 기존 구현체를 깨뜨리지 않고 새로운 공통 로직 추가 지원
    - **캡슐화 강화**: 인터페이스 내부에서만 사용하는 중복 코드를 은닉하여 응집도 향상

|**구분**|**도입 버전**|**주요 특징 및 역할**|**제약 및 주의사항**|
|---|---|---|---|
|**`default`**|Java 8|예외적으로 구현부(`{}`)를 가지는 인스턴스 메서드|다중 구현 시 충돌 발생하면 ==하위 클래스에서 오버라이딩 필수==|
|**`static`**|Java 8|객체 생성 없이 호출 가능한 유틸리티 메서드|`인터페이스명.메서드명()`으로 직접 호출 (오버라이딩 불가)|
|**`private`**|Java 9|`default`/`static` 메서드 간 중복 코드 분리 및 캡슐화|인터페이스 내부에서만 호출 가능 (자식 클래스로 상속 불가)|

```plaintext
[ 자바 버전별 인터페이스 스펙 변화 ]

Java 7 이하          Java 8                 Java 9 이후
┌──────────────┐     ┌─────────────────┐      ┌─────────────────┐
│ - 상수       │ ──▶ │ - 상수          │ ───▶ │ - 상수          │
│ - 추상 메서드│     │ - 추상 메서드   │      │ - 추상 메서드   │
└──────────────┘     │ - default 메서드│      │ - default 메서드│
                     │ - static 메서드 │      │ - static 메서드 │
                     └─────────────────┘      │ - private 메서드│ (내부 중복 로직 캡슐화)
                                              └─────────────────┘
```
```java
interface Printer {
    void print(String msg); // 추상 메서드 (자식에서 구현 필수)

    // default 메서드 1: 인터페이스 내부에서 공통 로직 제공
    default void printInfo(String msg) {
        logHeader("INFO");
        print(msg);
    }

    // default 메서드 2
    default void printError(String msg) {
        logHeader("ERROR");
        print(msg);
    }

    // private 메서드 (Java 9+): default 메서드들의 중복 로직을 내부로 숨김 (캡슐화)
    private void logHeader(String level) {
        System.out.println("=== [" + level + "] 출력 시작 ===");
    }
}

class ConsolePrinter implements Printer {
    // 순수 추상 메서드만 필수로 오버라이딩 (default 메서드는 선택적)
    @Override
    public void print(String msg) {
        System.out.println(msg);
    }
}

public class Main {
    public static void main(String[] args) {
        Printer p = new ConsolePrinter();
        // ConsolePrinter에는 printInfo가 없지만, 부모 인터페이스의 default 메서드를 실행함
        p.printInfo("시스템 정상 작동");  
    }
}
```
## 함수형 인터페이스와 람다식 (Functional Interface & Lambda)
- **함수형 인터페이스 (Functional Interface)**
    - **조건**: 내부에 ==단 하나의 순수 추상 메서드==만 존재하는 특수 인터페이스
    - **특징**: `default`나 `static` 메서드가 여러 개 포함되어 있어도 추상 메서드가 1개라면 조건 충족
    - **검증**: `@FunctionalInterface` 어노테이션 선언 시 컴파일러가 추상 메서드 단일성 엄격 검증
    - **어떻게?**: 스트림(Stream) API 처리나 이벤트 리스너 등에서 동작 파라미터(Behavior Parameterization)를 전달할 때 주로 활용
- **람다식 (Lambda Expression)**
    - **역할**: 익명 구현 객체 생성 시 발생하는 불필요한 보일러플레이트 코드 제거
    - **구조**: `(매개변수) -> { 실행문 }` 형태 (타입 추론이 가능하므로 매개변수 타입과 `return` 키워드 생략 가능)
    - **특징**: 메서드(동작) 자체를 일급 객체처럼 변수에 할당하거나 파라미터로 전달 가능
> [!example] 익명 내부 클래스와 람다식의 구현 비교
> 
> 람다식을 활용하면 인터페이스의 구현 객체 생성을 매우 직관적이고 간결하게 표현할 수 있습니다. 익명 클래스의 장황한 코드가 단 한 줄로 압축되는 것을 확인해 보세요.
```java
// 1. 함수형 인터페이스 선언 (추상 메서드가 1개뿐이어야 람다식 적용 가능)
@FunctionalInterface
interface Calculator {
    int calculate(int x, int y);
    // int add(int x, int y); // 주석 해제 시 추상 메서드가 2개가 되어 컴파일 에러 발생
}

public class Main {
    public static void main(String[] args) {
        // [방식 A] 기존 익명 내부 클래스 활용: 코드가 길고 가독성 저하
        Calculator calc1 = new Calculator() {
            @Override
            public int calculate(int x, int y) {
                return x + y;
            }
        };

        // [방식 B] 람다식 적용: 인터페이스 구현의 극단적 간소화
        // 컴파일러가 매개변수 타입(int)과 return을 유추하여 생략 가능
        Calculator calc2 = (x, y) -> x + y; 

        System.out.println(calc1.calculate(10, 20)); // 출력: 30
        System.out.println(calc2.calculate(10, 20)); // 출력: 30
    }
}
```
