# Spring MVC 아키텍처 및 요청 처리 전략
> [!summary] 핵심 요약
> 
> - Spring MVC의 전체 요청 처리 흐름, 파라미터 바인딩 및 데이터 응답 전략(JSP, REST, JPA) 명세
>     
> - 목적: 계층 간 느슨한 결합(Loose Coupling) 이해 및 데이터 수신/반환 전략 최적화
>     
> - 핵심: 커맨드 객체를 활용한 데이터 바인딩 및 응답 방식에 따른 컨트롤러 설계 숙달
>     
## 1. Controller 파라미터 수신 전략
- **클라이언트 데이터 바인딩 방식**
    - 클라이언트의 요청 데이터를 컨트롤러의 매개변수로 수신하는 4가지 핵심 전략

|**바인딩 방식**|**상세 설명**|**특징 및 활용**|
|---|---|---|
|HttpServletRequest|서블릿 기반 전통적 데이터 수신 방식|getParameter() 메서드 활용 (단일 데이터 수신)|
|@RequestParam|요청 파라미터 명시적 매핑|필수 여부(required) 및 기본값(defaultValue) 설정 가능|
|매개변수명 일치|어노테이션 생략 상태에서의 자동 바인딩|HTML input name과 변수명이 완전히 동일해야 동작|
|커맨드 객체(Command Object)|요청 파라미터와 특정 객체의 필드 간 자동 매핑|객체 내 Setter 필수, 대량 데이터 일괄 수신에 최적화|

> [!tldr]- 예시
> ```java
> @Controller
> public class UserController {
> 
>     /**
>      * 1. HttpServletRequest 방식
>      * 상세 설명: 서블릿 기반의 전통적인 방식
>      * 특징: getParameter()를 사용하여 직접 데이터를 추출 (단일 데이터 수신)
>      */
>     @PostMapping("/bind-request")
>     public String useHttpServletRequest(HttpServletRequest request) {
>         // 직접 파라미터 이름을 지정하여 꺼내와야 함 (문자열로 반환됨)
>         String name = request.getParameter("name");
>         String age = request.getParameter("age");
>         
>         System.out.println("HttpServletRequest -> 이름: " + name + ", 나이: " + age);
>         return "result";
>     }
> 
>     /**
>      * 2. @RequestParam 방식
>      * 상세 설명: 요청 파라미터를 명시적으로 매핑
>      * 특징: 필수 여부(required)나 기본값(defaultValue) 설정 가능
>      */
>     @PostMapping("/bind-param")
>     public String useRequestParam(
>             @RequestParam(name = "name") String userName, // 파라미터명과 변수명이 다를 때 유용
>             @RequestParam(value = "age", required = false, defaultValue = "0") int userAge
>     ) {
>         // age가 전달되지 않으면 기본값 0이 할당됨
>         System.out.println("@RequestParam -> 이름: " + userName + ", 나이: " + userAge);
>         return "result";
>     }
> 
>     /**
>      * 3. 매개변수명 일치 (어노테이션 생략) 방식
>      * 상세 설명: 어노테이션 생략 상태에서의 자동 바인딩
>      * 특징: HTML input의 name과 자바 변수명이 '완전히' 동일해야 함
>      */
>     @PostMapping("/bind-match")
>     public String useVariableMatch(String name, int age) {
>         // 별도의 어노테이션 없이도 변수명이 파라미터명과 같으면 자동으로 들어옴
>         System.out.println("변수명 일치 -> 이름: " + name + ", 나이: " + age);
>         return "result";
>     }
> 
>     /**
>      * 4. 커맨드 객체 (Command Object) 방식
>      * 상세 설명: 요청 파라미터와 특정 객체의 필드 간 자동 매핑
>      * 특징: 객체 내 Setter 필수, 대량 데이터(DTO) 일괄 수신에 최적화
>      */
>     @PostMapping("/bind-command")
>     public String useCommandObject(UserDTO userDTO) {
>         // Spring이 내부적으로 UserDTO의 Setter를 호출하여 데이터를 채워줌
>         // 예: userDTO.setName(request.getParameter("name"));
>         
>         System.out.println("커맨드 객체 -> " + userDTO.toString());
>         return "result";
>     }
> }
> ```

## 2. Spring MVC 아키텍처 통신 흐름
- **요청 처리 사이클 (Request Lifecycle)**
    - 클라이언트의 요청부터 데이터베이스 연동까지 이어지는 단방향 계층형 통신 구조
    - 각 계층은 부여된 역할만 수행하며, 인접한 하위 계층에만 의존 (Loose Coupling)
```plaintext
Spring MVC Architecture Flow
 ├─ 1. Client Request
 │   └─ DispatcherServlet: 모든 웹 요청을 가장 먼저 받는 프론트 컨트롤러 (Front Controller)
 ├─ 2. @Controller (Web Layer)
 │   ├─ 요청 파라미터 수신 및 바인딩
 │   └─ 비즈니스 로직 수행을 위해 Service 계층 호출
 ├─ 3. @Service (Business Layer)
 │   ├─ 핵심 비즈니스 로직 처리 및 @Transactional 트랜잭션 경계 설정
 │   └─ 데이터 연동을 위해 DAO/Repository 계층 호출
 └─ 4. @Repository (Data Layer)
     ├─ Mapper(MyBatis) 또는 EntityManager(JPA)를 통한 데이터베이스 접근
     └─ 벤더 종속적 DB 예외를 Spring 전용 예외(DataAccessException)로 추상화 변환
```
- **인터페이스 기반 설계 및 DB 연동 핵심 객체**

|**핵심 컴포넌트**|**역할 및 특징**|
|---|---|
|Service Interface / Impl|결합도 완화 및 확장성 확보를 위한 인터페이스-구현체 분리 설계|
|SqlSessionTemplate|MyBatis 쿼리 실행 및 트랜잭션 관리를 담당하는 핵심 빈(Bean) 객체|
|Mapper Namespace|XML 설정 내 정의된 네임스페이스와 쿼리 ID 호출을 통한 DB 연동|
|EntityManager|JPA 환경에서 영속성 컨텍스트(Persistence Context)를 관리하며 DB 연동 수행|

## 3. 데이터 응답 렌더링 패턴
- **[방식 1] JSP를 활용하는 뷰(View) 렌더링 패턴**
    - 처리 결과를 Model 객체에 담고, 이동할 JSP 파일의 논리적 이름(String) 반환
    - ViewResolver가 논리적 뷰 이름을 물리적 경로(예: `/WEB-INF/views/xxx.jsp`)로 변환하여 렌더링 수행
    - 데이터 전달: `Model.addAttribute()`를 통해 JSP에서 JSTL/EL로 접근할 데이터를 바인딩
```java
@Controller
@RequestMapping("/board")
public class BoardController {

    @Autowired
    private BoardService boardService;

    // JSP 페이지를 반환하는 전통적인 MVC 방식
    @GetMapping("/list")
    public String getBoardList(Model model) {
        // 1. Service를 통해 DB(Mapper)에서 데이터 조회
        List<Board> list = boardService.selectBoardList();
        
        // 2. JSP로 전달할 데이터를 Model에 담음
        model.addAttribute("boardList", list);
        
        // 3. ViewResolver가 렌더링할 JSP 파일명 반환
        return "board/list"; 
    }
}
```
- **[방식 2] Mapper 중심의 REST API 데이터 반환 패턴**
    - 뷰(JSP)를 렌더링하지 않고, DB에서 조회한 데이터 자체를 JSON/XML 포맷으로 클라이언트에 직접 반환
    - 데이터 전달: `HttpMessageConverter`가 자바 객체(DTO/List)를 JSON으로 자동 직렬화하여 HTTP Response Body에 삽입
    - 클라이언트 렌더링(React, Vue, 모바일 앱 등) 환경에서 필수적으로 사용
```java
@RestController // @Controller + @ResponseBody 결합 어노테이션
@RequestMapping("/api/board")
public class BoardRestController {

    @Autowired
    private BoardService boardService;

    // View 없이 JSON 데이터만 반환하는 RESTful 방식
    @GetMapping("/list")
    public ResponseEntity<List<Board>> getBoardList() {
        // 1. Service를 통해 DB(Mapper)에서 데이터 조회 (MyBatis는 보통 DTO를 바로 반환)
        List<Board> list = boardService.selectBoardList();
        
        // 2. 조회된 객체를 상태 코드와 함께 JSON 형태로 HTTP 본문에 직접 반환
        return ResponseEntity.ok(list);
    }
}
```
- **[방식 3] JPA 중심의 REST API 데이터 반환 (Entity & DTO 분리 패턴)**
    - 영속성 컨텍스트가 관리하는 ==Entity 객체를 직접 반환하지 않고, 반드시 API 스펙에 맞는 DTO로 변환==하여 반환
    - 데이터 전달: DB에서 조회한 Entity List를 자바 스트림(Stream)을 활용해 DTO List로 매핑 후 `HttpMessageConverter`를 통해 직렬화
> [!warning] Entity 직접 반환 금지
> 
> - **무한 루프(순환 참조)**: 양방향 연관관계를 가진 Entity를 그대로 직렬화하면 Jackson 라이브러리가 끝없이 참조를 따라가 뻗어버리는 문제 발생
>     
> - **API 스펙 결합**: DB 테이블 구조(Entity)가 외부로 직접 노출되며, Entity 내부 컬럼명이 변경되면 API 응답 스펙도 함께 깨지는 치명적 결함 유발
>     
> - **지연 로딩(Lazy Loading) 예외**: 트랜잭션 외부에서 프록시 객체를 초기화하려 할 때 `LazyInitializationException` 발생 위험
>     
```java
@RestController
@RequestMapping("/api/jpa/board")
@RequiredArgsConstructor // 생성자 주입
public class BoardJpaRestController {

    private final BoardJpaRepository boardRepository;

    // JPA 환경에서의 올바른 RESTful 데이터 반환 방식
    @GetMapping("/list")
    public ResponseEntity<List<BoardResponseDto>> getBoardList() {
        // 1. DB에서 Entity 조회 (보통 Service 계층에서 수행하는 것이 권장됨)
        List<BoardEntity> entities = boardRepository.findAll();
        
        // 2. Entity를 직접 반환하지 않고, 응답 전용 DTO로 변환 (순환 참조 방지 및 API 스펙 보호)
        List<BoardResponseDto> dtoList = entities.stream()
                .map(entity -> new BoardResponseDto(
                        entity.getId(), 
                        entity.getTitle(), 
                        entity.getAuthor()
                ))
                .collect(Collectors.toList());
        
        // 3. 변환된 DTO 리스트를 반환
        return ResponseEntity.ok(dtoList);
    }
}
```
