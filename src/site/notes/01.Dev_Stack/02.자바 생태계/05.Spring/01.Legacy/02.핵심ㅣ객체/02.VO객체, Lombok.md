# VO 객체와 Lombok 설계
> [!summary] 핵심 요약
> 
> - Lombok을 활용한 VO(Value Object)의 보일러플레이트 코드 제거 및 객체 생성 패턴
>     
> - 목적: 반복적인 코드(Getter/Setter/생성자) 자동화를 통한 가독성 향상 및 개발 생산성 확보
>     
> - 핵심: 목적에 맞는 어노테이션 조합, JSP EL 표기법 오류 방지를 위한 네이밍 규칙 준수 및 ==무분별한 @Data 사용 지양==
>     
## 주요 Lombok 어노테이션
- **핵심 어노테이션 종류 및 역할**
    - VO 및 DTO 객체 설계 시 반복 작성을 줄여주는 필수 어노테이션 목록

|**어노테이션**|**역할**|**비고**|
|---|---|---|
|`@NoArgsConstructor`|파라미터가 없는 기본 생성자 자동 생성|MyBatis, JPA 등 프레임워크 리플렉션 시 필수|
|`@AllArgsConstructor`|모든 필드를 매개변수로 받는 생성자 자동 생성|테스트 및 `@Builder` 연계 시 용이|
|`@Getter` / `@Setter`|필드의 접근자/설정자 자동 생성|캡슐화 보호를 위해 `@Setter`는 제한적 사용 권장|
|`@ToString`|객체의 상태를 문자열로 출력|양방향 연관관계 시 순환 참조 주의|
|`@Data`|`@Getter`, `@Setter`, `@ToString`, `@EqualsAndHashCode` 통합 지원|가장 편리하나 무분별한 캡슐화 훼손에 주의|
|`@Builder`|빌더 패턴(Builder Pattern) 지원|객체 생성 시 직관성 및 가독성 향상|

## Lombok 실무 적용 및 주의사항
- **무분별한 `@Data` 사용 지양 및 실무 권장 패턴**
    - `@Data`는 `@Setter`를 강제 개방하여 객체의 무결성(불변성)을 훼손할 수 있음
    - 내장된 `@ToString` 및 `@EqualsAndHashCode`가 JPA 등 ORM 환경에서 ==양방향 연관관계 객체를 순환 참조하여 무한 루프(StackOverflowError)를 발생==시킬 위험 존재
    - **실무 권장 방식**: 클래스 레벨에는 `@Getter`와 한정된 `@ToString`만 제한적으로 사용하고, `@Builder`는 클래스가 아닌 필요한 생성자나 메서드에 직접 적용할 것
- **`@ToString` 제어 트렌드 (`exclude` vs `of`)**
    - 객체 출력 시 연관관계 필드나 민감 정보의 노출을 막기 위해 속성 제어 필수
    - `exclude = "필드명"`: 출력에서 제외할 필드를 명시 (기존 방식)
    - ==`of = {"필드명1", "필드명2"}`==: 출력에 **포함할 필드만 명시** (최근 실무 권장 트렌드)
    - **권장 이유**: 향후 클래스에 새로운 필드가 추가되었을 때, 개발자의 실수로 민감한 정보가 로그 등에 의도치 않게 자동 노출되는 것을 원천 차단하기 위함
- **기본 생성자 접근 제어 (`AccessLevel.PROTECTED`)**
    - 의미 없는 빈 객체 생성을 막아 객체의 일관성과 무결성을 유지하기 위함
    - 특히 JPA 환경에서 엔티티(Entity)는 리플렉션(Reflection) 및 프록시(Proxy) 생성을 위해 기본 생성자가 필수적임
    - 기본 생성자를 `PRIVATE`로 설정 시 JPA가 프록시 객체를 생성하지 못해 에러가 발생하므로, ==JPA 스펙이 허용하는 가장 엄격한 제어자인 `PROTECTED`를 사용함==
```plaintext
Default Constructor Access Level Check (JPA Environment)
 ├─ PUBLIC
 │   └─ [경고] 무분별한 빈 객체 생성 허용 -> 객체 무결성 훼손
 ├─ PRIVATE
 │   └─ [에러] 상속 불가로 인한 JPA 프록시 객체 생성 실패 -> 지연 로딩(Lazy Loading) 불가
 └─ PROTECTED
     └─ [권장] JPA 프록시 생성 허용 + 외부의 무의미한 객체 생성 차단 -> 무결성 유지
```
- **필드 네이밍 규칙 (Naming Convention)**
    - ==필드명의 앞글자 2개는 반드시 소문자로 연속 작성==할 것
    - 잘못된 예: `uName` (JSP EL 표기법이나 Getter/Setter 자동 파싱 과정에서 인식 오류 발생 위험)
    - 올바른 예: `userName`
```java
/*
 * 실무 권장 패턴 및 올바른 네이밍 규칙을 적용한 Member 예시
 */
@Getter
// @ToString(exclude = "team") // 기존 방식: team 필드만 제외
@ToString(of = {"userId", "userName"}) // 최근 트렌드: 안전하게 노출할 필드만 명시적 포함
@NoArgsConstructor(access = AccessLevel.PROTECTED) // 기본 생성자 접근 제어 (안전성 확보)
public class Member {
    private String userId;
    private String userPwd; // 노출되면 안 되는 민감 정보 (of 속성 덕분에 자동 제외됨)
    private String userName; // uName(X) -> userName(O)
    private String email;
    
    // 실무에서는 클래스 레벨 대신 생성자에 Builder를 적용하여 필수/선택 값을 명확히 제어
    @Builder
    public Member(String userId, String userPwd, String userName, String email) {
        this.userId = userId;
        this.userPwd = userPwd;
        this.userName = userName;
        this.email = email;
    }
}
```
## Builder 패턴의 활용
- **객체 생성 (Object Creation)**
    - 메서드 체이닝을 통해 특정 필드에 주입되는 값을 직관적으로 확인
    - 전체 필드 주입뿐만 아니라, 필요한 일부 필드(Partial)만 선택적 초기화 가능 (나머지는 `null` 또는 기본값)
```java
public class BuilderExample {
    public static void main(String[] args) {
        // 1. 전체 필드 설정 (모든 데이터 주입)
        Member fullMember = Member.builder()
                                .userId("user123")
                                .userPwd("password!")
                                .userName("홍길동")
                                .email("hong.gd@example.com")
                                .build(); // 반드시 마지막에 .build() 호출

        // 2. 일부 필드만 설정 (필수 필드만 할당 시 유용)
        Member partialMember = Member.builder()
                                 .userId("guest456")
                                 .userName("방문자")
                                 // userPwd, email은 명시하지 않았으므로 null로 초기화됨
                                 .build();
    }
}
```
- **기존 객체 복사 및 부분 수정 (`toBuilder`)**
    - 원본 객체의 데이터는 유지하면서 특정 필드 값만 갱신된 새로운 객체 생성
    - 클래스 또는 생성자의 `@Builder` 선언부에 ==`toBuilder = true`== 속성 활성화 필수
```java
// 1. 원본 객체에서 데이터를 복사하여 새로운 빌더 인스턴스 생성
MemberBuilder newMemberBuilder = originalMember.toBuilder();

// 2. 수정이 필요한 필드만 체이닝으로 덮어쓰기
Member updatedMember = newMemberBuilder
                        .userPwd("new_password_456") // 값 변경 적용
                        .email("hong.new@example.com") // 값 변경 적용
                        // userId와 userName은 originalMember의 원본 값 그대로 유지
                        .build(); 
```
