# 📌 Spring 경로(Path)

> [!summary] 요약
> Spring 웹 개발 시 혼동하기 쉬운 4가지 경로 개념(**URL, View, Package, File**)을 명확히 구분하고, 상황별(Controller, JSP, XML) 올바른 사용법을 정리한 노트임.

## 1\. 4가지 경로 개념 구분

| 구분                    | 형태 (Syntax)                     | 사용 주체                        | 주 용도                            |
| :-------------------- | :------------------------------ | :--------------------------- | :------------------------------ |
| **1. URL (웹 주소)**🌐   | `/`로 시작<br>`/list.bo`           | **브라우저**<br>(Client)         | 컨트롤러를 호출하거나,<br>다른 주소로 이동할 때    |
| **2. View 경로 (화면)**📂 | `/`로 구분 (확장자 X)<br>`board/list` | **Spring**<br>(ViewResolver) | 사용자에게 보여줄<br>**JSP 파일**을 지정할 때  |
| **3. 패키지 경로 (자바)**📦  | `.`으로 구분<br>`com.kh...`         | **Java/XML**<br>(Compiler)   | 클래스 파일을 로딩하거나<br>빈(Bean)을 등록할 때 |
| **4. 실제 경로 (파일)**💾   | `\` 또는 `/`<br>`C:\upload`       | **OS (운영체제)**                | 파일을 실제로 저장하거나<br>삭제할 때 (업로드 등)  |

```text
C:\dev\workspace\springProject  <-✨ 실제 경로
├── .metadata
├── src
│   └── main
│       └── java <- ✨ 패키지 경로 src.main.java
...
└── WebContent  <- ✨ URLRoot(Context Path)가 여기서부터 논리적으로 시작
    │                 이 이후는 매핑으로 url을 직접 만들어야한다.(폴더 구조와 무관함)
    ├── resources
    │   └── css
    │       └── style.css
    └── WEB-INF
        ├── views <- ✨ 보통 ViewResolver 를 적용하는 곳
        │   └── board
        │       └── boardListView.jsp
		...
```
## 2\. 상황별 사용법 (코드 예시)⭐

### A. Controller에서 `return` 할 때
- 컨트롤러 메서드의 반환값에 따라 **보여줄 화면**인지, **이동할 주소**인지가 결정됨.

#### 1) 화면(JSP)을 보여줄 때 (Forwarding)

  * $ **형식:** View 경로 (`폴더명/파일명`)
  * **동작:** `ViewResolver`가 `prefix`와 `suffix`를 붙여 실제 JSP 파일을 찾음.

```java
// 실제 파일 위치: /WEB-INF/views/board/boardListView.jsp
return "board/boardListView"; 
```

#### 2) 다른 주소로 보낼 때 (Redirect)

  * $ **형식:** URL 경로(`redirect:` + `URL`)
  * **동작:** 브라우저에게 "이 주소로 다시 요청해\!"라고 명령함 (새로고침 효과).

```java
// 작업 완료 후 목록 페이지(URL) 재요청
return "redirect:list.bo"; 
```

### B. AJAX 요청 보낼 때 (JSP/JS)

  * **형식:** `URL`
  * **동작:** 컨트롤러의 `@RequestMapping` 매핑값과 일치해야 함.

```javascript
$.ajax({
    url : "topList.bo", // Controller의 @RequestMapping 값
    // ...
});
```

### C. XML 설정 파일 (`servlet-context.xml`)

  * **형식:** `패키지 경로` (Full Class Name)
  * **동작:** 스프링이 해당 클래스를 찾아 객체(Bean)로 생성함.

```xml
<beans:bean class="com.kh.spring.common.interceptor.TestInterceptor"/>
```

## 3\. ✨ 핵심 포인트 & 팁 (Plus Alpha)

### 💡 1. ViewResolver의 역할

`return "board/list";`만 써도 되는 이유는 `servlet-context.xml` 설정 덕분임.

  * **Prefix**: `/WEB-INF/views/` (앞)
  * **Suffix**: `.jsp` (뒤)
  * **결과**: `/WEB-INF/views/` + `board/list` + `.jsp`

### 💡 2. Context Path (프로젝트명) 주의

개발 시 `/list.bo`라고 써도, 실제 브라우저는 `localhost:8080/projectName/list.bo`로 인식함. JSP에서 경로 작성 시 **Context Path**를 포함하는 것이 안전함.

```jsp
<form action="${pageContext.request.contextPath}/insert.bo" method="post">

<form action="/spring/insert.bo" method="post">
```

### 💡 3. 정적 리소스 (CSS, JS, Image) 경로 매핑 (추가됨)

JSP 파일 내에서 이미지나 CSS가 안 보인다면 **Resource Mapping**을 확인해야 함.

  * **설정 (servlet-context.xml)**:
    URL로 `/resources/**` 요청이 오면, 실제 폴더 `/resources/`를 보여주라는 뜻.
    ```xml
    <resources mapping="/resources/**" location="/resources/" />
    ```
  * **사용 (JSP)**:
    ```html
    <link href="${pageContext.request.contextPath}/resources/css/style.css" rel="stylesheet">
    ```

### 💡 4. 데이터 전송 경로 패턴 (QueryString vs PathVariable) (추가됨)

URL 뒤에 데이터를 붙여서 보낼 때 두 가지 방식이 있음.

| 방식 | 형태 | 받는 방법 (Controller) | 용도 |
| :--- | :--- | :--- | :--- |
| **Query String** | `/detail.bo?no=3` | `@RequestParam int no` | 검색 필터, 페이징, 일반적인 GET 데이터 |
| **Path Variable** | `/detail/3` | `@PathVariable int no` | 리소스 식별 (REST API 스타일) |
