# Spring JPA 서비스 계층 핵심 패턴 및 문법
> [!summary] 핵심 요약
> 
> 스프링 데이터 JPA의 서비스(Service) 계층에서 빈번하게 활용되는 예외 처리, 트랜잭션 제어, 영속성 컨텍스트 활용 기법 요약. `Optional` 객체 제어를 통한 안전한 데이터 검증 및 비즈니스 로직의 안정성 확보.
## Optional 기반 데이터 조회 및 예외 처리
- **`orElseThrow()`**
    - `findById()` 등 단건 조회 결과가 없을 경우 특정 예외를 발생시키는 가장 표준적인 방식
    - 람다식(`() -> new Exception(...)`)을 활용하여 상황에 맞는 커스텀 예외(Custom Exception) 발생 구조
- **`orElseGet()`**
    - 조회된 데이터가 없을 때만 특정 로직이나 내부 메서드를 실행하여 기본값을 반환하는 지연 연산(Lazy Evaluation) 방식
- **`orElse()`**
    - 데이터 존재 여부와 무관하게 괄호 안의 객체를 항상 메모리에 생성하는 방식 (사용 시 불필요한 객체 생성에 따른 성능 저하 주의 필요)
- **`ifPresent()`**
	- 값이 존재할때만 따라오는 람다식을 실행
	- NullpointerException 발생하지 않도록 지정
```java
// Optional 예외 처리 및 데이터 추출 예시
@Service
public class MemberService {
    
    // 조회 실패 시 즉시 예외 발생 (가장 권장되는 패턴)
    public Member findMember(Long memberId) {
        return memberRepository.findById(memberId)
                .orElseThrow(
	                () -> new IllegalArgumentException("존재하지 않는 회원입니다. ID: " + memberId));
    }
}
```
## 트랜잭션 관리 및 영속성 컨텍스트 제어
- **`@Transactional`**
    - 서비스 계층의 메서드 내 모든 데이터베이스 연산을 하나의 논리적 작업 단위로 묶는 어노테이션
    - 런타임 예외(RuntimeException) 발생 시 자동 롤백 수행 처리
- **`readOnly = true` 속성 부여**
    - 데이터 변경(C/U/D)이 없는 단순 조회 메서드에 적용하여 JPA의 성능 최적화
    - 영속성 컨텍스트의 스냅샷 생성 방지 및 플러시(Flush) 생략으로 메모리 사용량 감소
- **Dirty Checking (변경 감지)**
    - 트랜잭션 범위 내에서 조회한 영속 상태의 엔티티는 필드 값 변경 시 자동으로 Update 쿼리 발생
    - 수정 로직 수행 후 별도의 `repository.save()` 명시적 호출 불필요
```java
// 트랜잭션 및 Dirty Checking 활용 예시
@Service
public class MemberService {

    // 단순 조회용 트랜잭션 (성능 최적화)
    @Transactional(readOnly = true)
    public MemberResponseDto getMember(Long memberId) {
        Member member = memberRepository.findById(memberId)
                .orElseThrow(() -> new EntityNotFoundException("회원 조회 실패"));
        return new MemberResponseDto(member);
    }

    // 수정용 트랜잭션 (Dirty Checking 작동)
    @Transactional
    public void updateMemberName(Long memberId, String newName) {
        Member member = memberRepository.findById(memberId)
                .orElseThrow(() -> new EntityNotFoundException("회원 조회 실패"));
        
        // 엔티티의 상태만 변경하면 트랜잭션 종료 시 데이터베이스에 자동 반영 (save 호출 불필요)
        member.updateName(newName);
    }
}
```
## 데이터 검증 및 비즈니스 로직 패턴
- **`existsBy...` 메서드 활용 검증**
    - 회원가입 시 이메일 중복 체크 등 특정 데이터 존재 여부만 `boolean` 타입으로 확인하는 방식
    - 전체 엔티티를 조회(`findBy...`)하는 것보다 쿼리 성능이 우수하여 조건 검증에 적합
- **도메인 모델 패턴 (Domain Model Pattern)**
    - 데이터의 상태 변경 로직(비즈니스 로직)을 서비스 계층이 아닌 엔티티 내부의 메서드로 위임하는 설계 방식
    - 서비스 계층은 트랜잭션 시작, 엔티티 조회, 엔티티 메서드 호출의 흐름만 담당하도록 역할 분리
