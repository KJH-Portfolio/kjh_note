# Spring Data JPA 심화: 성능 최적화, 동시성 제어 및 페이징
> [!summary] 핵심 요약
> 
> 실무 환경에서의 성능 최적화, 동시성 제어 및 대용량 데이터 조회를 위한 JPA 핵심 기능 요약.
> 
> ==N+1 문제== 해결을 위한 `@EntityGraph`, 데이터 정합성을 위한 락(`@Lock`, `@Version`) 전략, 충돌 시 자동 재시도를 위한 `@Retryable`, 무한 스크롤에 최적화된 `Slice` 객체의 원리 및 활용.
## 성능 최적화: @EntityGraph
- **`@EntityGraph`의 목적 및 특징**
    - JPA 사용 시 발생하는 대표적 성능 이슈인 ==N+1 문제== 해결의 핵심 요소
    - 연관된 엔티티를 한 번의 SQL로 함께 가져오는 방식 (Eager Load)
    - 복잡한 JPQL 작성 없이도 ==페치 조인(Fetch Join)==과 동일한 효과 부여
    - 코드 가독성 향상 및 쿼리 메서드와의 조합 용이
```java
public interface MemberRepository extends JpaRepository<Member, Long> {

    // Member 조회 시 연관된 Team 엔티티도 한 번에(JOIN) 로드
    @EntityGraph(attributePaths = {"team"})
    List<Member> findAll();
}
```
## 데이터 정합성 보장: 락(Lock) 전략
- **동시성 제어의 필요성**
    - 다중 사용자의 동일 데이터 동시 수정 시 발생하는 데이터 꼬임 현상 방지
    - 돈, 재고 등 정합성이 생명인 도메인에서의 필수적 동시성 제어 도구
- **비관적 락 (Pessimistic Lock)**
    - 데이터베이스 수준에서 실제 락을 걸어 타 트랜잭션의 접근 원천 차단
    - 주로 `@Lock(LockModeType.PESSIMISTIC_WRITE)` 형태 (SELECT ... FOR UPDATE)로 활용
- **낙관적 락 (Optimistic Lock)**
    - 애플리케이션 수준에서의 데이터 충돌 감지 방식
    - 엔티티 내부의 `@Version` 어노테이션을 통한 버전 관리 로직 활용
    - 트랜잭션 커밋 시점에 버전을 비교하여 충돌 발생 시 예외(`ObjectOptimisticLockingFailureException`) 발생
- **낙관적 락 예외 처리: @Retryable**
    - 낙관적 락 충돌 발생 시 비즈니스 로직 단에서의 자동 재시도(Retry) 구현을 위한 어노테이션
    - Spring Retry 라이브러리 의존성 추가 필요 (`@EnableRetry` 설정 동반)
    - `value` 속성으로 대상 예외 지정, `maxAttempts`로 최대 재시도 횟수 설정, `backoff`로 재시도 간격 조정
```java
// 1. 낙관적 락을 위한 엔티티 설정
@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    
    @Version
    private Long version; // 데이터 수정 시 자동 증가 및 충돌 감지용 필드
}

// 2. 비관적 락을 위한 레포지토리 설정
public interface MemberRepository extends JpaRepository<Member, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("select m from Member m where m.id = :id")
    Optional<Member> findByIdWithPessimisticLock(@Param("id") Long id);
}

// 3. 낙관적 락 충돌 시 자동 재시도를 위한 서비스 계층 설정 (@Retryable)
import org.springframework.retry.annotation.Backoff;
import org.springframework.retry.annotation.Retryable;
import org.springframework.orm.ObjectOptimisticLockingFailureException;

@Service
public class MemberService {

    // 낙관적 락 충돌 예외 발생 시 최대 3번, 1초(1000ms) 간격으로 자동 재시도
    @Retryable(
        value = ObjectOptimisticLockingFailureException.class,
        maxAttempts = 3,
        backoff = @Backoff(delay = 1000)
    )
    @Transactional
    public void updateMemberProfile(Long memberId, String newName) {
        // ... (조회 및 수정 로직)
    }
}
```
## 대용량 데이터 조회: Slice
- **`Slice` 인터페이스의 특징 및 이점**
    - 전체 데이터 개수를 세는 `Count` 쿼리를 생략하여 조회 성능 극대화
    - 요청한 `size`보다 1개의 데이터를 추가로 조회하여 다음 페이지 존재 여부(`hasNext()`) 파악
    - 전체 페이지 수 파악이 불필요한 모바일 UI의 ==무한 스크롤(Infinite Scroll)== 구현에 최적화
    - `Page` 인터페이스 대비 불필요한 DB 부하 감소 및 응답 속도 향상
```java
import org.springframework.data.domain.Pageable;
import org.springframework.data.domain.Slice;

public interface MemberRepository extends JpaRepository<Member, Long> {

    // Count 쿼리 없이 limit + 1 개수만 조회하여 무한 스크롤에 활용
    Slice<Member> findSliceByAge(int age, Pageable pageable);
}
```
## Pageable 사용법
### 1. Pageable의 핵심 3요소
가장 기본적으로 다음 정보들을 조합해서 요청할 수 있습니다.
- **`page` (페이지 번호):** 몇 번째 페이지를 볼 것인가?
    - **주의:** 0부터 시작합니다. (0페이지가 첫 번째 페이지!)
- **`size` (페이지당 개수):** 한 페이지에 몇 개의 데이터를 담을 것인가? (기본값 20)
- **`sort` (정렬 조건):** 어떤 기준으로 정렬할 것인가?
    - 예: `id,desc` (ID 내림차순), `name,asc` (이름 오름차순)
---
### 2. 실제로 어떻게 사용하나요?
#### ① 클라이언트가 URL로 요청할 때
프론트엔드에서 서버로 API를 호출할 때 쿼리 파라미터로 아주 쉽게 던질 수 있습니다.
> `http://localhost:8080/members?page=0&size=10&sort=name,desc`
> 
> ➜ "이름 내림차순으로 정렬해서, 첫 번째 페이지의 데이터 10개만 가져다줘!"
#### ② 서버 코드 내에서 직접 만들 때 (`PageRequest`)
테스트 코드나 서비스 로직 안에서 직접 `Pageable` 객체를 생성해야 할 때는 `PageRequest`라는 구현체를 사용합니다.
```java
// 페이지 번호, 사이즈, 정렬 조건을 직접 지정
Pageable pageable = PageRequest.of(0, 10, Sort.by("name").descending());

// 레포지토리에 그대로 전달
sliceRepository.findSliceByAge(20, pageable);
```
#### ③ 컨트롤러에서 기본값 설정하기 (`@PageableDefault`)
사용자가 아무런 조건을 안 보냈을 때를 대비해 기본값을 설정할 수도 있습니다.
```java
@GetMapping("/members")
public Slice<Member> getMembers(
    @PageableDefault(size = 5, sort = "id", direction = Sort.Direction.DESC) Pageable pageable) {
    return memberService.findAll(pageable);
}
```
---
### 3. Pageable이 내부적으로 만드는 SQL (Limit & Offset)
우리가 `Pageable`을 넘기면, Spring Data JPA는 이걸 해석해서 DB가 알아듣는 **SQL(Limit, Offset)** 문법으로 번역해 줍니다.
- **`size`** ➜ SQL의 **`LIMIT`** (몇 개 가져올지)
- **`page * size`** ➜ SQL의 **`OFFSET`** (어디서부터 건너뛸지)
> **예시:** `page=2, size=10` 이라면?
> 
> ➜ 앞에 20개는 건너뛰고(`OFFSET 20`), 그 다음부터 10개만 가져와(`LIMIT 10`)!
