# Spring Boot & Caffeine Cache 구현 전략
> [!summary] 핵심 요약
> 
> Spring Boot의 PSA 기반 캐시 추상화와 Caffeine의 고성능 In-memory 기술을 결합한 백엔드 최적화 아키텍처.
> 
> 반복적인 I/O 발생 지점에 Look-aside 패턴을 적용하여 Latency 감소 및 처리량(Throughput) 극대화.
> 
> 어노테이션 기반의 선언적 적용을 통해 비즈니스 로직과 캐싱 로직의 결합도 최소화.
## 1. 캐싱(Caching) 아키텍처 개요
- **캐시(Cache) 정의**
    - 고속 데이터 접근을 위한 임시 저장 계층.
    - 저장소 속도 차이(Disk vs Memory)에 의한 병목 현상 해소.
- **도입 목적**
    - **성능 최적화:** DB/Disk 접근 최소화를 통한 응답 속도(Latency) 개선.
    - **리소스 효율화:** 반복 연산 및 쿼리 부하 감소.
- **적용 타겟 (Target Criteria)**
    - 데이터 변경 빈도가 낮고 조회 빈도가 높은 정보 (예: 공통 코드, 카테고리).
    - 동일 입력에 대해 항상 동일 출력이 보장되는 연산 (Idempotent).
## 2. Spring Boot Server-Side 구성
- **Spring Cache Abstraction**
    - **PSA (Portable Service Abstraction):** 구현체(Redis, Caffeine 등)와 무관한 일관된 캐싱 인터페이스 제공.
    - **CacheManager:** 캐시 생명주기 및 설정을 관리하는 최상위 인터페이스.
- **Caffeine Cache 특성**
    - **고성능 In-Memory:** Java 8 기반, Guava Cache를 계승한 로컬 캐시.
    - **W-TinyLFU 알고리즘:** 최적화된 퇴출(Eviction) 알고리즘을 통한 높은 적중률(Hit Rate) 보장.
    - **로컬 캐시 한계:** 단일 서버 메모리 사용, 분산 환경(Scale-out) 시 데이터 불일치 가능성 존재.
### 필수 의존성 (Gradle)
```gradle
dependencies {
    // Spring Boot 캐시 추상화
    implementation 'org.springframework.boot:spring-boot-starter-cache'
    // Caffeine 구현체
    implementation 'com.github.ben-manes.caffeine:caffeine'
}
```
### 환경 설정 (Configuration)
- **주요 설정 포인트**
    - **TTL (Time To Live):** `expireAfterWrite` (저장 후 만료), `expireAfterAccess` (조회 후 만료).
    - **Eviction:** `maximumSize` (메모리 보호를 위한 최대 엔트리 제한).
```java
@Configuration
@EnableCaching // 캐싱 기능 활성화
public class CacheConfig {

    @Bean
    public Caffeine<Object, Object> caffeineConfig() {
        return Caffeine.newBuilder()
                .expireAfterWrite(10, TimeUnit.MINUTES) // 10분 후 만료
                .maximumSize(1000) // 최대 1000개 제한
                .recordStats(); // 모니터링용 통계 기록
    }

    @Bean
    public CacheManager cacheManager(Caffeine<Object, Object> caffeine) {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(caffeine);
        return cacheManager;
    }
}
```
## 3. 실무 적용 (Method Level Implementation)
- **동작 원리 (AOP 기반)**
    - 프록시 패턴을 사용하여 메서드 호출 전후에 캐시 로직 개입.
    - **Look-aside 전략:** 캐시 확인(Hit) → 반환 / 캐시 없음(Miss) → 로직 실행 → 캐시 저장.
### 어노테이션 상세 명세
- **@Cacheable (조회 최적화)**
    - 데이터 조회 메서드에 적용.
    - 캐시 Hit 시 메서드 본문 실행 생략(Skip) 후 저장된 값 반환.
- **@CachePut (동기화 갱신)**
    - 데이터 수정/생성 시 적용.
    - 메서드를 **항상 실행**하고, 반환값을 캐시에 덮어쓰기(Update).
- **@CacheEvict (만료 및 삭제)**
    - 데이터 삭제 시 적용.
    - 특정 키 또는 전체 캐시(`allEntries=true`) 제거.
### 적용 예시 코드 (Service Layer)
```java
@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;

    /**
     * [조회] 캐시 적용
     * key: 캐시 식별자 (spEL 사용 가능, 여기서는 id)
     * value: config에서 설정한 캐시 저장소 이름
     * unless: 특정 조건(예: 결과가 null)일 경우 캐시하지 않음
     */
    @Cacheable(value = "products", key = "#id", unless = "#result == null")
    public ProductDto getProduct(Long id) {
        return productRepository.findById(id)
                .map(ProductDto::from)
                .orElse(null);
    }

    /**
     * [수정] 캐시 갱신
     * DB 업데이트 후 반환값을 캐시에 다시 저장하여 정합성 유지
     */
    @CachePut(value = "products", key = "#id")
    public ProductDto updateProduct(Long id, ProductDto distinctDto) {
        Product product = productRepository.findById(id)
                .orElseThrow(() -> new EntityNotFoundException("Product not found"));
        
        product.updateInfo(distinctDto);
        Product saved = productRepository.save(product);
        
        return ProductDto.from(saved);
    }

    /**
     * [삭제] 캐시 제거
     * DB 데이터 삭제 시 캐시 데이터도 함께 제거
     */
    @CacheEvict(value = "products", key = "#id")
    public void deleteProduct(Long id) {
        productRepository.deleteById(id);
    }
    
    /**
     * [전체 삭제] 리스트 조회 등 데이터 변경 범위가 클 때 유용
     */
    @CacheEvict(value = "products", allEntries = true)
    public void clearCache() {
        // 모든 'products' 캐시 데이터 삭제
    }
}
```
