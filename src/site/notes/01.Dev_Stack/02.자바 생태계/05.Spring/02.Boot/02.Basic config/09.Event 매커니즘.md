# Spring Event 기반 비즈니스 로직 분리 가이드
> [!summary] 핵심 요약
> 
> **Spring Event**를 활용하여 핵심 비즈니스 로직과 부가 기능(알림, 로그 등)의 **강한 결합도를 해소**하고 관심사를 분리함. **트랜잭션 상태**에 따른 실행 시점 제어와 **비동기 처리**를 통해 시스템 안정성 및 사용자 응답 속도를 최적화하는 아키텍처 구현 가이드임.
## 1. 프로젝트 설정 및 이벤트 정의
- **비동기 처리 활성화**
    - 이벤트 리스너를 별도 스레드에서 실행하려면 `@EnableAsync` 설정.
    - 애플리케이션 메인 클래스 또는 설정 클래스에 선언 필요.
```java
@Configuration
@EnableAsync
public class AsyncConfig {
    // 비동기 실행을 위한 ThreadPoolTaskExecutor 추가 설정 가능
}
```
- $ [[3. Resource/01.Dev_Stack/02.자바 생태계/05.Spring/02.Boot/02.Basic config/08.AsyncConfig\|08.AsyncConfig]] ThreadPoolTaskExecutor 관련 참조
- **Event Class (POJO/Record)**
    - 전달할 데이터를 담는 순수 객체.
    - Spring 4.2 이후 `ApplicationEvent` 상속 없이 일반 객체(POJO) 또는 `record` 사용 가능.
```java
// 채팅 초대 정보를 담는 이벤트 객체
public record ChatInvitationEvent(
    Long chatRoomId,
    Long inviterId,
    List<Long> inviteeIds
) {}
```
## 2. 이벤트 발행 및 구독 구현
- **Publisher (발행자)**
    - `ApplicationEventPublisher`를 주입받아 `publishEvent()` 메서드 호출.
    - 핵심 로직 종료 시점에 이벤트를 던져 부가 로직과의 의존성 제거.
```java
@Service
@RequiredArgsConstructor
public class ChatService {
    private final ApplicationEventPublisher eventPublisher;

    @Transactional
    public void inviteUsers(Long roomId, Long inviterId, List<Long> inviteeIds) {
        // 1. 핵심 로직: DB 저장 등
        // chatRepository.save(...);

        // 2. 이벤트 발행: 부가 기능 호출을 이벤트로 위임
        eventPublisher.publishEvent(new ChatInvitationEvent(roomId, inviterId, inviteeIds));
    }
}
```
- **Listener (구독자)**
    - `@EventListener`를 사용하여 발행된 이벤트를 수신.
    - 하나의 이벤트에 대해 여러 리스너를 생성하여 확장 가능(OCP 준수).
```java
@Component
@Slf4j
public class NotificationListener {

    @EventListener
    public void handleBasicEvent(ChatInvitationEvent event) {
        log.info("이벤트 수신 완료: {}", event.chatRoomId());
    }
}
```
## 3. 트랜잭션 및 실행 순서 정밀 제어
- **트랜잭션 동기화 (`@TransactionalEventListener`)**
    - `phase` 옵션을 통해 핵심 로직의 트랜잭션 상태(성공/실패)에 따라 실행 시점 결정.
    - `AFTER_COMMIT`: 메인 로직이 DB에 완전히 반영된 후 실행하여 데이터 무결성 보장.
- **실행 순서 지정 (`@Order`)**
    - 동일 이벤트를 처리하는 여러 리스너의 우선순위 결정.
    - 낮은 숫자일수록 먼저 실행됨.
- **AFTER_COMMIT 시점의 DB 쓰기 주의사항**
    - 커밋 완료 후에는 기존 트랜잭션이 종료된 상태이므로 추가 DB 수정 시 `@Transactional(propagation = Propagation.REQUIRES_NEW)` 필수 사용.
```java
@Component
@RequiredArgsConstructor
public class AdvancedListener {

    private final LogService logService;

    @Order(1) // 첫 번째로 실행
    @Async // 비동기 스레드에서 실행
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Transactional(propagation = Propagation.REQUIRES_NEW) // 새 트랜잭션 시작
    public void handleWithTransaction(ChatInvitationEvent event) {
        // 커밋 후 별도 트랜잭션에서 로그 DB 기록 및 외부 API 호출
        logService.saveHistory(event);
    }
}
```

## 3. [코드 분석] 실시간 채팅 이벤트 리스너 (ChatEventListener)
- **핵심 역할: 데이터 무결성 보장 및 알림 전송**
    - DB에 채팅 내용이 완전히 저장된 것을 확인한 후 프론트엔드에 웹소켓 메시지 전송.
    - 데이터 불일치(Dirty Read) 현상 방지를 위해 트랜잭션 커밋 시점을 동기화함.
- **주요 어노테이션 및 필드 분석**
    - **`SimpMessagingTemplate`**: 스프링 메시징 인프라를 활용하여 특정 경로(Topic)로 데이터를 송신하는 도구.
    - **`@TransactionalEventListener`**: 단순 이벤트 수신을 넘어, 발행 측의 트랜잭션 상태에 종속되어 실행.
```java
@Slf4j
@Component
@RequiredArgsConstructor
public class ChatEventListener {

    private final SimpMessagingTemplate messagingTemplate;

    /**
     * [분석] TransactionPhase.AFTER_COMMIT 사용 이유
     * 1. 문제 상황: DB 저장이 완료되기 전 웹소켓 알림이 먼저 도달하면, 프론트엔드에서 데이터를 조회할 때 결과가 없을 수 있음.
     * 2. 해결 방법: 커밋(Commit)이 완료된 '직후'에만 실행되도록 강제하여 동시성 이슈 해결.
     */
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handleChatEvent(ChatEvent event) {
        // 특정 구독 대상(/topic/...)에게 실제 페이로드 전송
        messagingTemplate.convertAndSend(event.getDestination(), event.getPayload());
    }
}
```
