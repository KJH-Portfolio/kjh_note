# 리액트 컨텍스트 API (Context API) 및 전역 상태 관리
> [!summary] 요약: 컴포넌트 트리 전역 데이터 공유 및 Props Drilling 해결을 위한 리액트 내장 메커니즘. 계층 구조와 무관한 데이터 접근성 확보 및 관리 효율성 증대 목적.
---
## 1. 핵심 개념 및 도입 배경
- **Props Drilling 해결**
    - 하위 자손에게 데이터를 전달하기 위한 중간 컴포넌트들의 불필요한 Props 수령 구조 개선.
- **전역 데이터 공유**
    - 로그인 정보, 테마(다크모드), 언어 설정 등 앱 전반 공통 데이터의 효율적 관리.
- **컴포넌트 결합도 분리**
    - 데이터 전달 목적의 중간 단계 코드 제거를 통한 가독성 및 유지보수성 향상.
---
## 2. Context API 3대 구성 요소

|**요소**|**역할**|**주요 특징**|
|---|---|---|
|**`createContext`**|**저장소 생성**|전역 데이터를 담을 객체 생성 및 기본값 설정 기능.|
|**`Provider`**|**데이터 제공**|하위 컴포넌트에 데이터 주입 및 공유 범위 결정 래퍼(Wrapper).|
|**`useContext`**|**데이터 사용**|필요한 위치에서 직접 Context 데이터를 추출하여 사용하는 훅(Hook).|

---
## 3. 실전 구현 프로세스 (Custom Provider 패턴)
### 3.1 Context 및 Custom Provider 정의
- **상태와 로직의 캡슐화**: 상태 관리와 업데이트 함수를 하나의 Provider 컴포넌트 내부에 구성.
- **성능 최적화**: `useMemo`를 통한 객체 주소값 고정으로 불필요한 하위 리렌더링 차단.
```javascript
import { createContext, useState, useMemo } from 'react';

// // 1. 전역 데이터 저장소 생성 (초기값 null 설정)
export const UserContext = createContext(null);

// // 2. 데이터 제공 및 상태 관리를 전담하는 래퍼 컴포넌트 정의
export const UserProvider = ({ children }) => {
  // // 3. 공유할 전역 상태 정의 (초기 사용자 정보)
  const [user, setUser] = useState({ name: "홍길동", role: "Guest" });

  // // 4. [중요] value 객체 메모이제이션
  // // 매 렌더링마다 새로운 객체가 생성되어 하위 컴포넌트가 재렌더링되는 현상 방지
  // // user 상태가 변경될 때만 새로운 객체 참조 생성
  const value = useMemo(() => ({
    user, // // 현재 유저 상태 값
    login: (name) => setUser({ name, role: "Admin" }), // // 유저 로그인 업데이트 함수
    logout: () => setUser(null) // // 유저 로그아웃 초기화 함수
  }), [user]);

  return (
    // // 5. Context.Provider를 통해 value를 하위 트리(children)에 주입
    // // .Provider : Context라는 저장소에 데이터를 Provide(제공) 하는 역할
    <UserContext.Provider value={value}>
      {children}
    </UserContext.Provider>
  );
};
```
### 3.2 앱 전체에 Provider 적용
- **범위 설정**: 데이터 공유가 필요한 최상위 컴포넌트를 Provider로 래핑.
```javascript
import { UserProvider } from './UserContext';

function App() {
  return (
    // // 6. 유저 관련 데이터가 필요한 모든 컴포넌트의 부모 단계에서 감싸기
    <UserProvider>
      <Header /> // // 내부에서 useContext(UserContext) 사용 가능
      <Main />   // // 내부에서 useContext(UserContext) 사용 가능
    </UserProvider>
  );
}
```

## 4. 성격 비교: 내장 도구 vs 외부 통신

|**구분**|**Context API**|**서버/Web API (REST 등)**|
|---|---|---|
|**정체**|**리액트 내장 기능** (도구 세트)|**외부 시스템**과의 통신 규칙|
|**목적**|앱 내부 **컴포넌트 간** 소통|**클라이언트-서버 간** 데이터 교환|

---
## 5. 최적화 및 주의사항
- **메모이제이션 필수 적용**
    - Provider의 `value` 전달 시 객체 리터럴 대신 `useMemo` 사용 권장.
- **Context 분리 (Splitting)**
    - 잦은 업데이트가 발생하는 데이터와 정적인 데이터를 별도 Context로 분리하여 영향 범위 최소화.
- **적정 기술 선택**
    - 단순 1~2단계 전달은 Props 사용 권장.
    - 복잡한 상태 로직이나 높은 업데이트 빈도 발생 시 전문 상태 관리 라이브러리(Zustand 등) 검토 필요.
