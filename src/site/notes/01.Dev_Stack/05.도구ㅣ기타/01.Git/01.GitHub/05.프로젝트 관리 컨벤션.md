# Git 커밋 메시지 및 프로젝트 관리 컨벤션
> [!summary] 요약: 협업 가독성 및 이력 추적 효율성 극대화를 위한 커밋 메시지 구조 표준화, 유형별 접두어(Prefix) 정의 및 작성 세부 규칙 가이드.
## 커밋 메시지 3단계 구조
- **제목 (Subject)**
    - 작업 내용의 핵심을 한 줄로 요약.
    - 접두어(Type)와 본문 내용 사이에 콜론(:) 및 공백 활용.
- **본문 (Body)**
    - 변경 이유 및 상세 내용 설명.
    - 제목만으로 설명이 충분할 경우 생략 가능.
- **바닥글 (Footer)**
    - 해결된 이슈 번호(Issue Tracker ID) 또는 관련 링크 기재.
    - 선택 사항으로 운용.
## 주요 접두어(Type) 표준
| **Type**     | **설명**                           |
| ------------ | -------------------------------- |
| **Feat**     | 새로운 기능 개발 및 추가                   |
| **Fix**      | 버그 및 오류 수정                       |
| **Docs**     | README, 주석 등 문서 생성 및 수정          |
| **Style**    | 코드 형식, 세미콜론 누락 등 (로직 변경 없음)      |
| **Refactor** | 코드 리팩토링 (기능 변경 없는 코드 성능 및 구조 개선) |
| **Test**     | 테스트 코드 작성 및 수정                   |
| **Chore**    | 빌드 설정, 패키지 매니저 설정 등 기타 부수적인 작업   |
## 작성 세부 원칙
- **간결성 유지**
    - 제목 끝 마침표(.) 사용 금지.
    - 영문의 경우 첫 글자 대문자 사용 및 명령문(Imperative) 형식 지향.
- **명확한 정보 전달**
    - "수정", "작업 완료" 등 추상적인 표현 배제.
    - 변경 대상과 목적이 드러나도록 구체적으로 기술.
- **변경 단위 분리**
    - 하나의 커밋에는 하나의 논리적 변경 사항만 포함.
    - 대규모 변경 시 기능별 소규모 커밋 분할 권장.
```xml
// 커밋 메시지 작성 예시
Feat: 사용자 로그인 유효성 검사 로직 추가
- 이메일 형식 체크 정규식 적용
- 비밀번호 8자리 이상 필수 조건 추가

Fix: DB 연결 타임아웃 오류 수정
- 커넥션 풀 설정 값 조정 (10s -> 30s)
// XML 주석 형식 활용
```
