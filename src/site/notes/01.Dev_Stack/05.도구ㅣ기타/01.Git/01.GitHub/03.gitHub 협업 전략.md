# 🐙 0단계: 깃허브(GitHub) 협업 전략 및 운영 가이드

> [!summary]
> 원활한 협업과 코드 품질 유지를 위해 팀원 간의 **브랜치 관리 모델, 커밋 메시지 규칙(Convention), 코드 리뷰 및 권한 관리** 프로세스를 정의합니다. 이 규칙은 코드 충돌을 최소화하고 히스토리 추적을 용이하게 합니다.

## 1. 브랜치 관리 전략 (Git Flow)

팀의 규모와 배포 주기에 따라 적절한 전략을 선택합니다. 가장 대중적인 **Git Flow**를 기준으로 설명합니다.

* **주요 브랜치(Main Branches):**
	- `main` (또는 `master`): 제품으로 출시될 수 있는 **최종 상태**의 코드.
	* `develop`: 다음 출시 버전을 위해 개발 중인 **통합 브랜치**.
* **보조 브랜치(Supporting Branches):**
	* `feature/{기능명}`: 새로운 기능 개발. 
		* `develop`에서 분기하여 완료 후 다시 `develop`으로 합침.
	* `release`: 배포 전 최종 버그 수정 및 준비 단계.
	* `hotfix`: 배포된 서비스(`main`)에서 긴급한 버그 발생 시 즉시 수정.

## 2. 단계별 권한 부여 방법 (Step-by-Step)

1. **팀원 초대:** `Add people` 버튼을 클릭합니다.
2. **ID 검색:** 초대할 팀원의 **GitHub ID** 또는 **이메일**을 입력합니다.
3. **역할 선택:** 검색된 팀원 옆의 드롭다운 메뉴에서 **권한 레벨(Role)**을 선택합니다.
    - 이때 위에서 언급한 `Admin`, `Maintain`, `Write`, `Read` 중 하나를 선택하게 됩니다.
4. **초대 발송:** `Add <ID> to this repository`를 클릭합니다.
5. **초대 수락:** 초대받은 팀원이 본인의 이메일이나 GitHub 알림에서 **[Accept Invitation]** 을 눌러야 최종적으로 권한이 부여됩니다.

| 권한 레벨        | 주요 권한                             | 대상             |
| ------------ | --------------------------------- | -------------- |
| **Admin**    | 모든 설정 변경, 레포지토리 삭제, 권한 관리         | 팀장, PL         |
| **Maintain** | 브랜치 보호 설정, 이슈/PR 관리 (설정 제외 모든 권한) | 시니어 개발자        |
| **Write**    | 코드 푸시(Push), 브랜치 생성 및 삭제          | 일반 개발자(참여 팀원)  |
| **Read**     | 코드 조회 및 이슈 제기 (수정 불가)             | 외부 관계자, 단순 참조자 |


> [!TIP]
> 1. 수정경로: `Repository 상단 메뉴` > `Settings` > `Access` > `Collaborators and teams`)
> 2. 프로젝트 종료 직후나 팀원 탈퇴 시 `Settings > Collaborators` 메뉴에서 즉시 권한을 삭제하여 소스 코드 유출을 방지해야 합니다.

## 3. 브랜치 보호 설정

#### 1. 설정 위치 및 진입 방법

이 설정은 저장소의 **Admin** 권한을 가진 사람만 수행할 수 있습니다.
1. GitHub 저장소 상단의 **`Settings`** 탭을 클릭합니다.
2. 왼쪽 사이드바 메뉴에서 **`Code and automation`** 섹션의 **`Branches`**를 선택합니다.
3. **`Branch protection rules`** 옆의 **`Add branch protection rule`** 버튼을 클릭합니다.
    

#### 2. 핵심 보호 규칙 설정 (Recommended)

가장 많이 쓰이고 강력한 4가지 핵심 옵션을 추천해 드립니다.

##### ① Branch name pattern

보호할 브랜치의 이름을 입력합니다. (예: `main` 또는 `develop`)

##### ② Require a pull request before merging(**추천**)

이 옵션을 켜면 `main`으로의 직접적인 `push`가 금지되며, 반드시 **PR을 통해서만** 코드를 합칠 수 있습니다.
- **Required approvals:** 코드를 합치기 위해 필요한 최소 승인 인원수를 정합니다. (보통 1명 이상 추천)
- **Dismiss stale pull request approvals...:** 새로운 코드가 `push`되면 기존에 받은 승인을 무효화하여 다시 검토받게 합니다.
##### ③ Require status checks to pass before merging
테스트 자동화(CI/CD)를 구축했다면, 모든 테스트가 통과(Success)했을 때만 머지가 가능하도록 제한합니다.

##### ④ Lock branch
브랜치를 읽기 전용으로 만들어 아무도 수정할 수 없게 합니다. (프로젝트 종료 시 사용)

#### 3. 권한 및 강제성 설정
- **Restrict who can push to matching branches:** 특정 인원이나 팀만 이 브랜치에 코드를 올릴 수 있게 제한합니다.
- **Include administrators:** 이 옵션을 켜면 **관리자(Admin)조차도** 위 규칙을 지켜야 합니다. (관리자의 실수를 막기 위해 체크하는 것을 권장합니다.)

## 4. 커밋 및 머지 규칙 (Convention)

### 4.1.커밋 메시지 컨벤션 (Commit Message)

메시지의 서두에 작업 성격을 나타내는 **Type**을 명시합니다.

* `feat`: 새로운 기능 추가
* `fix`: 버그 수정
* `docs`: 문서 수정 (README 등)
* `style`: 코드 포맷 변경, 세미콜론 누락 등 (로직 변경 없음)
* `refactor`: 코드 리팩토링
* `test`: 테스트 코드 추가/수정

> 기능 구현을 바꾼 부분이나 설정 파일을 커밋 할때는 적어주어야한다.

### 4.2.머지(Merge) 프로세스

단순 Merge 대신 **PR(Pull Request)** 방식을 권장합니다.

1. `feature` 브랜치 작업 완료 후 `Push`.
2. GitHub에서 `develop` 브랜치로 **Pull Request** 생성.
3. 최소 1인 이상의 팀원에게 **Code Review** 승인(Approve)을 받음.
4. 승인 후 `Merge` 실행 (필요 시 `Squash and merge`로 커밋 내역 합치기).

## 💡 예시

- 단계별 통합 프로세스 (Main Merge Workflow)

각 팀원이 자신의 브랜치(`hwan`, `sun` 등)에서 작업을 완료했을 때 수행할 표준 절차입니다.
#### ① 로컬 동기화 (Sync before PR)

내 브랜치를 `main`에 합치기 전, **다른 팀원이 이미 `main`에 올린 코드**와 내 코드가 충돌하지 않는지 먼저 확인해야 합니다.
1. 로컬에서 `main` 브랜치로 이동 후 최신 코드를 `pull` 받습니다.
2. 다시 내 브랜치(`hwan` 등)로 돌아와 `main`의 내용을 내 브랜치로 `merge`하여 충돌 여부를 확인합니다.
#### ② Pull Request(PR) 생성

- 프로젝트 저장소 페이지에 접속하면 **`Compare & pull request`** 노란색 버튼이 나타납니다.(안 보인다면 `Pull requests` 탭 > `New pull request` 클릭)
- **내용 작성:**
    - **base:** `main` (합쳐질 곳) ← **compare:** `hwan` (내가 작업한 곳) 확인.
    - **Title:** `[feat] 입양 로직 수정` 등 핵심 요약.
    - **Description:** 어떤 파일을 수정했는지, 테스트는 해봤는지 간단히 기록.
- **Reviewers 지정:** 오른쪽 메뉴에서 코드를 봐줄 팀원을 선택합니다.
- **Create pull request:** 버튼을 누르면 요청이 완료됩니다.

#### ③ 코드 리뷰 및 승인

팀원들이 코드를 확인하고 `Approve`를 누를 때까지 기다립니다. 이 과정에서 수정 사항이 생기면 내 브랜치에서 수정 후 다시 `push`하면 PR에 자동 반영됩니다.
#### ④ Squash and Merge 실행 (핵심)

승인이 완료되면 `Merge` 버튼 옆의 화살표를 눌러 **[Squash and merge]**를 선택합니다.

- **이유:** 이미지 속 `chan`, `dong` 브랜치의 수많은 자잘한 커밋들(예: "오타 수정", "제거")을 `main`에 다 남기지 않고, **"봉사 후기 기능 구현 완료"라는 하나의 깔끔한 커밋**으로 합쳐서 `main`에 기록하기 위함입니다.
