# 신규 환경 프로젝트 배포 및 실행 가이드
> [!summary] 핵심 요약
> 
> - **개념**: Docker 기반 프로젝트의 신규 PC 환경 구축 및 실행 프로세스
>     
> - **왜?**: 개발 환경의 파편화를 방지하고, 컨테이너 기술을 통해 동일한 실행 상태 보장
>     
> - **언제?**: PC 교체, 신규 팀원 합류, 배포 서버 초기 설정 시
>     
> - **핵심**: ==WSL2 기반의 Docker 환경 세팅==, `.env` 파일 수동 복구, `docker-compose` 일괄 빌드
>     
## 사전 준비 및 도구 설치 (WSL2 & Docker)
- **WSL2 (Windows Subsystem for Linux) 설치**
    - Windows 환경에서 Docker 성능 극대화를 위한 필수 리눅스 서브시스템
    - PowerShell 관리자 권한으로 `wsl --install` 실행 후 PC 재부팅 필수
- **Docker Desktop 설치 및 WSL2 연동**
    - 최신 버전 설치 시작 시 ==Use WSL 2 instead of Hyper-V== 옵션이 기본적으로 체크되어 나옴
    - 해당 옵션을 체크된 상태로 유지하고 설치하면 Docker가 최적의 WSL 설정을 자동으로 구성함
    - 설치 완료 후 설정(Settings > General)에서 해당 옵션 활성화 상태 재확인 권장
- **필수 도구 검증**
    - Git: 소스 제어를 위한 클라이언트 설치
    - 상태 확인: 명령어 입력 전, Docker Desktop 앱 정상 실행 상태(고래 아이콘 녹색) 확인
```plaintext
[ Environment Setup Flow ]

      [ WSL2 ]               [ Docker ]               [ Git ]
         │                       │                       │
         ▼                       ▼                       ▼
  ┌──────────────┐        ┌──────────────┐        ┌──────────────┐
  │ Install WSL  │        │ Install App  │        │  Git Clone   │  <-- 소스 복구
  └──────┬───────┘        └──────┬───────┘        └──────┬───────┘
         │                       │                       │
         ▼                       ▼                       ▼
  ┌──────────────┐        ┌──────────────┐        ┌──────────────┐
  │  Reboot PC   │        │ Check WSL 2  │        │  Project CD  │  <-- 폴더 이동
  └──────────────┘        └──────────────┘        └──────────────┘
```
## 프로젝트 복제 및 초기화 점검
- **코드 체크아웃 및 권한 확인**
    - 마운트 권한 오류 방지를 위해 터미널(PowerShell/CMD) ==관리자 권한 실행== 권장
    - 저장소로부터 프로젝트 전체 데이터를 로컬로 복제
- **SQL 초기화 파일 점검**
    - 복제된 폴더 내 `FinalProject/sql` 구문 디렉토리 확인
    - 테이블 생성용 `.sql` 파일 존재 여부 반드시 체크 (해당 파일 누락 시 DB가 빈 상태로 기동됨)
- **어떻게?**: 터미널을 통한 환경 확인 및 복제 실행
```bash
# 1. 도커 프로세스 실행 확인 (데스크톱 앱 실행 필수)
docker -v

# 2. 프로젝트 소스 코드 복제
git clone [레포지토리_주소]

# 3. 프로젝트 루트 디렉토리(docker-compose.yml 존재 위치)로 이동
cd 2026-bootcamp_final_project
```
## 환경 변수 설정 및 컨테이너 실행
- **환경 변수(.env) 복구**
    - 보안상 `.gitignore`에 의해 제외된 설정 파일을 수동으로 생성
    - `.env.example` 파일을 참조하여 실제 인증키(JWT_SECRET 등) 기입
- **컨테이너 일괄 빌드 및 구동**
    - 정의된 서비스(DB, API, Web)를 한꺼번에 빌드하고 백그라운드(`-d`)에서 실행
    - 최초 실행 시 대용량 베이스 이미지(Oracle DB, Maven, Node.js 등) 다운로드 및 컴파일로 인하여 약 5~15분 소요
    - Docker Desktop 버전에 따라 `docker-compose` 대신 하이픈이 없는 ==`docker compose`== 명령어 사용 필요
```plaintext
[ Variable Setup Flow ]

    .env.example            .env (Actual)
  ┌──────────────┐        ┌──────────────┐
  │  Templates   │        │  Real Values │  <-- 값 수동 입력
  └──────┬───────┘        └──────┬───────┘
         │      (Copy)           │
         └───────────────────────┘
                                 │
  [Warning] Do Not Upload ───────┘  <-- Git 업로드 금지
```
- **어떻게?**: Docker Compose 명령어를 통한 서비스 구동 및 트러블슈팅
```bash
# 이미지 빌드 및 컨테이너 백그라운드 실행
docker-compose up --build -d
# (명령어 인식 실패 시: docker compose up --build -d)

# [Tip] 설정 꼬임 또는 깨끗한 상태로 재기동 필요 시 (컨테이너 및 DB 볼륨 데이터 삭제)
docker-compose down -v
docker-compose up --build -d
```
## 서비스 정상 작동 검증
- **상태 확인**
    - 모든 컨테이너가 Up 상태인지 확인 후 브라우저 및 DB 클라이언트를 통해 접속
    - Nginx 프록시(`/spring` 경로 전달) 설정에 의해 프론트엔드와 백엔드 모두 8080 포트로 통합 접속 가능
- **검증 대상 리스트**

|**검증 대상**|**확인 방법 및 주소**|**비고**|
|---|---|---|
|프론트엔드|`http://localhost:8080`|사용자 UI 작동 확인|
|백엔드 (API)|`http://localhost:8080/spring/swagger-ui/index.html`|8080 프록시를 통해 API 문서 접속|
|데이터베이스|`localhost:1523` 접속|계정: EASYEARTH (SID: xe, ServiceName: XEPDB1)|
