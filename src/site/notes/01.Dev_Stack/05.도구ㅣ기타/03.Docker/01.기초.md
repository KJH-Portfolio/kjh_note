# 현대 개발의 필수 도구: Docker 입문 가이드
> [!summary] 핵심 요약
> 
> - **개념**: 애플리케이션을 구동 환경에 구애받지 않고 실행할 수 있게 돕는 컨테이너 기반 가상화 플랫폼
>     
> - **왜?**: "내 컴퓨터에서는 정상 동작하는데 서버에서는 왜 안 되지?"라는 환경 불일치 문제의 근본적 해결
>     
> - **언제?**: 로컬 개발 환경 통일, 클라우드 서버 배포, CI/CD 자동화 파이프라인 구축 시
>     
> - **핵심**: 정적인 '이미지(설계도)'와 동적인 '컨테이너(실행 환경)'의 개념 분리 및 필수 명령어 숙달
>     
## 1. 도커(Docker)의 핵심 개념
- **컨테이너 가상화**
    - OS 전체를 가상화하는 무거운 VM(Virtual Machine)과 달리, ==프로세스 격리== 방식을 채택함.
    - 호스트 OS의 커널을 공유하여 시스템 자원 소모가 적고 부팅 속도가 매우 빠름.
- **배송용 컨테이너 비유**
    - 내부 내용물(Java, Python, DB 등)에 상관없이 규격화된 도커 컨테이너에 담아 어떤 인프라에서든 동일하게 실행함.
```plaintext
[ 가상화 방식 비교 (Virtualization Comparison) ]

 [VM (Virtual Machine)]                  [Docker (Container)]
 ┌────────────────────┐                  ┌────────────────────┐
 │   App 1  │  App 2  │                  │   App 1  │  App 2  │
 ├──────────┼─────────┤                  ├──────────┴─────────┤
 │ Guest OS │Guest OS │                  │   Docker Engine    │
 ├──────────┴─────────┤                  ├────────────────────┤
 │     Hypervisor     │                  │   Host OS Kernel   │
 ├────────────────────┤                  ├────────────────────┤
 │   Infrastructure   │                  │   Infrastructure   │
 └────────────────────┘                  └────────────────────┘
 * OS 전체 가상화 (무겁고 느림)           * 프로세스 격리 (가볍고 빠름)
```
## 2. 도커 구성 3요소 (Image, Container, Registry)
- **도커 이미지 (Image)**
    - 서비스 구동에 필요한 프로그램, 라이브러리, 소스 코드를 포함한 ==읽기 전용(Read-only) 설계도==.
- **도커 컨테이너 (Container)**
    - 이미지를 기반으로 실행된 상태이자 독립된 공간에서 동작하는 애플리케이션 ==실행 인스턴스==.
- **도커 레지스트리 (Registry)**
    - 생성된 이미지를 저장하고 공유하는 원격 저장소 (예: Docker Hub, GitHub Container Registry).

|**구분**|**성격**|**비유**|**특징**|
|---|---|---|---|
|**이미지 (Image)**|정적 (Static)|붕어빵 틀 / 설계도|변경 불가능, 컨테이너 실행의 뼈대|
|**컨테이너 (Container)**|동적 (Dynamic)|실제 붕어빵 / 제품|독립된 실행 환경, 독자적인 상태 가짐 (생성/삭제 가능)|
|**레지스트리 (Registry)**|저장소 (Repository)|앱스토어 / 물류 창고|`push`(업로드) 및 `pull`(다운로드) 수행|

## 3. 입문자를 위한 필수 명령어
- **컨테이너 라이프사이클 관리**
    - 도커 환경에서 가장 빈번하게 사용하는 기본 조작 및 관리 명령어 모음.

|**명령어**|**동작**|**핵심 주의사항**|
|---|---|---|
|`docker pull {이미지}`|레지스트리에서 로컬로 이미지 다운로드|버전(태그) 생략 시 `latest` 버전 다운로드|
|`docker run {이미지}`|==새로운 컨테이너 생성 및 즉시 실행==|로컬에 이미지가 없으면 레지스트리에서 자동 pull 후 실행함|
|`docker ps`|실행 중인 컨테이너 목록 확인|종료된 컨테이너까지 모두 보려면 `-a` 옵션 사용|
|`docker stop {ID\|이름}`|실행 중인 특정 컨테이너 중지|강제 종료가 아닌 안전한 종료(SIGTERM) 요청|
|`docker rm {ID\|이름}`|중지된 컨테이너 완전 삭제|==실행 중인 컨테이너는 삭제 불가== (먼저 stop 해야 함)|
|`docker images`|로컬 시스템에 저장된 이미지 목록 확인|불필요한 이미지가 쌓이면 디스크 용량을 크게 차지함|
|`docker rmi {이미지}`|로컬 이미지 삭제|해당 이미지를 참조하여 실행된 컨테이너가 없어야 삭제 가능|

## 4. 도커 표준 워크플로우
- **코드에서 배포까지의 흐름**
    - 개발한 애플리케이션을 도커 이미지로 패키징하고 서버에 배포하는 일련의 표준 파이프라인 과정.
    1. **Dockerfile 작성**: 애플리케이션 설치 및 설정 과정을 명세서(텍스트) 형태로 기록함.
    2. **Image Build**: 작성된 Dockerfile을 실행하여 실행 가능한 이미지를 생성함.
    3. **Image Push**: 생성된 로컬 이미지를 Docker Hub 등 외부 레지스트리에 저장함.
    4. **Container Run**: 배포할 서버에서 저장된 이미지를 내려받아 즉시 구동함.
```plaintext
[ 도커 배포 파이프라인 (Docker Workflow) ]

 [1. Code]         [2. Build]            [3. Ship]              [4. Run]
 Dockerfile ──────▶ Docker Image ───────▶ Docker Registry ───────▶ Docker Container
 (환경 설정 명세)    (패키징된 이미지)       (중앙 이미지 저장소)      (운영 서버 인스턴스)
      │                 │                     │                      │
      └── docker build ─┘                     │                      │
                        └─── docker push ─────┘                      │
                                              └── docker pull/run ───┘
```
