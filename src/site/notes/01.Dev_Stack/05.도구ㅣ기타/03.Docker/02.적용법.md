# Spring Boot + React + Oracle DB 프로젝트 도커화 가이드
> [!summary] 핵심 요약
> 
> - **개념**: 백엔드(Spring), 프론트엔드(React), 데이터베이스(Oracle)를 독립적인 컨테이너로 패키징하고 단일 네트워크로 묶어 배포하는 아키텍처
>     
> - **왜?**: 로컬/서버 간 환경 불일치 문제를 해결하고, 명령어 한 줄로 전체 시스템 구동을 자동화하기 위함
>     
> - **언제?**: 개발 완료 후 운영 서버 배포 준비 단계 또는 팀원 간 동일한 개발 환경 세팅 시
>     
> - **핵심**: 소스코드 내 하드코딩된 변수 제거, 멀티스테이지 빌드를 통한 이미지 최적화, Nginx 리버스 프록시 구성
>     
## 1. 아키텍처 및 디렉토리 구조
- **목표 디렉토리 구조 설계**
    - 기존 소스 코드를 유지하면서 도커 관련 설정 파일(`Dockerfile`, `docker-compose.yml`, `nginx.conf`)을 각 계층의 최상단에 배치
    - 민감한 정보는 외부에 노출되지 않도록 최상단 `.env` 파일로 통합 분리
    - **어떻게?**: 전체 프로젝트를 아우르는 루트 디렉토리 및 하위 폴더별 설정 파일 추가 구성
```plaintext
[ 최종 파일 구조 목표 (File Structure) ]

2026-bootcamp_final_project/
 ├─ docker-compose.yml        (전체 오케스트레이션 / 신규)

 ├─ .env.example              (키 이름 템플릿 / 신규)

     │   └─ src/.../application.properties (환경변수 참조 / 수정)

```
## 2. 백엔드 환경 구성 (Spring Boot)
- **Dockerfile 설계**
    - 소스 코드 컴파일과 최종 실행 환경을 분리하는 ==2단계 멀티스테이지 빌드== 적용
    - 최종 컨테이너에는 무거운 JDK(개발 도구)를 빼고 JRE(실행 환경)만 포함하여 이미지 경량화
    - **어떻게?**: `easyearth-backend/restfulProject/Dockerfile` 생성 및 작성
```dockerfile
# 1단계: 빌드 환경 (Maven + Java 17)

#컨테이너의 기초 운영체제 설정, 컨테이너 내부 작업 폴더 이름 설정
FROM maven:3.9.6-eclipse-temurin-17-alpine AS build
WORKDIR /app

# pom.xml 복사, 오프라인 모드, 배치모드
COPY pom.xml .
RUN mvn dependency:go-offline -B

# 소스 코드 복사후 빌드
COPY src ./src
RUN mvn clean package -DskipTests -B

# 2단계: 실행 환경 (JRE만 포함된 경량 이미지)

# 실행 운영체제 설정, 컨테이너 내부 작업 폴더 이름 설정
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app

# 1단계 결과물 파일 가져오기
COPY --from=build /app/target/*.war app.war
RUN mkdir -p /root/uploadFiles

# 환경 변수 설정
ENV LANG=ko_KR.UTF-8 \
    TZ=Asia/Seoul

# 포트 설정
EXPOSE 8080
# JRE, 자바 실행파일, 프로젝트 백엔드 파일 실행
ENTRYPOINT ["java", "-jar", "/app/app.war"]
```
### 💡 코드 상세 설명: 백엔드 Dockerfile
**[ 1단계: 빌드 환경 ]**
- `FROM maven:3.9.6-eclipse-temurin-17-alpine AS build`
    - `FROM`: 컨테이너의 기초가 되는 운영체제(베이스 이미지)를 지정합니다.
    - `maven:3.9.6`: 빌드 도구인 Maven이 설치되어 있음을 의미합니다.
    - `eclipse-temurin-17`: Java 17 환경을 제공합니다. (Temurin은 Eclipse 재단에서 만든 안정적인 오픈소스 Java 배포판입니다.)
    - `alpine`: 불필요한 기능을 모두 뺀 초경량 리눅스 운영체제(Alpine Linux)를 사용하여 이미지 용량을 크게 줄입니다.
    - `AS build`: 이 첫 번째 단계의 이름을 'build'로 명명하여, 나중에 2단계에서 결과물을 가져올 때 참조할 수 있게 합니다.
- `WORKDIR /app`
    - 컨테이너 내부의 작업 폴더를 `/app`으로 설정합니다. 이후 실행되는 명령어는 모두 이 폴더 안에서 이루어집니다.
- `COPY pom.xml .` / `RUN mvn dependency:go-offline -B`
    - 소스 코드 전체를 복사하기 전에 `pom.xml`만 먼저 복사합니다. 코드가 수정되더라도 라이브러리 목록(`pom.xml`)이 바뀌지 않았다면, 이전 빌드에서 다운로드한 라이브러리를 재사용(캐싱)하여 빌드 속도를 획기적으로 높이기 위함입니다. `go-offline`은 라이브러리를 미리 다운로드하는 명령입니다.
- `COPY src ./src` / `RUN mvn clean package -DskipTests -B`
    - 실제 소스 코드를 복사하고 압축(`.war` 또는 `.jar`)하여 빌드합니다. 빠른 빌드를 위해 `-DskipTests`로 테스트 코드 실행을 건너뜁니다.

**[ 2단계: 실행 환경 ]**
- `FROM eclipse-temurin:17-jre-alpine`
    - 1단계와 달리 `maven`과 `jdk`(개발 도구)가 빠진 `jre`(실행 전용 환경) 이미지를 사용합니다. 서버를 돌리는 데는 개발 도구가 필요 없으므로, 이렇게 하면 최종 컨테이너 용량이 대폭 줄어듭니다.
- `COPY --from=build /app/target/*.war app.war`
    - 1단계(`build`)에서 완성된 결과물 파일만 현재 컨테이너로 쏙 빼옵니다.
- `ENV LANG=ko_KR.UTF-8 \ TZ=Asia/Seoul`
    - `ENV`는 환경 변수를 설정합니다. 시스템 언어를 한국어로 설정해 한글 깨짐을 방지하고, 타임존(`TZ`)을 한국 시간으로 맞춰 DB나 로그의 시간 기록이 엇나가지 않게 합니다.
- `EXPOSE 8080`
    - 이 컨테이너가 내부적으로 8080 포트를 사용할 것임을 명시합니다. (실제 방화벽을 여는 역할은 아니며 문서화 목적이 큽니다.)
- `ENTRYPOINT ["java", "-jar", "/app/app.war"]`
    - 컨테이너가 켜질 때 백엔드 서버를 띄우기 위해 항상 실행되는 핵심 명령어입니다.

**[ 설정 파일(Properties) 수정 ]**
- 소스 코드 내부에 존재하던 DB 접속 정보 및 API 키를 환경변수(`${...}`) 참조 방식으로 변경
- **어떻게?**: `src/main/resources/application.properties` 수정
```properties
spring.datasource.url=${SPRING_DATASOURCE_URL:jdbc:oracle:thin:@db:1521:xe}
spring.datasource.username=${SPRING_DATASOURCE_USERNAME:EASYEARTH}
spring.datasource.password=${SPRING_DATASOURCE_PASSWORD:EASYEARTH}

jwt.secret=${JWT_SECRET}
kakao.client-id=${KAKAO_CLIENT_ID}
kakao.client-secret=${KAKAO_CLIENT_SECRET}
gemini.api.key=${GEMINI_API_KEY}
```
### 💡 코드 상세 설명: 환경변수 설정 (`application.properties`)
- `${외부변수명:기본값}` 구조
    - 이 문법은 외부(도커 환경)에서 해당 이름의 변수가 주입되면 그 값을 사용하고, 주입되지 않으면 콜론(`:`) 뒤에 적힌 '기본값'을 대신 사용하라는 뜻입니다.
    - 예를 들어 로컬 개발 환경에서는 `application.properties`에 적힌 기본값으로 작동하고, 운영 환경(도커)에서는 보안상 중요한 비밀번호 등을 `.env` 파일을 통해 외부에서 주입받아 작동하게 하는 핵심 원리입니다.
> [!warning] 호스트명 일치 주의
> 
> JDBC URL의 `@db:1521` 부분에서 `db`는 반드시 향후 작성할 `docker-compose.yml` 내의 ==데이터베이스 서비스 이름과 완벽히 일치==해야 함.
## 3. 프론트엔드 환경 구성 (React + Nginx)
- **Dockerfile 설계**
    - Node.js 환경에서 정적 파일로 변환(빌드) 후, 가벼운 웹 서버인 Nginx를 통해 브라우저에 제공
    - **어떻게?**: `easyearth-frontend/my-react-app/Dockerfile` 생성 및 작성
```dockerfile
# 1단계: 빌드 환경 (Node.js)

# Node.js가 포함된 가상 컴퓨터 준비
FROM node:20-alpine AS build
WORKDIR /app

# 의존성 파일 복사, 설치
COPY package*.json ./
RUN npm ci

# 코드 전부 복사, 서비스용으로 빌드
COPY . .
RUN npm run build

# 2단계: 실행 환경 (Nginx)
# 웹서버 번용 가상 컴퓨터 Nginx 준비
FROM nginx:stable-alpine

# 1단계에서 완성된 폴더를 Nginx에서 서비스 할 수 있게 복사
COPY --from=build /app/dist /usr/share/nginx/html
# 설정 파일 복사
COPY nginx.conf /etc/nginx/conf.d/default.conf

# 80번 포트 사용
EXPOSE 80
# Nignx 실행, 전역에서 컨테이너 계속 유지하도록 daemon off
CMD ["nginx", "-g", "daemon off;"]
```
### 💡 코드 상세 설명: 프론트엔드 Dockerfile
**[ 1단계: 빌드 환경 ]**
- `FROM node:20-alpine AS build`
    - React 앱을 빌드하기 위해 Node.js 20버전이 설치된 경량(alpine) 이미지를 기반으로 1단계(`build`)를 시작합니다.
- `COPY package*.json ./` / `RUN npm ci`
    - 백엔드와 마찬가지로 의존성 파일(`package.json` 등)을 먼저 복사하여 캐싱을 활용합니다. `npm ci`는 `package-lock.json`에 명시된 버전을 기준으로 빠르고 정확하게 라이브러리를 설치하는 명령어입니다.
- `COPY . .` / `RUN npm run build`
    - React 소스 코드를 브라우저가 바로 읽을 수 있는 최적화된 정적 파일(HTML, CSS, JS)로 변환(빌드)합니다.

**[ 2단계: 실행 환경 ]**
- `FROM nginx:stable-alpine`
    - Node.js는 프론트엔드를 '빌드'할 때만 필요합니다. 실제 서비스할 때는 가볍고 빠른 웹 서버인 `Nginx` 이미지를 사용합니다.
- `COPY --from=build /app/dist /usr/share/nginx/html`
    - 1단계에서 완성된 정적 파일 폴더(`/dist`)를 Nginx가 기본적으로 화면에 띄워주는 경로(`/usr/share/nginx/html`)로 복사합니다.
- `COPY nginx.conf /etc/nginx/conf.d/default.conf`
    - 아래에서 별도로 작성할 Nginx 설정 파일(`nginx.conf`)을 컨테이너 내부의 기본 설정 파일 위치에 덮어씁니다.
- `CMD ["nginx", "-g", "daemon off;"]`
    - `CMD`는 컨테이너 실행 시 작동할 기본 명령입니다. Nginx는 기본적으로 백그라운드(보이지 않게)에서 돌려고 하는데, 컨테이너는 실행 중인 포그라운드 프로세스가 없으면 꺼져버립니다. `-g daemon off;` 옵션은 Nginx를 강제로 포그라운드에 묶어두어 컨테이너가 계속 살아있게 만드는 역할을 합니다.
> [!info] 빌드 디렉토리 확인
> 
> 사용 중인 프론트엔드 도구에 따라 빌드 결과물 경로가 다름. Vite는 `/dist`, Create React App(CRA)은 `/build`에 생성되므로 본인 프로젝트에 맞게 `COPY --from=build` 경로를 조정해야 함.

**[ Nginx 라우팅 및 프록시 설정 ]**
- React의 특징(SPA)을 지원하기 위한 주소 처리 및 백엔드와의 통신(CORS 오류 방지)을 위한 ==리버스 프록시 설정==
- **어떻게?**: `Dockerfile`과 같은 경로에 `nginx.conf` 생성 및 작성
```nginx
server {
    listen 80;                         # 80번 포트(기본 웹 포트)로 들어오는 요청을 듣겠다.
    server_name localhost;             # 서버의 이름은 localhost로 설정한다.
    root /usr/share/nginx/html;        # 리액트 빌드 파일(index.html 등)이 들어있는 실제 폴더 경로.
    index index.html;                  # 주소만 치고 들어왔을 때 가장 먼저 보여줄 기본 파일.

    # 1. SPA(Single Page Application) 라우팅 지원
    # 리액트는 페이지가 하나뿐이라서, 주소를 옮겨 다닐 때 Nginx가 당황하지 않게 돕는 설정입니다.
    location / {
        # 사용자가 요청한 파일($uri)이 있으면 주고, 없으면 폴더($uri/)를 찾고, 
        # 그것도 없으면 무조건 index.html을 보여줘라! (리액트가 알아서 처리하게 넘기는 것)
        try_files $uri $uri/ /index.html;
    }

    # 2. 백엔드 API 리버스 프록시 (중간 다리 역할)
    # 주소창에 "/spring"이 포함된 요청이 오면 백엔드 컨테이너로 토스합니다.
    location /spring {
        # 요청을 'backend'라는 이름의 컨테이너 8080 포트로 전달해라.
        proxy_pass http://backend:8080;
        
        # 원래 요청을 보낸 사람의 호스트 정보를 백엔드에 전달.
        proxy_set_header Host $host;
        # 원래 요청을 보낸 사람의 실제 IP 주소를 백엔드에 전달 (로그 확인용).
        proxy_set_header X-Real-IP $remote_addr;

        # 웹소켓(채팅 등) 통신을 위한 설정 시작
        proxy_http_version 1.1;
        # 통신 방식을 일반 HTTP에서 웹소켓으로 '업그레이드' 하겠다는 신호 전달.
        proxy_set_header Upgrade $http_upgrade;
        # 연결 상태를 'Upgrade'로 유지해라.
        proxy_set_header Connection "upgrade";
        
        # 서버 응답을 기다려주는 시간 (5분 동안 반응 없으면 끊음).
        proxy_read_timeout 300s;
    }

    # 사용자가 업로드하는 파일의 최대 크기를 10MB로 제한 (이미지 업로드 등 고려).
    client_max_body_size 10M;
    
    # 404 에러(페이지 없음)가 나면 리액트 메인 페이지(index.html)로 보내버려라.
    error_page 404 /index.html;
}
```
### 💡 코드 상세 설명: Nginx 설정 (`nginx.conf`)
- `try_files $uri $uri/ /index.html;` (SPA 라우팅 지원)
    - React는 페이지가 하나인(Single Page Application) 구조입니다. 사용자가 `도메인.com/about` 같이 특정 주소로 새로고침을 하면, Nginx는 실제 `/about`이라는 폴더나 파일이 없으므로 404 에러를 냅니다.
    - `try_files` 설정은 "요청받은 파일(`$uri`)이 없으면 무조건 `index.html`을 띄워라"라는 뜻입니다. 그러면 React가 화면에 로드된 후 내부적으로 주소를 판단해 올바른 화면을 그려냅니다.
- `location /spring` (리버스 프록시)
    - 주소가 `/spring`으로 시작하는 API 요청이 오면, 프론트엔드(Nginx)가 직접 처리하지 않고 백엔드 컨테이너(`backend:8080`)로 조용히 패스(전달)합니다.
    - **왜 이렇게 하나요?**: 프론트엔드 서버와 백엔드 서버의 주소(포트)가 다르면 브라우저는 보안상 이유로 통신을 차단(CORS 에러)합니다. 프록시를 설정하면 브라우저 입장에서는 프론트엔드 서버 한 곳하고만 통신하는 것처럼 보이게 되어 이 에러를 원천 차단할 수 있습니다.
- `proxy_set_header Upgrade $http_upgrade;` / `Connection "upgrade";`
    - 채팅 등 실시간 통신(웹소켓)을 할 때 서버와의 연결이 끊기지 않도록 통신 방식을 '업그레이드' 상태로 유지해주는 필수 설정입니다.
## 4. 오케스트레이션 및 컨테이너 실행 (Docker Compose)
- **Docker Compose 구성**
    - 3개의 독립된 프로그램(DB, Backend, Frontend)을 한 번에 실행하고 관리하는 통합 설계도 작성
    - 의존성(`depends_on`)을 부여하여 DB가 완전히 켜진 후 Backend가 실행되도록 안전장치 마련
    - **어떻게?**: 최상단 루트 디렉토리에 `docker-compose.yml` 작성
```yaml
services:                # 관리할 서비스(컨테이너)들의 목록을 정의합니다.

  # 1. 데이터베이스 서비스 (Oracle XE)
  db:
    image: gvenzl/oracle-xe:21-slim      # 오라클 공식 이미지를 가져와서 사용합니다.
    container_name: easyearth_db         # 컨테이너의 이름을 'easyearth_db'로 고정합니다.
    environment:                         # DB 접속을 위한 환경 변수(비밀번호 등) 설정입니다.
      - ORACLE_PASSWORD=EASYEARTH        # 시스템(관리자) 비밀번호
      - APP_USER=EASYEARTH               # 우리가 사용할 일반 사용자 이름
      - APP_USER_PASSWORD=EASYEARTH      # 사용자 비밀번호
    ports:
      - "1523:1521"                      # 내 컴퓨터의 1523번 포트와 컨테이너의 1521번을 연결합니다.
    volumes:                             # 데이터 보존을 위한 설정입니다.
      - oracle_data:/opt/oracle/oradata  # 컨테이너를 지워도 DB 데이터는 유지되도록 저장소 연결.
      - ./FinalProject/init_db.sql:/docker-entrypoint-initdb.d/init.sql # 초기 테이블 생성용 SQL 실행.
    healthcheck:                         # DB가 실제로 "사용 가능한 상태"인지 검사하는 기능입니다.
      test: ["CMD", "healthcheck.sh"]    # 오라클의 헬스체크 스크립트를 실행해 봅니다.
      interval: 30s                      # 30초마다 검사합니다.
      timeout: 20s                       # 응답이 20초 넘게 없으면 실패로 간주합니다.
      retries: 10                        # 10번 연속 실패하면 문제가 있다고 판단합니다.
    networks:
      - easyearth_net                    # 같은 네트워크망에 소속시켜 서로 통신하게 합니다.

  # 2. 백엔드 서비스 (Spring Boot)
  backend:
    build:                               # 이미지를 새로 만듭니다(빌드).
      context: ./FinalProject/easyearth-backend/restfulProject # 소스 코드가 있는 경로입니다.
      dockerfile: Dockerfile             # 사용할 설계도(Dockerfile) 파일명입니다.
    container_name: easyearth_backend    # 컨테이너의 이름입니다.
    env_file:                            # 민감한 정보(API 키 등)가 담긴 .env 파일을 불러옵니다.
      - .env
    ports:
      - "8081:8080"                      # 내 컴퓨터 8081포트로 백엔드에 접속할 수 있게 연결합니다.
    depends_on:                          # 실행 순서를 정합니다.
      db:
        condition: service_healthy       # DB가 완전히 켜져서 "정상(healthy)"일 때만 백엔드를 켭니다.
    volumes:
      - upload_files:/root/uploadFiles   # 사용자가 업로드한 이미지 파일이 안 날아가게 저장소 연결.
    networks:
      - easyearth_net

  # 3. 프론트엔드 서비스 (React + Nginx)
  frontend:
    build:
      context: ./FinalProject/easyearth-frontend/my-react-app # 리액트 소스 코드 경로입니다.
      dockerfile: Dockerfile
    container_name: easyearth_frontend
    ports:
      - "80:80"                          # 브라우저에서 localhost(80번)로 접속하면 리액트를 보여줍니다.
    depends_on:
      - backend                          # 백엔드가 먼저 켜진 후에 프론트엔드를 실행합니다.
    networks:
      - easyearth_net

# 영구 저장 장치(가상 하드디스크)들을 선언합니다.
volumes:
  oracle_data:                           # DB 데이터용 하드
  upload_files:                          # 업로드 이미지용 하드

# 컨테이너들이 서로를 이름으로 부를 수 있게 가상 네트워크 망을 만듭니다.
networks:
  easyearth_net:
    driver: bridge                       # 기본 방식인 '브릿지' 네트워크를 사용합니다.
```
### 💡 코드 상세 설명: Docker Compose
- `services:`
    - 도커 컴포즈가 함께 관리할 컨테이너들의 묶음을 정의합니다. (db, backend, frontend 3개)
- `environment:` (db)
    - 오라클 데이터베이스 컨테이너가 처음 생성될 때 사용할 최고 관리자 계정, 그리고 우리가 애플리케이션에서 사용할 전용 사용자 계정과 비밀번호를 설정합니다.
- `ports:` (`"1523:1521"` 구조)
    - "외부포트:내부포트"의 구조입니다. 내 컴퓨터에서 1523번 포트로 접속하면, 도커 내부의 1521번(오라클 기본) 포트로 길을 연결해줍니다. 포트 충돌을 피하기 위해 외부 포트(1523, 8081)를 다르게 설정한 것입니다.
- `volumes:`
    - 도커 컨테이너는 삭제되면 내부 데이터도 모두 날아갑니다. 이를 방지하기 위해 `volumes`로 영구 저장소(가상 하드디스크)를 선언하고 컨테이너 내부 폴더와 연결합니다.
    - `oracle_data`는 DB 데이터를, `upload_files`는 사용자가 업로드한 이미지를 보존합니다.
    - `./FinalProject/init_db.sql:/docker-.../init.sql`: 내 컴퓨터의 SQL 파일을 컨테이너 내부의 자동 실행 폴더에 덮어씌워, DB가 처음 만들어질 때 테이블이 알아서 생성되게 돕는 역할입니다.
- `healthcheck:` / `depends_on:` (실행 순서 제어)
    - 도커는 여러 컨테이너를 동시에 켜버립니다. 백엔드보다 DB 부팅이 훨씬 느리기 때문에, 백엔드가 먼저 켜지면 DB 연결 실패로 서버가 뻗어버립니다.
    - `healthcheck`로 DB가 정상 작동하는지 30초마다 검사하고, 백엔드의 `depends_on: condition: service_healthy` 설정을 통해 DB가 완전히 '정상(healthy)' 판정을 받을 때까지 기다렸다가 안전하게 켜지도록 순서를 맞춥니다.
- `networks:`
    - 독립된 컨테이너들이 서로를 이름(`http://backend` 등)으로 부르며 자유롭게 통신할 수 있도록 가상의 내부 네트워크 망(`easyearth_net`)으로 묶어줍니다.
## 5. 보안 및 실행 관리
- **보안 및 환경 변수 처리**
    - 소스 코드 저장소에 민감한 정보가 올라가지 않도록 철저히 분리
    - **어떻게?**: 루트 디렉토리에 `.env`, `.env.example` 작성 및 `.gitignore` 업데이트
> [!warning] .env 파일
> 
> `.env` 파일은 실제 서비스 구동에 필요한 비밀 키를 담고 있으므로, ==형상관리(Git) 업로드를 엄격히 차단==해야 함 (보안 사고 원인 1순위)
```properties
# .env (실제 구동 환경 변수 - Git 커밋 절대 금지. 나만 보는 파일)
SPRING_DATASOURCE_URL=jdbc:oracle:thin:@db:1521:xe
SPRING_DATASOURCE_USERNAME=EASYEARTH
SPRING_DATASOURCE_PASSWORD=EASYEARTH
JWT_SECRET=your_real_secret_key_here
KAKAO_CLIENT_ID=your_real_kakao_id
GEMINI_API_KEY=your_real_gemini_key

# .env.example (형상관리 업로드용 템플릿 - 팀원들에게 양식을 알려주는 껍데기 파일)
SPRING_DATASOURCE_URL=
SPRING_DATASOURCE_USERNAME=
SPRING_DATASOURCE_PASSWORD=
JWT_SECRET=
KAKAO_CLIENT_ID=
GEMINI_API_KEY=

# .gitignore (Git의 변경사항 추적 대상에서 영구적으로 제외할 파일 목록)
.env
```
- **최종 빌드 및 서비스 구동**
    - 전체 시스템의 생성과 삭제를 터미널 명령어 한 줄로 통제
    - **어떻게?**: 최상단 루트 폴더(`docker-compose.yml` 위치)에서 터미널을 열고 아래 명령어 수행
```bash
# 기존 컨테이너 및 찌꺼기 볼륨(-v)까지 안전하게 중지 및 완전 삭제 (초기화)
docker-compose down -v

# 최신 코드를 반영하여 이미지를 새로 만들고(--build), 숨김 모드(-d)로 실행 유지
docker-compose up --build -d
```
